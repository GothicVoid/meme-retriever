### 5.3.2.3 标签搜索加权规则（优先级：P1）

#### 修订记录
| 版本号 | 修订日期 | 修订人 | 修订内容 | 修订原因 |
|-------|---------|--------|---------|---------|
| v1.0 | 2026-04-14 | 开发 | 新增标签搜索加权规则文档 | 分离标签属性与搜索策略，降低耦合 |
| v1.1 | 2026-04-15 | 开发 | 细化标签分支计分公式、来源修正、置信度修正与单图聚合规则 | 在保持现有搜索框架稳定的前提下引入新标签系统 |
| v1.2 | 2026-04-15 | 开发 | 明确搜索阶段的知识库查询归一化与别名命中规则 | 保证标准标签落库后仍能被别名查询命中 |

#### 功能描述
搜索阶段在现有 OCR、标签、CLIP 融合排序基础上，引入标签分类权重，使不同类型标签对最终排序的影响可控且可解释。

#### 设计原则
- 分类权重属于搜索策略，不直接回写标签存储
- 手动标签优先级高于自动标签，但不能完全压制其他信号
- 分类权重只影响标签分支，不改写 OCR 和 CLIP 的基础得分
- 新标签系统优先重构“标签分支”，不重写整体搜索总公式
- 单图标签分支需避免因标签数量过多而被刷分

#### 分类权重建议
| 维度 | 默认权重 | 说明 |
|-----|---------|------|
| 自定义标签 | 1.15 | 用户主动维护，可信度最高，但不应过强 |
| 梗类标签 | 1.0 | 表情包检索主场景 |
| 人物标签 | 0.85 | 视觉主体重要，但第一阶段自动命中可信度相对保守 |
| 出处标签 | 0.75 | 常用于辅助定位，但不应被过度削弱 |

说明：
- 分类权重属于轻量修正，不应成为压倒性因素
- 若后续线上或本地验证表明 `person/source` 被低估，可继续收窄权重差距

#### 标签分支计分公式
标签分支建议按以下结构计算：

```
标签命中得分 = 标签文本匹配得分 × 分类权重 × 来源修正系数 × 置信度修正系数
```

其中：
- `标签文本匹配得分`：描述查询与标签本身的匹配强度
- `分类权重`：体现 `meme/person/source/custom` 的业务差异
- `来源修正系数`：体现手动标签与自动标签可信度差异
- `置信度修正系数`：体现自动标签自身质量差异

#### 标签文本匹配得分建议
| 命中方式 | 建议得分 | 说明 |
|---------|---------|------|
| canonical 完全命中 | 1.0 | 查询与标准标签完全一致 |
| alias 归一命中 | 0.9 | 查询命中知识库别名，归一后命中标准标签 |
| 标签部分命中 | 0.8 | 查询与标签文本部分匹配 |
| 关联命中 | 0.6 | 查询经知识库扩展后间接命中 |

#### 搜索阶段知识库归一化规则
搜索阶段应与入库阶段保持一致的标准化思路，不仅对图片标签做标准化，也要对用户查询做知识库归一化。

推荐流程：
1. 用户输入原始查询
2. 系统查询知识库，判断该查询是否命中某个 `canonical` 或 `alias`
3. 若命中 `alias`，将其归一到对应 `canonical`
4. 使用归一后的标签文本参与标签分支匹配
5. 同时保留原始查询用于 OCR 和其他分支，避免过度丢失原查询语义

示例：
- 图片落库标签：`蚌埠住了`
- 知识库别名：`绷不住了`
- 用户查询：`绷不住了`
- 搜索阶段归一：`绷不住了 -> 蚌埠住了`
- 最终命中图片标签 `蚌埠住了`

说明：
- 图片标签存储建议统一写 `canonical`
- 搜索阶段必须支持由 `alias` 到 `canonical` 的归一命中
- 若只在入库阶段做标准化而搜索阶段不做，会削弱标准标签体系的价值

#### 来源修正系数建议
| 来源 | 建议权重 | 说明 |
|-----|---------|------|
| manual | 1.0 | 用户主动维护，可信度最高 |
| ocr+file_name | 0.92 | 多来源自动命中，略高于单 OCR |
| ocr | 0.9 | 第一阶段主自动来源 |
| example_image | 0.88 | 后续启用时的视觉样本来源 |
| clip_text | 0.8 | 后续启用时的语义文本来源 |
| file_name | 0.75 | 仅弱辅助来源，权重最低 |

说明：
- 第一阶段实际可用来源主要为 `manual`、`ocr`、`ocr+file_name`、`file_name`
- 后续来源值保留为扩展，不要求首期全部上线

#### 置信度修正系数建议
若标签记录包含 `confidence`，建议使用平滑修正，而非直接线性裸乘：

```
ConfidenceWeight = 0.7 + 0.3 × confidence
```

说明：
- 当 `confidence=1.0` 时，修正系数为 `1.0`
- 当 `confidence=0.5` 时，修正系数为 `0.85`
- 当 `confidence=0.0` 时，修正系数为 `0.7`
- 若手动标签不存 `confidence`，可直接按 `1.0` 处理

#### 单图标签分支聚合规则
为避免标签多的图片天然占优，单图标签分支不建议简单累加。

第一阶段建议：
```
S_tag(image) = max(标签命中得分)
```

可选增强方案：
```
S_tag(image) = max(标签命中得分) + 0.1 × second_best
```

第一阶段默认采用 `max`，保持排序稳定且可解释。

#### 与现有搜索总公式的关系
总分仍沿用现有搜索融合框架，仅替换原有简单标签分支：
```
总相关性 = OCR分支 + 标签分支 + CLIP分支
最终排序 = 相关性 + 热度修正
```

若采用现有三分支加权形式，建议表达为：
```
Relevance = w_tag × S_tag + w_ocr × S_ocr + w_clip × S_clip
FinalScore = 0.75 × Relevance + 0.25 × Popularity
```

说明：
- `w_tag` 可沿用当前 `关键词/标签分支权重`
- 新标签系统主要替换 `S_tag` 的定义，不要求同步改写 OCR 与 CLIP 公式

#### 匹配规则
- 完全匹配：标签文本与查询完全一致
- 别名匹配：查询命中知识库别名并映射到 canonical 标签
- 部分匹配：标签文本包含查询关键片段
- 关联匹配：查询经知识库扩展后间接命中标签

#### 搜索阶段标签命中优先级
搜索阶段建议按以下优先级判断标签分支命中：
1. `canonical` 完全命中
2. 知识库 `alias` 归一命中
3. 标签文本部分命中
4. 知识库扩展后的关联命中

说明：
- `alias` 归一命中属于强命中，不应视为普通部分匹配
- 关联命中置信度低于别名归一命中，建议保留更低权重

#### 限制条件
- 分类权重不得导致标签分支恒定压过 OCR 和 CLIP
- 若单图存在大量自定义标签，应限制标签分支总加成
- 对空查询不应用分类加权逻辑
- `custom` 标签不得因权重过高而显著破坏整体排序
- 自动标签不得仅凭数量优势压过高质量 OCR 或 CLIP 命中

#### 边界情况处理
| 场景描述 | 触发条件 | 系统行为 | 提示信息 |
|---------|---------|---------|---------|
| 同图多标签命中 | 多个标签同时命中查询 | 默认取最高标签分，避免叠加失真 | 无 |
| 用户滥用自定义标签 | 单图自定义标签过多 | 限制标签分支总上限 | 无 |
| 自动标签误命中 | 自动标签低置信度但命中查询 | 降低来源修正系数或按置信度过滤 | 无 |
| 分类缺失 | 历史标签无分类字段 | 迁移期按 `custom` 或兼容默认值处理 | 无 |
| 无 confidence 字段 | 历史数据或手动标签未提供 confidence | 按 `1.0` 处理，不影响排序 | 无 |
| 来源值未知 | 历史数据缺失 sourceStrategy | 迁移期按保守默认值处理，建议视为 `manual` 或配置默认值 | 无 |

#### 验收标准
**功能指标：**
- [ ] 标签分类可参与排序
- [ ] 手动标签优先级高于自动标签
- [ ] 搜索结果排序可解释，不出现明显失真
- [ ] 兼容历史无分类标签数据
- [ ] 单图标签数量不会直接带来线性刷分
