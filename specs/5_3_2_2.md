### 5.3.2.2 标签分类与存储模型（优先级：P1）

#### 修订记录
| 版本号 | 修订日期 | 修订人 | 修订内容 | 修订原因 |
|-------|---------|--------|---------|---------|
| v1.0 | 2026-04-14 | 开发 | 新增标签分类与存储模型文档 | 对齐结构化标签需求与现有字符串模型差异 |
| v1.1 | 2026-04-14 | 开发 | 收敛首期标签模型，补充知识库最小字段、匹配规则、打分阈值、冲突裁决与维护策略 | 明确知识库如何支撑自动打标 |

#### 功能描述
将现有“纯字符串标签”升级为“结构化标签对象”，支持分类、来源、自动/手动状态和置信度，用于支撑自动录入、分类展示和搜索加权。

#### 标签分类
| 分类 | 标识值 | 说明 | 示例 |
|-----|-------|------|------|
| 梗 | meme | 表示表情包语义或梗点 | 蚌埠住了、破防了 |
| 人物 | person | 表示人物、角色、主体 | 马云、甄嬛 |
| 出处 | source | 表示影视、节目、作品来源 | 甄嬛传、让子弹飞 |
| 自定义 | custom | 用户主动维护的标签 | 常用、斗图群专用 |

#### 标签对象定义
```
TagObject {
  text: string,
  category: "meme" | "person" | "source" | "custom",
  isAuto: boolean,
  sourceStrategy: "manual" | "ocr" | "file_name" | "ocr+file_name" | "clip_text" | "example_image" | "fallback",
  confidence?: number
}
```

说明：
- 第一阶段实际使用的自动来源主要为 `ocr`、`file_name`、`ocr+file_name`
- `clip_text` 与 `example_image` 保留为后续扩展值，不作为首期必选实现

#### 存储规则
- 每张图片的标签以结构化记录保存
- 自动标签和手动标签可并存
- 用户修改自动标签后，保存结果按用户最终输入覆盖原自动标签属性
- 自定义标签默认 `category=custom`
- 文本相同但分类不同的标签默认视为冲突，不允许重复存储

#### 分类判定规则
- 由知识库条目直接给出分类时，优先使用知识库分类
- 手动新增标签默认归类为 `custom`
- 用户在详情页调整分类后，以用户选择为准
- 第一阶段不做多分类并存，一个标签仅属于一个主分类

#### 知识库设计
第一阶段知识库定位为“结构化概念词典”，核心职责是：
- 定义标准标签名
- 定义该标签的别名与匹配线索
- 定义该标签所属分类

第一阶段知识库条目建议具备以下字段：
```
KbEntry {
  canonical: string,
  category: "meme" | "person" | "source",
  aliases: string[],
  match_terms: string[],
  description: string,
  tags: string[],
  match_mode?: "exact" | "contains" | "exact_or_contains",
  priority?: number,
  example_images?: string[]
}
```

说明：
- `canonical`：标准标签名，最终落库统一写入该值
- `category`：标签分类，第一阶段必须提供
- `aliases`：同义词、口语化写法、变体表达
- `match_terms`：用于命中的线索词，不要求一定是标准别名
- `description`：给人看的解释字段，不参与第一阶段核心匹配
- `tags`：可保留为附属信息字段，但不作为第一阶段必需匹配字段
- `match_mode`：控制匹配策略，默认建议为 `contains`
- `priority`：候选冲突时的轻量排序权重，默认 `1.0`
- `type` 字段建议统一收敛为 `category`
- `example_images` 为预留扩展字段，第一阶段可选

#### 第一阶段最小可行字段
若只做首期自动打标，知识库最小可行字段建议为：
```
KbEntry {
  canonical: string,
  category: "meme" | "person" | "source",
  aliases: string[],
  match_terms: string[]
}
```

#### 字段职责说明
- `canonical` 用于标准化落库，避免同一概念出现多个标签文本
- `aliases` 用于表达等价说法
- `match_terms` 用于放置命中线索，如台词片段、角色称呼、常见短语
- `aliases` 与 `match_terms` 可以重叠，但不建议混为同一个概念字段

示例：
```json
{
  "canonical": "甄嬛传",
  "category": "source",
  "aliases": ["后宫甄嬛传"],
  "match_terms": ["甄嬛传", "甄嬛", "皇上", "臣妾", "熹贵妃"]
}
```

其中：
- `后宫甄嬛传` 是别名
- `皇上`、`臣妾` 更适合作为匹配线索，而不是标准别名

#### 第一阶段知识库样例
```json
{
  "version": 1,
  "entries": [
    {
      "canonical": "蚌埠住了",
      "category": "meme",
      "aliases": ["绷不住了", "蚌住了", "绷住了"],
      "match_terms": ["蚌埠住了", "绷不住了", "蚌住了", "绷住了"],
      "description": "表示忍不住笑了",
      "match_mode": "contains",
      "priority": 1.0
    },
    {
      "canonical": "马云",
      "category": "person",
      "aliases": ["杰克马"],
      "match_terms": ["马云", "杰克马"],
      "description": "人物标签",
      "match_mode": "exact_or_contains",
      "priority": 0.9
    },
    {
      "canonical": "甄嬛传",
      "category": "source",
      "aliases": ["后宫甄嬛传"],
      "match_terms": ["甄嬛传", "甄嬛", "皇上", "臣妾", "熹贵妃"],
      "description": "电视剧出处标签",
      "match_mode": "contains",
      "priority": 0.95
    }
  ]
}
```

#### 知识库校验规则
- `canonical` 必须唯一
- `category` 必须为合法枚举值
- `aliases` 与 `match_terms` 内部应去重
- 不允许高歧义的超短泛词作为高优先级强匹配项
- 若某个词可命中多个条目，应通过 `priority` 或人工治理解决冲突
- `custom` 不由知识库自动生成

#### 条目维护建议
按以下维度分组维护知识库：
- `meme`：梗短语、反应表达、网络流行语
- `source`：影视、节目、游戏、动画等作品来源
- `person`：真实人物、角色、虚拟角色

第一阶段不建议引入：
- 多级标签关系
- 负样本规则
- 脚本化规则
- 全量示例图库
- 自动学习权重

#### 知识库创建与维护策略
第一阶段知识库应被视为“应用内置的规则资产”，而非普通用户可直接维护的业务功能。

#### 数据来源建议
第一阶段知识库数据建议来自以下来源：
1. 人工整理的高频梗、人物、出处条目
2. 从现有图库中抽样提炼的常见 OCR 文本
3. 从历史搜索词或常见检索意图中反推的高价值标签

收录优先级建议：
- OCR 易识别且用户高频搜索的条目优先
- 别名多、人工归一价值高的条目优先
- 歧义低、分类明确的条目优先

第一阶段不建议直接依赖：
- 在线抓取热门梗
- 自动爬虫构建知识库
- 大模型自动批量生成条目后直接上线
- 未经审核的社区共建数据

#### 维护责任
第一阶段知识库建议由项目维护者维护，具体可为：
- 产品与开发共同定义字段规范与准入规则
- 固定维护者负责条目录入、修订和冲突治理
- 知识库更新通过版本管理随应用版本发布

说明：
- 第一阶段不建议将条目维护责任下放给普通用户
- 知识库质量优先于规模，宁可小而准，不宜大而脏

#### 交付形式
第一阶段建议采用本地 JSON 文件交付：
- 随应用发布
- 进入版本控制
- 可在本地读取加载
- 支持后续增量扩展

推荐形式：
```
knowledge_base.json
```

推荐特征：
- 应用内置默认知识库
- 可随版本更新
- 不依赖联网

#### 用户维护策略
第一阶段不建议提供普通用户可视化维护功能，不做以下能力：
- 知识库管理页面
- 条目新增/编辑/删除 UI
- 导入导出界面
- 复杂审核与冲突处理流程

原因：
- 知识库维护本身具备较高规则门槛
- 功能范围会迅速膨胀为配置管理系统
- 在自动打标价值尚未验证前，维护 UI 的投入产出比低

#### 后续扩展策略
后续可按以下顺序逐步开放：
1. 内置静态知识库
2. 支持本地附加词库或用户自定义词库文件
3. 评估是否提供知识库维护 UI

若后续支持用户自定义词库，建议采用：
- 用户本地文件导入
- 应用启动时合并“内置词库 + 用户词库”
- 用户词库优先级高于内置词库

#### 产品边界结论
第一阶段：
- 知识库由应用内置
- 由项目维护者维护
- 普通用户不直接维护知识库

后续阶段：
- 可为高级用户预留自定义词库入口
- 是否提供可视化维护功能，需在验证自动打标效果后再评估

#### OCR 到知识库的匹配规则
第一阶段建议采用可解释的规则匹配，不引入复杂分词或大规模模糊召回：
1. 对 OCR 文本执行基础归一化
2. 遍历知识库条目
3. 用 `canonical + aliases + match_terms` 进行命中判断
4. 命中的条目进入候选集
5. 最终统一写入 `canonical`

基础归一化建议包括：
- 大小写统一
- 全角半角统一
- 去多余空白
- 去常见噪声标点

命中类型建议包括：
- 精确命中
- 子串命中
- 别名命中

候选排序建议优先考虑：
- 命中类型
- 命中词长度
- 条目 `priority`
- 分类策略

说明：
- 第一阶段不要求做复杂中文分词
- 第一阶段不建议依赖 description 全文匹配
- 第一阶段不建议把 OCR 原文直接落库成标签，应统一归并为 `canonical`

#### 第一阶段命中分级规则
建议按以下优先级识别命中类型：

1. `canonical` 精确命中
2. `alias` 精确命中
3. `canonical/alias` 子串命中
4. `match_term` 命中
5. `file_name` 弱命中

说明：
- `canonical` 精确命中表示 OCR 文本中完整出现标准标签名
- `alias` 精确命中表示 OCR 文本命中别名，但最终归并到 `canonical`
- `match_term` 更偏线索命中，不等价于标准标签本身
- `file_name` 弱命中仅作为补充，不建议单独高置信入库

#### 第一阶段默认打分规则
建议默认分值如下：

| 命中类型 | 默认分值 | 说明 |
|---------|---------|------|
| canonical 精确命中 | 1.0 | 最高置信 |
| alias 精确命中 | 0.9 | 高置信 |
| canonical/alias 子串命中 | 0.8 | 中高置信 |
| match_term 命中 | 0.6 | 中等置信 |
| file_name 弱命中 | 0.3 | 低置信补充 |

增强规则建议：
- 同一条目被多个来源命中时，每增加一个独立来源可加 `0.1`
- 同一条目被多个不同 `match_terms` 命中时，每增加一个线索词可加 `0.05`
- 总分封顶 `1.0`

#### 候选对象建议
为便于调试与后续扩展，候选标签建议至少保留以下信息：
```ts
CandidateTag {
  canonical: string,
  category: string,
  score: number,
  matchedBy: "canonical" | "alias" | "match_term" | "file_name",
  matchedText: string,
  priority: number
}
```

#### 落库阈值建议
不同分类建议采用不同阈值，保持第一阶段保守策略：

| 分类 | 建议阈值 |
|-----|---------|
| meme | 0.6 |
| source | 0.7 |
| person | 0.8 |

说明：
- `person` 误打成本较高，建议阈值最高
- `meme` 主要依赖 OCR 文本，可适当放宽
- `custom` 不自动生成，因此无阈值

#### 裁剪规则建议
- 单张图片自动标签总数最多保留 `5` 个
- 同一分类自动标签最多保留 `2` 个
- 低于分类阈值的候选不落库
- 仅由 `file_name` 单独命中的候选默认不直接落库

#### 冲突裁决规则
多个条目同时命中时，建议按以下顺序裁决：
1. 总分高者优先
2. 命中类型优先级高者优先
3. 命中词长度更长者优先
4. 条目 `priority` 更高者优先
5. 同分类超过上限时，保留前 N 个

补充规则：
- `source` 与 `person` 可并存，但同类不宜过多
- 若一个条目被多个线索词命中，应优先于仅单次短词命中的条目
- 若冲突无法自动裁决，优先保守处理，宁可不自动落库

#### 短词限制规则
第一阶段需对短词严格限制，避免高噪声误判：
- 单字词默认不参与自动匹配
- 2 字词默认仅允许 `exact`
- 高歧义 2-3 字词需通过白名单或人工审核后才能作为强匹配项
- 人名、角色名等短词建议结合 `priority` 控制

不建议作为高置信强匹配项的典型词：
- `老师`
- `哥哥`
- `无语`
- `笑死`

#### 分类匹配策略建议
| 标签分类 | 第一阶段策略 |
|---------|-------------|
| meme | 可适度放宽，优先依赖 OCR 文本短语 |
| source | 保守，建议至少有明确作品名或强线索词 |
| person | 最保守，第一阶段不宜仅靠弱短词落库 |
| custom | 不自动生成 |

#### 编辑规则
- 用户可修改标签文本
- 用户可切换标签分类
- 用户可删除自动标签
- 用户新增标签默认作为手动标签保存
- 保存后不保留“系统锁定不可编辑”的自动标签

#### 边界情况处理
| 场景描述 | 触发条件 | 系统行为 | 提示信息 |
|---------|---------|---------|---------|
| 标签为空 | 输入空字符串或仅空格 | 不允许保存 | "标签不能为空" |
| 标签过长 | 标签字符数超过 50 | 拒绝保存 | "标签最长50字符" |
| 分类缺失 | 自动标签无有效分类 | 不入库或降级为 custom，具体由产品配置决定 | "标签分类异常，已按自定义处理" |
| 文本重复 | 同图内出现相同标签文本 | 自动去重，保留优先级更高的记录 | 无 |
| 文本冲突 | 同图内同文本不同分类 | 以用户手动分类为准，否则保留高置信度版本 | "发现冲突标签，已按优先级保留" |

#### 验收标准
**功能指标：**
- [ ] 标签支持四类分类
- [ ] 自动标签和手动标签可区分存储
- [ ] 用户可修改自动标签的文本与分类
- [ ] 结构化标签可向前兼容现有标签功能
