# 导入与恢复状态机

- 状态：approved
- 关联 Spec：`SPEC-LIBRARY`

## 目的

用一张状态机表统一约束 `导入中`、`中断恢复`、`恢复进行中`、`恢复完成结果` 在首页、启动时和图库页的表现，避免局部修改后再次出现口径漂移。

## 用户问题

这份状态机必须始终帮助用户回答下面 5 个问题：

1. 现在是不是还在处理？
2. 还剩多少没处理？
3. 我现在需不需要立刻做决定？
4. 刚才继续导入后结果怎么样？
5. 这个提示什么时候该消失？

## 状态源

| 状态源 | 含义 | 主要承载 |
|------|------|---------|
| `libraryStore.indexing` | 当前会话内是否正在正常导入或恢复处理 | 进行中进度 |
| `taskRecoveryStore.pendingCount` | 当前仍未完成的任务数 | 角标、恢复横幅、恢复弹层 |
| `taskRecoveryStore.activeRecovery` | 是否处于“继续导入未完成任务”的恢复执行中 | 区分待恢复与恢复中 |
| `taskRecoveryStore.completedRecoverySummary` | 最近一次恢复完成后的结果摘要 | 恢复结果摘要 |
| `get_latest_import_summary` | 最近一整批导入的历史汇总 | 普通导入结果摘要 |

## 状态定义

| 状态 ID | 状态名 | 用户理解 | 进入条件 | 退出条件 |
|--------|--------|----------|----------|----------|
| `IDLE` | 空闲 | 当前没有导入或恢复问题 | 无进行中任务，且无待恢复任务 | 发起普通导入；或检测到待恢复任务 |
| `IMPORTING` | 普通导入进行中 | 系统正在处理这次新导入 | 用户在当前会话发起导入，且 `libraryStore.indexing=true`，并非恢复流程 | 全部完成；或应用中断，留下未完成任务 |
| `RECOVERY_PENDING` | 待恢复 | 上次导入中断后还有图片没处理完，等用户决定 | `pendingCount>0`，且当前不在正常导入、不在恢复处理中 | 用户点击 `继续导入`；用户点击 `放弃剩余图片`；或遗留任务被其它方式收口 |
| `RECOVERY_RUNNING` | 恢复处理中 | 系统正在继续导入上次剩余任务 | 用户点击 `继续导入`，且 `activeRecovery=true` | 恢复完成；或再次中断 |
| `RECOVERY_DONE` | 恢复结果可确认 | 刚才继续导入的剩余任务已经有结果；如果用户关心，可以快速确认 | `activeRecovery=false`，`completedRecoverySummary!=null` | 用户查看失败项；查看本次新增；明确关闭/忽略本次恢复结果；发起新的普通导入；产生新的恢复结果 |
| `IMPORT_DONE` | 普通导入结果可查看 | 最近一整批导入已经完成；如果用户关心，可以查看历史汇总 | 无恢复结果待确认，且存在最近一次导入汇总 | 发起新的普通导入；进入恢复流程；被更新结果覆盖 |

## 状态转移

| 当前状态 | 触发事件 | 下一状态 | 说明 |
|---------|---------|---------|------|
| `IDLE` | 用户发起普通导入 | `IMPORTING` | 开始处理新导入 |
| `IMPORTING` | 当前批次全部完成 | `IMPORT_DONE` | 展示普通导入结果 |
| `IMPORTING` | 应用关闭/崩溃且仍有未完成任务 | `RECOVERY_PENDING` | 下次启动进入待恢复 |
| `RECOVERY_PENDING` | 用户点击 `继续导入` | `RECOVERY_RUNNING` | 开始恢复剩余任务 |
| `RECOVERY_PENDING` | 用户点击 `放弃剩余图片` | `IDLE` | 只清理未完成任务 |
| `RECOVERY_RUNNING` | 恢复全部完成 | `RECOVERY_DONE` | 展示本次恢复结果 |
| `RECOVERY_RUNNING` | 再次中断 | `RECOVERY_PENDING` | 下次仍可恢复 |
| `RECOVERY_DONE` | 用户查看失败项 | `IDLE` 或 `IMPORT_DONE` | 恢复结果完成承接后退出主位；如存在普通导入历史汇总，可回落到 `IMPORT_DONE` |
| `RECOVERY_DONE` | 用户查看本次新增 | `IDLE` 或 `IMPORT_DONE` | 同上 |
| `RECOVERY_DONE` | 用户明确关闭/忽略本次恢复结果 | `IDLE` 或 `IMPORT_DONE` | 用户已知晓本次恢复结果，但选择暂不处理，结果退出主位；如存在普通导入历史汇总，可回落到 `IMPORT_DONE` |
| `RECOVERY_DONE` | 用户发起新的普通导入 | `IMPORTING` | 旧恢复结果退场 |
| `IMPORT_DONE` | 用户发起新的普通导入 | `IMPORTING` | 新导入覆盖旧结果 |
| `IMPORT_DONE` | 检测到待恢复任务 | `RECOVERY_PENDING` | 恢复问题优先级更高 |

## 界面映射

| 状态 | 首页图库按钮 | 启动恢复弹层 | 图库页顶部 | 图库页结果摘要 |
|------|-------------|-------------|-----------|---------------|
| `IDLE` | 无角标 | 不显示 | 无恢复横幅 | 可无摘要或显示普通历史摘要 |
| `IMPORTING` | 不显示恢复角标；若用户不在图库页，至少提供可见但不打断的进行中提示入口 | 不显示 | 显示进行中进度 | 不显示恢复摘要 |
| `RECOVERY_PENDING` | 显示恢复角标 | 仅在需要用户立刻决策时显示 | 显示恢复横幅 | 不显示历史导入摘要主位 |
| `RECOVERY_RUNNING` | 不显示恢复角标；若用户离开图库页，至少提供可见但不打断的进行中提示入口 | 不显示 | 显示恢复进度 | 不显示恢复横幅 |
| `RECOVERY_DONE` | 不显示恢复角标 | 不显示 | 不显示恢复横幅 | 优先显示本次恢复结果，但不应强迫用户停下来处理 |
| `IMPORT_DONE` | 不显示恢复角标 | 不显示 | 无恢复横幅 | 仅在短时承接窗口内显示最近一次导入历史汇总；超时或已忽略后不再打断用户 |

## 默认动作

| 状态 | 默认主动作 |
|------|-----------|
| `RECOVERY_PENDING` | `继续导入` |
| `RECOVERY_PENDING` | 次动作：`放弃剩余图片` |
| `RECOVERY_DONE` 且有失败 | `查看失败项` |
| `RECOVERY_DONE` 仅新增成功 | `查看本次新增` |
| `RECOVERY_DONE` 仅重复 | 无强制主动作 |
| `IMPORT_DONE` 且有失败 | `查看失败项` |
| `IMPORT_DONE` 仅新增成功 | `查看本次新增` |

补充约束：

- `IMPORT_DONE` 的普通导入历史汇总不是长期历史卡片，只承接“刚完成”的结果
- 首版短时承接窗口按 `30 分钟` 控制，不跨天常驻
- 用户执行主动作或主动忽略后，同一批 `batch_id` 的汇总不应再次回到主位

## 导入过程中操作权限

这一节用于回答“系统正在导入或恢复时，用户还能做什么、不能做什么”，避免实现为了省事一刀切禁用，也避免用户在高风险时误操作。

### 设计原则

- 不影响当前导入结果正确性的操作，尽量允许
- 会改变任务基数、破坏进度语义、或让用户误以为结果已稳定的操作，应限制
- 被限制的操作要给出可理解原因，不能只做静默禁用
- 同一类限制在普通导入与恢复处理中应尽量保持一致

### 操作矩阵

| 操作 | `IMPORTING` | `RECOVERY_RUNNING` | 用户体验口径 |
|------|-------------|--------------------|-------------|
| 查看首页、切换页面 | 允许 | 允许 | 用户可以离开当前页，但系统要保留低干扰进行中承接 |
| 打开图库页查看当前进度 | 允许 | 允许 | 图库页是主承接界面，应允许随时回看 |
| 查看已有图片、打开详情 | 允许 | 允许 | 浏览与查看不应被重任务阻塞 |
| 搜索已存在内容 | 允许，但需明确提示结果可能尚未包含本轮未完成导入内容 | 允许，但需明确提示恢复未完成前结果仍不完整 | 不阻断主路径，但不能让用户误以为“当前结果已经是最终状态” |
| 复制、预览、查看详情中的非破坏性操作 | 允许 | 允许 | 不应因为后台导入而冻结已有内容使用 |
| 再次发起新一轮导入 | 默认不允许与当前导入并行；应引导用户等待当前批次结束后再发起 | 默认不允许；避免恢复批次和新批次混杂 | 重点是保持任务边界清晰，避免用户分不清哪一批出了问题 |
| 继续导入待恢复任务 | 不适用 | 不适用 | 同一时间只存在一个恢复执行流程 |
| 放弃剩余图片 | 不适用 | 默认不允许在恢复进行中直接执行 | 避免用户误以为正在处理的任务可无代价即时抛弃 |
| 删除图片、批量删除、清理失效图片 | 默认不允许 | 默认不允许 | 这些治理操作会改变图库基数和用户对结果的理解，应等当前处理结束 |
| 清空图库等重置类维护操作 | 不允许 | 不允许 | 高风险操作，不能与导入/恢复同时发生 |
| 修改会影响本轮结果承接的顶部主状态 | 不允许手动绕过 | 不允许手动绕过 | 不允许用户把“进行中 / 恢复结果 / 历史汇总”切成自相矛盾状态 |

### 必须允许的操作

下面这些操作属于“即使在导入中也不应被粗暴锁死”的最低要求：

1. 离开当前页并返回
2. 在图库页查看当前进度
3. 浏览已存在图片
4. 使用已有图片的非破坏性能力，例如预览、复制、查看详情
5. 在搜索页继续搜索已有内容，但要接受结果可能尚未包含未完成导入部分

### 必须限制的操作

下面这些操作默认应限制，除非后续明确设计了安全的并发语义：

1. 在当前导入/恢复尚未结束前，再发起一轮新的导入
2. 在当前导入/恢复进行中执行删除、批量删除、清理失效图片等会改变图库基数的治理动作
3. 在恢复进行中直接执行“放弃剩余图片”
4. 在导入/恢复进行中执行高风险重置能力，例如清空图库

### 限制时的反馈规则

- 按钮若被禁用，文案或邻近说明应能回答“为什么现在不能做”
- 若用户触发了被限制操作，反馈应指向可执行的下一步，例如：
  - “先等当前导入完成，再继续添加下一批”
  - “当前正在恢复剩余任务，恢复完成后再做批量清理”
- 不允许只因为技术实现复杂，就把整页大面积冻结成不可用

### 搜索结果不完整提示

- 当用户在 `IMPORTING` 或 `RECOVERY_RUNNING` 中进行搜索时，系统应提供最低限度的可见说明，明确“当前导入尚未完成，搜索结果可能不完整”
- 这类说明应是低干扰提示，不应压过搜索主路径
- 但它必须足够明确，避免用户把“暂时搜不到”误判成搜索质量问题或导入失败

### 搜索结果不完整提示的触发策略

- 默认不应在每次输入、每次搜索时重复强提示
- 建议优先在下面场景触发：
  1. 用户在当前处理会话中首次进入搜索态
  2. 搜索结果为空或明显偏少时
  3. 用户通过进行中承接入口回到搜索/首页后首次发起搜索
- 同一处理会话内，如果用户已经看过这类提示，后续应降级为轻提示或不重复展示
- 一旦当前处理完成，这类提示应立即退出，不应滞留到结果已完整的阶段

## 弹层使用规则

- 启动恢复弹层是阻断式提示，只用于 `RECOVERY_PENDING`
- 只有在“用户不先做决定就很容易误判图库状态”时才优先用弹层
- 至少满足下面任一条件时，才应考虑弹层而不是仅用角标/横幅：
  - 遗留任务数量明显大于 1
  - 上次中断发生在用户明确发起导入后不久，用户大概率仍期待这批素材可用
  - 如果不提示，用户会直接把当前图库状态误判为“导入已完成”
- 低风险、低规模遗留任务优先用首页角标和图库横幅承接，不强制打断
- 弹层、角标、横幅不应在同一时刻重复放大同一问题

## 进行中承接规则

- `IMPORTING` 和 `RECOVERY_RUNNING` 都属于“系统仍在处理”
- 进行中承接用于回答“还在处理”，不能伪装成“异常待恢复”
- 因此进行中提示与恢复角标必须分离
- 用户离开图库页后，如果后台仍在处理，不应完全静默到像“已经结束”
- 最小产品承诺是：用户离开图库页后，仍能通过首页或主工作台上的一个可见入口知道“导入还在处理”
- 这个入口至少要满足两点：
  - 不会被误解为异常角标
  - 用户点击后能回到正在处理的主承接界面
- 这个入口的最小语义应至少包含三件事中的两件：
  - “正在导入”或“正在继续导入”
  - 剩余数量或进度
  - 可点击返回查看
- 不接受只有一个抽象小点或无语义图标、需要用户自己猜测的方案

### 进行中承接的出现与消失

- 默认出现位置应优先选择用户离开图库页后仍高频可见的位置，例如首页主工作台或首页图库入口附近
- 默认出现时机：
  - 用户离开图库页，且 `IMPORTING` 或 `RECOVERY_RUNNING` 仍然成立
  - 或用户始终停留在首页/搜索页，但后台处理已经开始
- 默认消失时机：
  - 当前处理全部完成
  - 用户重新进入图库页并由图库页主承接块接手
  - 当前状态从 `IMPORTING / RECOVERY_RUNNING` 退出
- 一旦处理中断为 `RECOVERY_PENDING`，进行中承接必须退出，切回“待决定是否恢复”的语义

## 退场规则

### 恢复结果退场

- `RECOVERY_DONE` 不是长期公告，也不是必须处理的阻断任务
- 当用户完成下面任一动作后，恢复结果退出主位：
  - 查看失败项
  - 查看本次新增
  - 明确关闭或忽略本次恢复结果
  - 发起新的普通导入
  - 产生新的恢复结果

### 普通导入结果退场

- 新的普通导入结果会覆盖旧的普通导入结果
- 一旦进入恢复路径，恢复相关状态优先级高于普通导入结果
- 普通导入结果是“可查看摘要”，不是强制用户必须停下来处理的阻断项

## 放弃剩余图片路径

| 当前状态 | 触发事件 | 下一状态 | 用户预期 | 系统要求 |
|---------|---------|---------|---------|---------|
| `RECOVERY_PENDING` | 用户点击 `放弃剩余图片` | `IDLE` | 这批没处理完的任务被真正放弃，不再反复提醒我 | 仅清理未完成任务；不影响已成功入库内容；角标、弹层、横幅都应同步消失 |
| `RECOVERY_RUNNING` | 用户尝试放弃 | 不直接支持，需先结束当前恢复或另设明确中止能力 | 不应让用户误以为“处理中也能无成本立刻清掉” | 若后续支持中止，必须补单独状态与回滚/收尾规则 |

放弃剩余图片至少要让用户感知到这三个结果：

1. 未完成任务提醒已消失
2. 已成功导入的图片还在
3. 刷新或重启后，不会再次看到同一批待恢复任务

适用场景应明确告诉用户：

- 用户确认这批剩余任务暂时不重要，不想继续等
- 用户知道剩余文件本身有问题，继续导入意义不大
- 用户只是想把“未完成提醒”收口，先回到正常使用

不应用它暗示下面这些意思：

- 已成功导入内容也会被撤销
- 当前正在执行的恢复步骤可以无代价立即中止
- 放弃后未来永远不能重新导入这些文件

## 恢复失败轻提醒

- 当 `RECOVERY_DONE` 存在失败项，但用户还没有进入图库页查看恢复结果时，系统应至少提供一次低干扰提醒
- 这类提醒的目的不是再次强拉用户进图库，而是防止“恢复已结束但仍有失败”被完全静默吞掉
- 轻提醒默认应满足：
  - 只提醒一次，或在当前会话内有限次提醒
  - 文案明确是“恢复已结束，但仍有失败项”
  - 点击后能进入图库页并落到恢复结果/失败项承接位
- 若用户已经查看过恢复结果或失败项，这类提醒应退出，不重复追打
- 这类提醒不能复用异常恢复角标；角标用于 `RECOVERY_PENDING`，轻提醒用于 `RECOVERY_DONE` 且有失败

## 不允许出现的行为

- `IMPORTING` 时出现“上次还有未完成任务”的恢复提示
- `RECOVERY_RUNNING` 时角标、横幅、进度条同时表达同一问题
- 用户离开图库页后，后台仍在处理，但系统完全不给任何进行中承接
- `RECOVERY_DONE` 时只显示整批历史导入汇总，用户无法判断这次恢复结果
- 旧的恢复结果在无关页面长期停留，像仍有异常未处理
- `放弃剩余图片` 后，刷新或重启还能看到同一批待恢复任务
- 首页同一个入口同时承担“进行中提示”和“异常恢复角标”两种语义，导致用户误读

## 开发检查清单

后续凡是修改下面任一部分，都要回看这张状态机：

- 导入任务队列
- 普通导入进度
- 恢复逻辑
- 首页图库按钮
- 启动恢复弹层
- 图库页顶部横幅、进度条、结果摘要

至少确认：
