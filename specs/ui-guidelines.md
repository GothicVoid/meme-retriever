## 1. 文档目的

本规范用于约束产品在桌面端的基础视觉语言、组件样式、状态表达和页面节奏，确保后续在搜索页、首页、图库管理页、设置页、详情页和快速预览层中的界面表现保持一致。

本规范不替代功能 spec。功能 spec 负责约束“做什么、怎么交互、如何验收”；本规范负责约束“长什么样、信息如何分层、同类状态如何统一表达”。

## 2. 设计目标

- 让产品从“可用原型”过渡到“可持续日用工具”
- 优先强化主链路：导入、搜索、复制、再找回
- 通过统一层级、间距、状态和文案语气降低认知负担
- 保持桌面应用气质，避免网页后台感和内部工具感
- 在不过度装饰的前提下，建立稳定、可信赖、长期使用的视觉印象

## 3. 适用范围与非目标

**适用范围：**
- 首页 / 搜索页
- 图库管理页
- 设置页
- 单图详情页
- 快速预览层
- Toast、弹层、空状态、错误提示等跨页面组件

**本规范不覆盖：**
- 品牌 logo、插画资产、营销页视觉
- 移动端适配
- 复杂动画系统
- 底层组件库选型和具体技术实现细节

## 4. 整体视觉原则

### 4.1 关键词

- 清晰
- 稳定
- 克制
- 有层级
- 可长期使用

### 4.2 设计取向

- 避免“极简到像没设计”的原型感
- 避免高饱和、多强调色并存导致的工具噪音
- 避免把所有模块都做成同等权重，首屏必须有明显主次
- 避免大量装饰性边框、阴影、渐变叠加造成界面发虚
- 优先用排版、间距、分组和状态色建立完成度，而不是堆视觉效果

### 4.3 页面观感要求

- 首页 / 搜索页应突出搜索作为第一主操作
- 图库管理页应突出“管理感”而不是“陈列感”
- 设置页应降低视觉噪音，避免像调参面板
- 详情页和预览层应突出图片主体，不应让周边信息抢占注意力

## 5. 视觉基础规范

### 5.1 色彩语义

建议采用“中性主色 + 单一品牌强调色 + 状态语义色”的结构，不要让不同功能模块各自使用不同主色。

| 颜色类型 | 用途 | 规则 |
|---------|------|------|
| 主背景色 | 页面底色 | 使用低刺激中性色，保证长时间查看不疲劳 |
| 次级背景色 | 卡片、分组区、浮层底色 | 与主背景保持轻微层次差，避免纯白堆叠 |
| 主文字色 | 标题、正文 | 对比度必须足够，优先可读性 |
| 次级文字色 | 描述、说明、辅助文案 | 明显弱于主文字，但不能发灰到难读 |
| 强调色 | 主按钮、焦点态、关键交互入口 | 全局尽量只保留一种主强调色 |
| 成功色 | 成功提示、已完成状态 | 仅用于成功语义，不用于普通主按钮 |
| 警告色 | 风险提示、文件丢失、谨慎操作 | 用于提醒但不等同于删除危险操作 |
| 危险色 | 删除、不可逆操作、严重错误 | 仅用于高风险动作和严重异常 |
| 信息色 | 中性提示、恢复中、处理中 | 用于过程提示和轻量反馈 |

规则：
- 同一页面内同时出现的强调色不应超过 1 个主强调色 + 1 个状态色
- 状态色必须与语义绑定，不能今天用橙色表示警告，明天又拿来做普通按钮
- 背景层之间的区分优先依赖亮度和边框，不依赖大面积高饱和底色

### 5.2 字体与层级

桌面端优先使用系统无衬线字体栈，保持稳定和可读性。不要为了“设计感”引入识别成本高的展示字体。

| 文本层级 | 典型场景 | 规则 |
|---------|---------|------|
| 页面主标题 | `首页 / 搜索`、`图库管理`、`设置` | 权重最高，页面内唯一主标题 |
| 区块标题 | `最近搜索`、`最近复制`、`异常图片` | 明显强于正文，弱于页面主标题 |
| 卡片标题 / 重点信息 | 图片名、状态标题、弹层标题 | 用于局部主信息 |
| 正文 | 说明文案、描述信息 | 作为默认阅读层 |
| 辅助文案 | 次要描述、时间、来源、计数 | 低一层但需可读 |
| 注释文案 | 调试说明、非关键备注 | 尽量少用，不能替代正文说明 |

规则：
- 一个视觉区域内最多出现 3 个明显字号层级
- 同级标题在全局应使用一致字重，不允许每页自定义风格
- 说明文案优先通过字重和颜色区分，不靠全大写、下划线或多余图标强调

### 5.3 间距与布局节奏

统一采用固定节奏的间距体系，避免页面因“哪里看着空就补一点”而失去秩序。

| 间距层级 | 典型用途 |
|---------|---------|
| `4` | 图标与短文本之间 |
| `8` | 表单项内部、标签组内部 |
| `12` | 卡片内小模块间距 |
| `16` | 卡片内容主间距 |
| `24` | 页面区块内部主间距 |
| `32` | 页面区块之间的垂直间距 |
| `40~48` | 页面主区域上下留白 |

规则：
- 相邻页面区块优先使用 `24` 或 `32`
- 卡片内信息不应贴边，默认至少保留 `12~16`
- 首页首屏搜索区与后续内容区必须留出明显节奏断层，不能像一个连续长列表

### 5.4 圆角、边框与阴影

| 样式元素 | 规则 |
|---------|------|
| 圆角 | 全局控制在 2~3 个层级内，避免每类组件都不一样 |
| 边框 | 优先用于分层、选中态和输入控件轮廓 |
| 阴影 | 仅用于弹层、浮层、悬浮卡和高于页面的对象 |

规则：
- 常规卡片、输入框、按钮应共享接近的圆角语言
- 阴影要轻，强调悬浮关系，不要制造厚重浮夸的“卡片墙”
- 选中态优先使用边框色变化和背景微变化，避免全卡强饱和填充

## 6. 组件规范

### 6.1 顶层导航

- 一级导航数量控制在 3 项内
- 当前项必须具备稳定清晰的选中态
- 导航标签使用任务语言，不使用工程语言
- 导航区域应与页面内容区清晰分层，但不应抢走首屏主操作的视觉重心

### 6.2 搜索框

搜索框是产品第一主操作，应在首页和搜索页保持一致的核心视觉语言。

要求：
- 搜索框在首页首屏应具备高于普通表单控件的视觉权重
- placeholder 作为能力提示，不作为唯一说明来源
- 聚焦态必须明显，支持键盘用户快速识别当前位置
- 搜索历史下拉、建议词、快捷键提示应使用同一套浮层规范
- 搜索框周边可配少量示例词，但不能让首屏变成标签墙

不应：
- 把搜索框做成与普通设置输入框完全同权重
- 在搜索框附近堆叠多组按钮、筛选和调试入口

### 6.3 结果卡

结果卡是搜索页、首页模块和最近复制等场景的核心展示单元，必须维持统一结构。

建议结构：
1. 图片缩略图
2. 状态或格式角标
3. 主理由 / 证据摘要
4. 辅助操作或悬浮操作

规则：
- 图片主体优先，不允许解释文案覆盖图片主要区域
- 卡片信息区应与图片区分层，避免所有内容压在图片上
- Hover、focus、selected 的状态表达应一致，不允许不同列表中的卡片反应方式完全不同
- 最近复制、最近使用、常用表情等模块可复用同一结果卡样式，但应允许减少解释信息密度

### 6.4 按钮

按钮必须通过主次层级和语义色区分，而不是靠尺寸随机决定权重。

| 按钮类型 | 适用场景 | 规则 |
|---------|---------|------|
| 主按钮 | 当前区域唯一最推荐动作 | 每个区域原则上只保留一个主按钮 |
| 次按钮 | 可替代动作 | 视觉低于主按钮 |
| 文字按钮 | 低干扰辅助操作 | 用于次要跳转或轻量操作 |
| 危险按钮 | 删除、清空、不可逆动作 | 必须结合危险色和二次确认 |

规则：
- 同一区域不应出现两个同权重主按钮
- 危险按钮不允许默认高频出现于主路径首位
- 按钮文案优先用动作词，如 `导入图片`、`继续处理`、`查看详情`

### 6.5 Toast 与轻提示

- 仅用于短时反馈，如复制成功、设置已保存
- 默认停留时间应统一，避免每种 Toast 时长不同
- 若带撤销入口，按钮必须足够明显但不能盖过主文案
- Toast 不应承载长说明或复杂修复流程

### 6.6 弹层与抽屉

- 用于快速预览、确认、轻量编辑和附加信息承接
- 必须区分“可快速完成后回到原位置”的弹层与“需要完整页面承载”的复杂任务
- 背景遮罩、标题区、操作区和关闭路径应统一
- Esc、点击关闭按钮、点击遮罩的行为规则应在同类弹层中保持一致

### 6.7 表单与设置项

- 设置页输入控件应以“少而明确”为原则
- 每个设置项应具备标题和一句简短说明，避免用户猜配置影响
- 开关、单选、数字输入不要混排得像调试面板
- 高级能力区与基础设置区视觉上必须分层

## 7. 状态规范

### 7.1 加载态

要求：
- 必须说明正在做什么，例如“正在搜索”“正在整理新导入的图片”
- 搜索页加载时应尽量保留页面骨架，不应整个区域突然清空
- 列表加载优先使用骨架屏或占位卡，不建议只放孤立 spinner

### 7.2 空状态

要求：
- 不能只写“暂无数据”
- 必须包含原因说明和下一步动作
- 图库空、最近复制空、最近成功搜索空、异常图片空，这几类空状态语气应统一

建议结构：
1. 状态标题
2. 一句原因说明
3. 一个推荐动作
4. 必要时补充次动作

### 7.3 错误态

要求：
- 对用户隐藏技术细节，默认不直接暴露异常栈或底层报错文本
- 明确当前失败的是哪个动作
- 至少提供 `重试`、`返回`、`去图库管理`、`查看详情` 中的一种后续动作

### 7.4 文件丢失态

文件丢失是本产品的重要异常态，必须跨页面统一。

统一要求：
- 统一使用“原文件找不到了”这一类用户语言
- 在搜索结果、最近复制、最近使用、详情页和预览层中的语义保持一致
- 优先提供 `重新定位` 和 `删除记录` 两类动作
- 不允许某些页面静默过滤，另一些页面突然报错，造成用户理解断裂

### 7.5 入库中断恢复态

要求：
- 必须说明有未完成任务，不使用模糊提示
- 默认提供 `继续处理` 和 `放弃并清理` 两类动作
- 应出现在用户能重新回到该任务的位置，例如应用重新打开后或图库管理页

### 7.6 成功态

成功态应用于：
- 已复制
- 已保存设置
- 已完成导入
- 已恢复处理

规则：
- 成功态文案简短直接
- 不应与警告、错误共用同一颜色或图标
- 如果成功态需要下一步建议，应使用次级样式表达

## 8. 交互反馈规范

### 8.1 焦点态

- 所有可键盘访问元素必须具备清晰焦点态
- 焦点态不能只依赖浏览器默认细线外框
- 搜索结果卡、按钮、输入框、列表项必须使用统一焦点语言

### 8.2 Hover 与 Active

- Hover 仅用于增强可发现性，不应成为唯一可用线索
- Active 态应明确反馈“已经点击/按下”
- Hover 和 Active 的强度应低于选中态

### 8.3 Disabled

- 禁用态必须同时降低视觉权重和交互可达性
- 不允许通过“看起来像能点但点了没反应”的方式伪装禁用
- 若关键动作被禁用，应尽量给出原因说明

### 8.4 动效原则

- 动效只服务于层级切换、反馈确认和减少突兀感
- 搜索下拉、弹层、Toast、预览切换可使用短时过渡
- 不做花哨入场动画，不做与效率目标冲突的长动画

## 9. 页面级规范

### 9.1 首页 / 搜索页

- 首屏必须突出搜索框和能力说明
- 首页区块顺序应形成“开始搜索 -> 继续使用 -> 补充线索”的节奏
- 搜索结果页中的主结果区应始终高于辅助说明区
- 失败反馈区不应像报错日志，应像可继续操作的引导区

### 9.2 图库管理页

- 首屏应让用户一眼看出这里是“管理图库”的地方
- 批量操作区、视图切换区、图片列表区应分层清晰
- 异常图片、最近新增、全部图片等视图的视觉结构应统一
- 管理页允许比搜索页信息密度更高，但不能退化成表格工具面板感

### 9.3 设置页

- 以分组卡或清晰分节的方式组织，而不是长表单直出
- 基础设置、高级能力、开发调试必须清晰分层
- 高级能力入口应低干扰呈现，不能像主路径强操作

### 9.4 详情页

- 图片主体优先
- 元信息、标签、操作区应按阅读顺序组织
- 文件异常信息应靠近主信息区，不应藏在难以发现的位置

### 9.5 快速预览层

- 以“确认是不是这张图”为第一目标
- 大图区域优先，操作区次之
- 关闭路径明确，翻页逻辑稳定
- 不应把预览层做成功能过载的第二详情页

## 10. 文案规范

### 10.1 语气要求

- 直接
- 清楚
- 不卖弄术语
- 不居高临下
- 不用泛化安慰句掩盖问题

### 10.2 文案写法

建议：
- `搜台词、角色、动作、场景`
- `没找到这类图片，换个说法试试`
- `这张图片的原文件找不到了`
- `上次有图片还没整理完`
- `继续处理`

不建议：
- `发生未知错误`
- `暂无数据`
- `请稍后重试`
- `高相关度匹配结果`
- `系统异常`

### 10.3 按钮文案

- 优先写动作，不写结果
- 优先使用 2~6 个字的短动作
- 同类动作全局统一，例如统一使用 `导入图片`，不要一处叫 `添加图片`，另一处叫 `选择文件`

## 11. 一致性检查清单

在新页面或改版提交前，至少检查以下问题：

1. 页面主操作是否在首屏 3 秒内可识别
2. 同类卡片、按钮、输入框是否沿用统一样式语言
3. 加载、空、错误、文件丢失是否使用统一结构和语气
4. 是否存在两个以上同权重主按钮争抢注意力
5. 是否有说明文案只对开发者可读、对普通用户不可读
6. 焦点态、Hover、禁用态是否都明确可见
7. 页面间距和区块节奏是否稳定，而不是凭感觉拼接
8. 危险动作是否被清晰标记，并具备确认机制
9. 首页、搜索页、图库页、设置页是否仍保持各自清晰职责

## 12. 验收标准

**视觉一致性：**
- [ ] 首页、搜索页、图库管理页、设置页在字体层级、间距、卡片样式和状态提示上保持统一
- [ ] 同类组件在不同页面中的圆角、边框、阴影和交互反馈不出现明显割裂
- [ ] 页面首屏均存在明确主操作和次级信息层次

**交互一致性：**
- [ ] 搜索框、结果卡、按钮、Toast、弹层等核心组件在状态反馈上行为一致
- [ ] 键盘焦点、Hover、禁用态具备稳定规则，不依赖个别页面临时实现
- [ ] 文件丢失和入库中断恢复等关键异常态在跨页面场景下语义统一

**可用性：**
- [ ] 新用户首次进入后能快速理解产品主路径
- [ ] 高频用户在首页和搜索页中能快速识别主要操作区域
