### 5.6.3 导入与图库管理解耦及冷启动导入闭环功能（优先级：P2）

#### 修订记录

#### 功能描述
结合产品总定义、首页启动态、导航重构和聊天伴随型侧边栏工作台的既有结论，本次迭代对“导入”和“图库管理”的关系做一轮职责收口。

核心结论如下：
- 导入仍然写入同一套图库和索引体系，但交互入口不再默认绑定到图库管理页
- 冷启动首页中的主按钮应触发“快速导入”动作，而不是直接把用户带到展开态图库管理页
- 冷启动首页使用单一主按钮 `导入图片`，点击后通过轻弹层让用户选择 `选择图片` 或 `选择文件夹`
- 图库管理页继续保留导入能力，但其主职责应回到“整理、排查、修复和批量治理”
- 图库管理页继续保留 `添加图片` 和 `添加文件夹` 两个独立按钮
- 首次使用主链路应明确收敛为 `导入 -> 搜索 -> 复制`

本次调整不是删除图库管理页，而是将其从“首次导入承接页”调整为“搜索失败修复页 + 重度治理页”。

#### 设计目标
- 用户首次进入后 5 秒内能理解首个动作是 `导入图片`
- 用户首次进入后 10 秒内能理解主链路是 `导入 -> 搜索 -> 复制`
- 聊天伴随型侧边栏态下，首次导入不应强制打断为展开管理态
- 用户不需要先理解 `最近新增`、`异常图片`、`清除失效图片` 等治理概念，也能完成首次开始使用
- 图库管理页的职责更清晰，用户不会把它误认为另一个搜索首页或首次必经页

#### 范围说明
**本期做什么：**
- 将冷启动首页主按钮从“进入图库管理”调整为“快速导入”
- 定义冷启动页 `导入图片 -> 轻弹层 -> 选择图片 / 选择文件夹` 的入口交互
- 定义搜索页内快速导入闭环，包括选择、导入中、导入完成承接
- 明确快速导入与图库管理的职责分工
- 补充新的信息架构图和冷启动四状态交互稿
- 明确侧边栏态与展开态在导入链路中的使用原则

**本期不做什么：**
- 不改变图库底层数据结构、索引写入方式或已有入库能力
- 不删除图库管理页中的导入按钮和批量治理能力
- 不在本期新增拖拽导入、剪贴板导入等新输入方式
- 不重写搜索结果页、详情页或标签编辑页的核心行为
- 不引入新的一级导航页面

#### 调整结论
**1. 职责调整**

搜索页 / 冷启动首页应承担：
- 让用户快速开始使用
- 触发快速导入
- 展示导入中进度和首次导入完成承接
- 导入完成后立刻进入搜索和复制主链路

图库管理页应承担：
- 导入后的补充治理
- 批量追加图片
- 查看最近新增
- 处理异常图片
- 清除失效图片
- 单图补充信息和批量删除

**2. 交互原则**

- `导入图片` 是任务动作，不是页面跳转动作
- 冷启动页应优先降低首次决策成本，因此使用单一主按钮承接导入动作
- 图库管理页应优先保证治理效率，因此保留 `添加图片` 和 `添加文件夹` 两个直接入口
- 只要用户目标仍然是“马上开始找图”，应优先留在侧边栏态或当前搜索页内完成
- 只有当用户明确进入治理、批量整理或需要更大工作区时，才切换到展开态图库管理页

#### 用户流程说明
**主流程 1：首次进入并快速导入**
1. 用户打开应用，进入侧边栏态冷启动首页
2. 用户点击 `导入图片`
3. 系统在当前页就近弹出轻量动作层，提供 `选择图片` 和 `选择文件夹`
4. 用户选择其一后，系统打开对应的原生选择器，不切换到图库管理页
5. 用户完成选择后，当前页进入导入中状态
6. 导入完成后，当前页展示一次性承接提示
7. 用户直接输入搜索词或点击示例词开始真实搜索

**主流程 2：已有图库时继续快速找图**
1. 用户打开应用，进入首页 / 搜索页
2. 用户直接在底部搜索栏输入，或从最近常用候选池点图复制
3. 若搜索成功，直接复制或查看详情

**主流程 3：搜索失败后去图库管理修复**
1. 用户搜索失败或结果不理想
2. 用户点击 `去图库管理`
3. 系统切到展开态图库管理页，并按失败场景进入 `最近新增` 或 `异常图片`
4. 用户执行检查、补充信息、删除异常或追加图片
5. 用户返回首页 / 搜索页再次检索

**主流程 4：重度用户主动治理图库**
1. 用户从一级导航进入 `图库管理`
2. 系统切到展开态
3. 用户执行批量导入、整理、修复和清理动作
4. 完成后返回首页 / 搜索页

| 当前状态 | 触发条件 | 下一状态 | 系统行为 |

| 未导入 | 用户点击 `导入图片` | 选择中 | 打开轻弹层，不切路由 |
| 选择中 | 用户点击 `选择图片` | 选择中 | 打开图片文件选择器 |
| 选择中 | 用户点击 `选择文件夹` | 选择中 | 打开目录选择器 |
| 选择中 | 用户完成选择并确认 | 导入中 | 创建入库任务，展示进度 |
| 选择中 | 用户取消选择 | 未导入 | 回到默认冷启动态 |
| 导入中 | 所有图片入库完成 | 导入完成 | 写入首次导入承接标记，刷新首页数据 |
| 导入中 | 用户离开页面或退出应用 | 导入中断 / 恢复提示 | 按现有恢复机制处理 |
| 导入完成 | 用户关闭提示或开始真实搜索 | 普通首页 / 搜索结果页 | 清除一次性承接提示 |

#### 交互说明
**1. 冷启动主按钮**

页面元素：
- 主按钮：`导入图片`
- 轻弹层动作项：`选择图片`、`选择文件夹`
- 次级入口：`去图库管理`（仅在需要时出现，可放在更多菜单或导入中状态）

交互行为：
- 点击主按钮后，系统应在当前按钮附近弹出轻量动作层，而不是直接跳页
- 用户点击 `选择图片` 后，系统打开支持多选的图片文件选择器
- 用户点击 `选择文件夹` 后，系统打开目录选择器
- 用户如果主动选择 `去图库管理`，系统才切换到展开态并进入图库管理页

文案要求：
- 冷启动主按钮禁止继续使用会造成歧义的“进入图库导入流程”式行为
- 轻弹层只承载导入来源选择，不承载“去图库管理”等第二层导航
- 若产品仍保留跳转入口，其文案必须明确表达“管理图库”而不是“导入”

**2. 选择中状态**

页面元素：
- 状态标题：`正在等待你完成导入选择`
- 次按钮：`重新选择`
- 辅助入口：`去图库管理`

交互行为：
- 轻弹层关闭前，页面保持原位，不切到空白页或图库管理页
- 原生选择器尚未返回时，页面保持原位，不切到空白页或图库管理页
- 若选择器被取消，页面回到未导入状态

**3. 导入中状态**

页面元素：
- 进度条
- 已导入数量 / 总数
- 当前处理项提示
- 可选恢复提示

交互行为：
- 导入中时，用户仍可停留在当前页观察进度
- 若已有可展示的图片结果，可逐步替换弱化演示图，强化“图库正在成形”的感知
- 若用户中途退出应用，下次启动应通过既有恢复机制提示“上次有图片还没整理完”

**4. 导入完成状态**

页面元素：
- 一次性承接提示
- 可直接点击的示例词
- 已导入结果或最近常用候选池

交互行为：
- 导入完成后，系统应停留在首页 / 搜索页，不自动切到图库管理页
- 点击示例词应直接触发真实搜索
- 用户关闭提示、输入搜索或切换页面后，承接提示不再持续出现

**5. 图库管理页入口调整**

页面元素：
- 一级导航：`图库管理`
- 主操作按钮：`添加图片`
- 次操作按钮：`添加文件夹`
- 搜索失败反馈内：`去图库管理`
- 更多菜单中的次级入口：`管理图库`

交互行为：
- 图库管理页保留两个直接按钮，不通过轻弹层增加额外一步操作
- 图库管理入口继续保留，但不再承担冷启动主按钮职责
- 搜索失败时进入图库管理，应尽量带上默认视图，如 `recent` 或 `issues`

#### 窗口形态规则
| 场景 | 默认窗口形态 | 说明 |
|-----|-------------|-----|
| 冷启动首页 | `sidebar` | 保持聊天伴随型启动观感 |
| 快速导入 | 保持当前形态 | 导入是任务动作，不是强制切换形态的理由 |
| 图库管理 | `expanded` 优先 | 需要更大空间做治理和批量处理 |
| 搜索失败后修复 | `expanded` | 与图库管理工作页职责一致 |

#### 数据与状态要求
| 数据项 | 用途 | 最低要求 |
|-------|------|---------|
| `home_import_state` | 冷启动导入状态机 | 至少支持 `idle` / `picking` / `importing` / `imported` |
| `post_import_home_hint` | 首次导入承接标记 | 支撑一次性承接提示 |
| `pending_import_task_count` | 导入中断恢复提示 | 与现有恢复能力复用 |
| `gallery_view_mode` | 图库管理默认视图 | 支撑从搜索失败进入对应治理视图 |

补充要求：
- 冷启动导入状态不得与搜索结果状态混用
- 图库管理页数据源与首页导入状态应保持解耦，避免首页仅为导入而强依赖图库页加载完成

#### 边界情况处理
| 场景描述 | 触发条件 | 系统行为 | 提示信息 |
|---------|---------|---------|---------|
| 用户取消选择 | 打开选择器后未选中文件 | 回到未导入状态 | 无 |
| 导入过程中部分图片失败 | 部分图片索引失败或路径无效 | 继续处理其他图片，并记录失败数量 | `已导入 X 张，Y 张处理失败` |
| 导入过程中用户退出应用 | 应用关闭时仍有待处理任务 | 下次启动显示恢复提示 | `上次有图片还没整理完` |
| 首次导入完成但无可展示首页图 | 入库成功但首页候选池暂未生成 | 展示承接提示和基础空态，不阻塞搜索 | `已导入完成，现在可以开始搜索` |
| 用户在冷启动页主动要做批量治理 | 用户从更多菜单点击 `管理图库` | 切换到展开态图库管理页 | 无 |
| 搜索失败跳转后异常视图为空 | `issues` 无数据 | 回退到 `recent` 或 `all` | `当前没有异常图片，可先看看最近新增` |

| 埋点事件 | 触发时机 | 采集参数 | 用途说明 |
|---------|---------|---------|---------|

#### 迭代计划
- 后续可评估是否在快速导入中补充拖拽导入，但前提是不破坏当前冷启动结构
- 后续可评估在导入中状态里展示最近完成的真实缩略图，进一步强化“正在成库”的反馈
- 后续可评估是否在普通首页和冷启动页共用同一导入组件，减少状态分叉

#### 验收标准
**功能指标：**
- [ ] 冷启动首页主按钮点击后不再直接跳转到图库管理页
- [ ] 冷启动首页主按钮点击后先打开轻弹层，供用户选择 `选择图片` 或 `选择文件夹`
- [ ] 冷启动首页存在明确的四状态：未导入、选择中、导入中、导入完成
- [ ] 首次导入完成后，用户停留在首页 / 搜索页，并看到一次性承接提示
- [ ] 图库管理页保留 `添加图片` 和 `添加文件夹` 两个独立按钮，不引入额外弹层步骤
- [ ] 图库管理页继续保留导入、最近新增、异常图片和批量治理能力
- [ ] 搜索失败时，页面仍提供 `去图库管理` 的明确修复入口

**体验指标：**
- [ ] 用户能把冷启动主按钮理解为“开始导入”，而不是“进入另一个页面”
- [ ] 首次导入不强制打断聊天伴随型侧边栏使用路径
- [ ] 用户不会在首次使用时被 `最近新增`、`异常图片` 等治理概念打断
- [ ] 图库管理页职责对用户更清晰，更像治理页而不是首次引导页

**可靠性指标：**
- [ ] 用户取消选择、部分导入失败、任务中断恢复等边界场景都有稳定反馈
- [ ] 冷启动导入状态与图库管理数据源解耦，不因图库页未加载而失真
- [ ] 搜索页、图库管理页和窗口形态之间的文案与职责不互相矛盾

#### 假设
- 假设当前版本继续沿用现有添加图片 / 添加文件夹能力，不新增新的底层导入协议
- 假设冷启动快速导入在技术上可直接复用现有图库导入命令，而不必先进入图库管理页才能调用

#### 开放问题
- 导入中状态是否允许用户同步发起真实搜索，当前建议首次导入阶段先弱化该能力，避免用户在图库尚未准备好时得到失真预期
