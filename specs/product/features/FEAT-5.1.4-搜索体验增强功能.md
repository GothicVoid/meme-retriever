#### 功能描述
在现有记忆式搜索和结果列表能力可用的基础上，补充一轮以“更容易说对话、更容易看懂结果、更容易从失败中继续下一步”为目标的搜索体验增强。该功能不改变搜索引擎的核心召回路线，而是重点优化用户在搜索框、失败反馈、结果列表文案和历史复用链路中的可理解性与可恢复性。

本 spec 属于 `5.1.1 记忆式搜索功能` 与 `5.1.2 结果列表展示功能` 的 P1 迭代补充，主要约束前台交互、用户态文案和少量新数据能力，不单独引入新的搜索模式页面。

#### 设计目标
- 用户第一次进入搜索页时，能通过 placeholder、示例词或历史下拉快速理解“应该怎么搜”
- 用户搜索失败时，页面必须提供明确下一步，而不是只给一句泛化安慰文案
- 用户看到结果时，优先理解“这张为什么出现”和“这一组结果大概有多靠谱”，不需要理解算法术语
- 高频用户可以通过搜索历史下拉快速重搜、改搜和清理单条记录

#### 范围说明
**本期做什么：**
- 升级搜索框 placeholder 与示例引导文案
- 增加搜索框聚焦态的最近搜索下拉
- 重做无结果与弱相关结果场景的失败反馈区
- 将结果分区表达从算法术语改为用户语言
- 收敛排序解释文案，只保留用户能理解的人话理由

**本期不做什么：**
- 不改动检索主路选择策略和底层打分公式
- 不引入搜索语法、高级筛选、权重调节入口
- 不实现跨设备同步的搜索历史
- 不在本期增加“智能改写查询”自动生成能力，仅提供静态引导建议

#### 依赖与影响范围
- 依赖 `5.1.1` 已定义的记忆式搜索主流程与空查询切换规则
- 依赖 `5.1.2` 已定义的结果卡解释字段和高/较/弱相关分层
- 与 `5.1.3` 首页启动态共享搜索框，但首页示例词与本 spec 的聚焦态历史下拉不得互相覆盖
- 新增本地搜索历史读写能力，用于首页最近搜索与搜索框下拉共用

#### 流程说明
1. 用户进入搜索页或在首页搜索框聚焦
2. 系统展示升级后的 placeholder；若当前输入为空且存在历史记录，则显示最近搜索下拉
3. 用户输入查询后，系统沿用现有 debounce 触发检索
4. 检索返回后，系统按统一相关度阈值决定进入“正常结果态”“弱相关建议态”或“无结果态”
5. 在正常结果态下，结果列表使用用户化分区文案和简化排序理由展示
6. 在弱相关建议态或无结果态下，页面展示明确的下一步建议，并保留查看低相关结果、查看最近用过或重用历史搜索的入口
7. 用户可点击最近搜索词一键重搜，也可删除单条历史记录
8. 用户成功触发非空搜索后，系统更新搜索历史顺序；空白词不写入历史

#### 交互说明
**1. 搜索框输入引导**

页面元素：
- 搜索输入框
- placeholder 文案
- 静态示例词或轻量轮播示例词
- 聚焦态最近搜索下拉

默认文案要求：
- placeholder 应从 `搜索表情包` 升级为 `搜台词、角色、动作、场景`
- 搜索框附近应提供 3~5 个示例词，优先使用短词组而非完整句子
- 示例词建议与首页保持同一语义方向，但允许按页面宽度裁剪数量

交互行为：
- 输入框为空并获得焦点时，若存在最近搜索，则展示下拉列表
- 用户开始输入后，下拉优先切换为与当前输入相关的历史匹配项；若本期不做匹配过滤，则至少展示最近 5 条
- 点击某条历史记录后，立即填入并触发搜索
- 每条历史记录右侧应提供删除入口，仅删除该条记录，不影响其他记录
- 点击输入框外区域、按 `Esc` 或开始执行搜索后，收起下拉

**2. 搜索失败反馈**

失败反馈区适用于两类场景：
- 完全无结果
- 仅存在弱相关结果，且默认结果区为空

页面元素：
- 失败标题
- 简短原因说明
- 4~5 条下一步建议
- 可选快捷入口：`查看最近用过`、`展开低相关结果`

默认建议文案池：
- `试试图片里的原文`
- `试试角色名 + 动作`
- `试试更短的关键词`
- `试试更长一点的描述`
- `看看最近用过`

交互行为：
- 当无结果时，应展示失败反馈区，不直接留空白
- 当只有弱相关结果时，应先展示失败反馈区，再提供 `展开低相关结果` 按钮
- 若最近使用区有数据，失败反馈区应提供 `看看最近用过` 快捷入口，点击后滚动到首页对应模块或打开最近使用弹层；本期建议优先复用首页模块滚动
- 若用户点击某条建议，不要求自动改写查询；可先以静态提示文案方式呈现

**3. 结果分区用户化表达**

本功能不改变 `高相关 / 较相关 / 弱相关` 的内部阈值，只改变默认对用户展示时的标题和说明文案。

映射规则：

| 内部等级 | 用户可见标题 | 展示规则 |
|---------|-------------|---------|
| 高相关 | 最像你要找的 | 默认展示 |
| 较相关 | 可能也对 | 默认展示 |
| 弱相关 | 不太确定 | 默认折叠，用户手动展开 |

交互要求：
- 区块标题应优先使用用户标题，不在普通模式暴露“高相关 / 较相关 / 弱相关”术语
- 当页面没有明显分区切割，仅通过连续流展示时，也应在弱相关折叠处使用 `不太确定，展开看看` 一类文案
- 开发调试模式可继续显示内部等级术语，但不得污染普通模式

**4. 排序理由简化**

普通模式下，只保留以下四类用户可理解理由：
- 命中文字
- 角色命中
- 图片内容接近
- 最近常用

展示规则：
- 每张卡最多展示 1 条主理由 + 0~1 条补充证据
- `最近常用` 只能作为辅助理由，不能成为搜索结果排前的唯一理由
- 当后端已返回具体命中词时，应优先展示如 `命中文字：撤回消息`
- 当理由属于角色匹配时，优先展示 `角色命中：阿布`
- 当仅有语义接近信号时，展示 `图片内容接近`

禁止项：
- 不展示 `OCR 0.82 / 语义 0.71 / 标签 0.13` 等底层分值
- 不展示 `高相关度匹配`、`多模态综合得分` 这类工程化词汇作为主理由

**5. 搜索历史下拉**

页面元素：
- 最近搜索下拉列表
- 单条删除按钮
- 空态提示

交互行为：
- 触发条件：输入框聚焦且当前输入为空或可匹配到历史词
- 默认展示最近 5 条，按最近一次搜索时间倒序
- 同词重复搜索时，仅更新时间，不重复新增
- 删除单条后，下拉列表应立即刷新
- 当历史为空时，不展示下拉；如需占位，文案应简洁，如 `暂无最近搜索`

#### 数据要求
**新增或明确的数据字段：**

| 数据项 | 用途 | 最低要求 |
|-------|------|---------|
| `recent_searches` | 首页最近搜索与输入框下拉共用 | 保存搜索词、最近一次搜索时间 |
| `matched_ocr_terms` | 支撑“命中文字”展示 | 至少返回 1 个命中词或短语 |
| `matched_role_name` | 支撑“角色命中”展示 | 返回命中的角色主名 |
| `result_reason_type` | 支撑前端映射理由 | 枚举值至少覆盖 `ocr` / `role` / `semantic` / `popularity` |

补充要求：
- 搜索历史本地存储即可，不要求服务端同步
- 搜索历史需与首页最近搜索共用一份数据源，避免两套排序不一致
- 空白字符、仅标点查询默认不写入历史

#### 边界情况处理
| 场景描述 | 触发条件 | 系统行为 | 提示信息 |
|---------|---------|---------|---------|
| 搜索历史为空 | 用户从未执行过非空搜索 | 不展示下拉列表 | 无 |
| 输入为空并聚焦 | 用户点击输入框但未输入内容 | 展示最近搜索下拉或首页默认示例 | 无 |
| 输入仅空格 | 用户输入后又仅剩空白字符 | 不发起搜索，不写入历史 | 无 |
| 无结果 | 所有候选都被过滤 | 展示失败反馈区 | `没找到这类图片，换个说法试试` |
| 仅有弱相关结果 | 所有结果均落入弱相关 | 默认展示失败反馈区，并提供展开低相关入口 | `先试试更准确的描述，也可以看看不太确定的结果` |
| 后端未返回具体命中词 | 只有抽象理由类型，无证据字段 | 回退到泛化文案 | `命中文字` / `角色命中` / `图片内容接近` |
| 历史记录过长 | 本地历史超过上限 | 按时间淘汰旧记录，建议最多保留 20 条 | 无 |
| 删除最后一条历史 | 用户删掉仅有的一条搜索记录 | 关闭下拉或展示空态 | `暂无最近搜索` |

#### 数据埋点
| 埋点事件 | 触发时机 | 采集参数 | 用途说明 |
|---------|---------|---------|---------|
| search_history_dropdown_show | 输入框聚焦并展示历史下拉 | history_count | 评估历史下拉曝光量 |
| search_history_click | 用户点击历史词重搜 | query, rank_position | 评估历史复用率 |
| search_history_delete | 用户删除单条历史 | query, remain_count | 评估清理需求 |
| search_zero_result_show | 展示无结果反馈区 | query | 评估搜索失败率 |
| search_low_confidence_show | 展示弱相关建议态 | query, weak_result_count | 评估弱相关触发率 |
| search_guidance_click | 用户点击失败反馈区中的快捷入口 | action_type, query | 评估哪类引导更有效 |

#### 迭代计划
- 后续可增加基于当前输入前缀过滤历史词
- 后续可补充“直接替换为建议词重新搜索”的半自动改写能力
- 后续可根据真实点击数据继续校准 `最像你要找的 / 可能也对 / 不太确定` 的文案和出现方式
- 后续可在失败反馈区加入最近搜索、最近使用和示例词的混合推荐卡片，但本期不做复杂推荐

#### 验收标准
**功能指标：**
- [ ] 搜索框 placeholder 已从泛化文案升级为搜索线索引导文案
- [ ] 输入框聚焦时可展示最近搜索下拉
- [ ] 点击历史词可一键重搜
- [ ] 支持删除单条搜索历史
- [ ] 无结果时展示明确失败反馈区，而不是仅显示一条弱提示
- [ ] 仅有弱相关结果时，默认优先展示失败反馈，并支持展开低相关结果
- [ ] 结果列表普通模式下使用 `最像你要找的`、`可能也对`、`不太确定` 等用户化表达
- [ ] 结果卡主理由限制为用户可理解的四类人话解释

**体验指标：**
- [ ] 新用户第一次搜索时，能从搜索框文案或示例中理解可输入台词、角色、动作、场景
- [ ] 用户在搜索失败后，能看到至少 3 条明确下一步建议
- [ ] 普通用户无需理解算法术语，也能大致判断某张图为什么出现
- [ ] 高频用户可以在 1 次聚焦 + 1 次点击内重用最近搜索

**可靠性指标：**
- [ ] 空白词不会写入搜索历史
- [ ] 搜索历史顺序与首页最近搜索保持一致
- [ ] 删除单条历史后，不影响其余历史记录和首页最近搜索渲染
- [ ] 当后端缺失具体命中词字段时，前端仍可稳定回退展示泛化理由

#### 假设
- 假设搜索历史以本地持久化方式实现，并可被首页与搜索框下拉共享
- 假设 `5.1.2` 中的低相关折叠能力已存在，本 spec 仅补充其用户引导表达
- 假设失败反馈区优先以静态建议文案落地，不依赖大模型或动态改写服务

#### 开放问题
- 搜索历史是否记录所有非空查询，还是仅记录真正发起检索且有返回的查询；当前建议记录所有有效非空查询
- 失败反馈区中的 `看看最近用过` 是否直接滚动到首页模块，还是在搜索页内以内联区块呈现；当前建议优先滚动复用首页模块
- 搜索框示例词是否采用静态展示还是轮播展示；当前建议默认静态展示，降低干扰
