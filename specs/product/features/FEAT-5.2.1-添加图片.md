#### 功能描述
支持多种方式将图片添加到图库，采用**路径引用**方式存储，不复制文件，自动触发索引构建。使用图片时检查文件状态，若文件不存在则提示用户。

在交互层面，系统继续保留：
- `添加图片`
- `添加文件夹`

两个独立入口，用于适配原生系统选择器的差异和用户的明确任务预期。

补充规则如下：
- 图库管理页继续使用 `添加图片` 和 `添加文件夹` 两个独立按钮
- 冷启动页、搜索页等启动型场景可使用单一主按钮 `导入图片`
- 冷启动页点击 `导入图片` 后，应通过轻弹层让用户二选一：`选择图片` 或 `选择文件夹`

在实现层面，系统应将两类入口统一收敛到同一套导入协议、导入状态机和索引任务管线中。也就是说：
- 图片导入和文件夹导入只在“输入源采集”阶段不同
- 进入导入执行阶段后，应共享同一套导入任务模型
- 不应长期维护两套重复的前端进度逻辑和两套后端索引分发协议

#### 设计目标
- 用户能明确区分 `选择图片` 和 `选择文件夹` 两种操作，不被混合文件选择器的不稳定行为干扰
- 前端搜索页、冷启动页和图库管理页可以复用同一套导入状态与进度展示能力
- 后端能够统一接收文件与目录输入，展开、去重后进入同一条索引任务管线
- 后续若新增拖拽导入或混合来源导入，不需要再新增第三套导入协议

#### 范围说明
**本期做什么：**
- 保留 `添加图片` 和 `添加文件夹` 两个交互入口
- 允许冷启动页使用单一 `导入图片` 主按钮，并通过轻弹层分流为 `选择图片` / `选择文件夹`
- 统一前端导入状态机与进度逻辑
- 统一后端导入协议，支持文件和目录作为输入项
- 统一总数计算规则，由后端返回真实待入库图片总数

**本期不做什么：**
- 不强求系统文件选择器在一次对话框中同时多选文件和文件夹
- 不新增剪贴板导入、网络导入等新来源
- 不改变路径引用、不复制原文件的基础存储策略

#### 流程说明
1. 用户点击按钮选择添加方式
2. 若用户选择 `添加图片`，系统打开支持多选的图片文件选择器
3. 若用户选择 `添加文件夹`，系统打开目录选择器
4. 前端将用户选择结果统一映射为导入输入项列表
5. 后端对输入项列表进行展开：
   - 文件项直接纳入候选路径
   - 目录项递归扫描图片文件后纳入候选路径
6. 系统对展开后的候选路径做格式过滤、去重和有效性校验
7. 系统计算真实待入库图片总数，并进入统一导入状态机
8. **存储路径引用：将文件路径记录到数据库，不复制文件**
9. 触发统一索引构建（OCR、特征提取）
10. 更新图库列表，显示进度
11. 后台任务持久化：索引构建任务记录到本地队列
12. 进度保存：每完成一张图片的索引，更新进度记录
13. 恢复机制：下次启动时检查未完成任务
14. 完成后 Toast 提示 `已添加 X 张图片`

#### 导入协议统一方案

**1. 交互分流**

页面继续提供两个入口：
- `添加图片`
- `添加文件夹`

冷启动等启动型页面可使用一个上层入口：
- `导入图片`
  - `选择图片`
  - `选择文件夹`

原因如下：
- 不同平台和原生文件选择器对“文件 + 文件夹混选”的支持不稳定
- 用户在认知上也更容易理解“我现在是在选图片”还是“我现在是在选目录”
- 冷启动页适合先收敛为单一主动作，再在轻弹层做二选一
- 图库管理页适合直接暴露两个按钮，减少治理场景的额外一步操作

**2. 协议统一**

前后端应统一使用“导入输入项”作为桥梁模型，而不是继续长期维持：
- 一套 `addImages(paths)` 协议
- 一套 `addFolder(path)` 协议

建议统一后的输入模型如下：

```text
ImportEntry
├── kind: file | directory
└── path: string
```

系统行为要求：
- 前端选择器只负责产出 `ImportEntry[]`
- 后端统一接收 `ImportEntry[]`
- 后端负责展开、扫描、过滤、去重和统计总数
- 前端只根据后端返回的真实总数展示统一进度

**3. 状态统一**

前端不应再分别维护两套近似重复的导入进度逻辑。统一后的状态机至少应支持：

```text
idle -> preparing -> importing -> completed / partial_failed / failed
```

含义如下：
- `idle`：未开始导入
- `preparing`：正在等待后端展开目录、过滤和统计
- `importing`：正在索引图片
- `completed`：全部成功
- `partial_failed`：部分成功、部分失败
- `failed`：导入任务整体失败

**4. 总数统一**

当前图片导入与文件夹导入在总数来源上不一致：
- 图片导入通常直接以用户选中文件数作为总数
- 文件夹导入则依赖目录扫描结果作为总数

统一后，系统应以“后端展开并过滤后的真实图片数”作为唯一总数来源。

这样可以正确处理以下情况：
- 文件夹内混入非图片文件
- 多目录扫描后出现重复路径
- 文件和目录混合输入时目录展开数与用户表面选择数不一致
- 某些路径在展开阶段即被判定为无效

#### 数据持久化设计

**任务队列表结构：**
```
task_queue
├── task_id (主键)
├── task_type (添加图片/构建索引)
├── status (待处理/进行中/已完成/已失败)
├── total_count (总数)
├── processed_count (已处理数)
├── file_list (待处理文件列表JSON)
├── error_message (错误信息，失败时记录)
├── create_time
└── update_time
```

**恢复逻辑：**
1. 应用启动时查询`status='进行中'`的任务
2. 对比`file_list`和已索引图片，计算剩余任务
3. 弹窗询问用户：「继续添加」或「取消任务」
4. 如选择继续，从断点处恢复执行

#### 去重机制设计

**去重策略：**
- **轻量级去重**：通过文件路径、文件大小、修改时间等元数据快速筛选
- **文件哈希去重**：计算SHA-256哈希值，确保内容完全相同的图片不被重复添加

**性能优化策略：**
- **批量添加优化**：先通过轻量级方式初步筛选，再对潜在重复文件计算哈希值
- **单张添加**：直接进行完整去重检查
- **索引优化**：在file_path和file_size字段上建立复合索引，提升查询效率

**去重流程：**

**批量添加场景：**
1. **第一步**：轻量级筛选
   - 检查文件路径是否已存在于数据库
   - 检查文件大小和修改时间组合是否与现有记录冲突
   - 快速过滤明显不重复的文件
2. **第二步**：哈希值计算（仅对潜在重复文件）
   - 对轻量级筛选后的可疑文件计算SHA-256哈希值
   - 查询数据库中是否存在相同哈希值
   - 如存在相同哈希值，跳过该文件并记录重复

**单张添加场景：**
1. **第一步**：检查文件路径是否已存在于数据库
2. **第二步**：计算文件SHA-256哈希值
3. **第三步**：查询数据库中是否存在相同哈希值
4. **第四步**：如存在相同哈希值，跳过该文件并记录重复

**数据库设计：**
```
images
├── image_id (主键)
├── file_path (文件路径，索引)
├── file_size (文件大小，索引)
├── file_modified_time (文件修改时间，索引)
├── file_hash (SHA-256哈希值，唯一索引)
├── file_status (文件状态：正常/丢失/移动)
├── last_check_time (最后检查时间)
└── add_time (添加时间)
```

**索引设计：**
- 复合索引：(file_path, file_size, file_modified_time) - 支持轻量级去重查询
- 唯一索引：file_hash - 支持哈希值快速查找

#### 文件状态检查机制

**检查策略：**
- **访问时验证**：用户使用图片（复制、查看详情）时验证文件是否存在
- **搜索时过滤**：检索结果中过滤掉文件不存在的记录

**文件丢失处理：**
- 复制时发现文件不存在：Toast提示"原文件已丢失，无法复制"
- 查看详情时发现文件不存在：显示错误页面，提供"重新定位"选项
- 搜索结果中：文件丢失的图片显示占位图，标记为"文件已丢失"

**重新定位功能：**
- 提供"重新定位"按钮，点击后打开文件选择器
- 用户可选择新的图片文件来替换丢失的记录
- 支持基于文件名和哈希值的智能匹配建议

#### 交互说明
**页面元素：**
- "添加图片"按钮：主操作按钮
- "添加文件夹"按钮：批量导入
- 冷启动页主按钮：`导入图片`
- 冷启动页轻弹层动作：`选择图片`、`选择文件夹`
- 进度条：多张图片时显示处理进度
- 全局进度指示器：固定在窗口顶部，显示后台任务进度
- 任务恢复对话框：应用启动时显示（如有未完成任务）

**交互行为：**
- 点击"添加图片"：打开系统文件选择器，支持多选
- 点击"添加文件夹"：打开文件夹选择器，递归扫描

- 点击轻弹层 `选择图片`：打开系统文件选择器，支持多选
- 点击轻弹层 `选择文件夹`：打开文件夹选择器，递归扫描

- 若未来首页冷启动、搜索页快捷导入或图库管理页都支持导入，应共用同一套导入状态组件，而不是各自维护独立进度实现
- 切换页面后，顶部常驻迷你进度条
- 点击进度条可跳转回添加进度详情
- 任务完成后Toast提示，进度条自动消失

**反馈提示：**
- 准备中：`正在整理导入项...`
- 添加中：进度条显示"正在处理X/Y张图片"
- 成功：Toast"已添加X张图片"
- 部分失败：Toast"成功X张，失败Y张，查看详情"

#### 技术约束说明
**1. 为什么不强行统一成一个系统选择器**

系统不要求在同一个原生文件选择器中同时选择：
- 多个文件
- 一个或多个文件夹

原因如下：
- 该能力跨平台支持不稳定
- 即便部分平台可行，也会显著增加兼容和测试成本
- 这类“强统一”会把系统选择器能力问题暴露给产品主路径，收益低于风险

因此本期推荐方案是：
- 交互入口分流
- 导入协议统一

**2. 为什么要统一后端协议**

统一后的后端协议能够带来以下收益：
- 目录扫描、文件过滤、去重、索引分发只保留一套逻辑
- 冷启动页、图库管理页、拖拽导入等不同来源都能共用同一命令
- 进度总数由后端统一计算，避免前端各自猜测
- 后续支持混合输入来源时不需要再次重构命令层

**3. 为什么要统一前端状态**

统一后的前端状态管理能够带来以下收益：
- 搜索页与图库页的导入体验保持一致
- 进度条、恢复提示、完成提示只需维护一套
- 更方便对接冷启动页的四状态导入闭环
- 降低后续改动时出现“图片导入修好了、文件夹导入漏改”的风险

#### 边界情况处理
| 场景描述 | 触发条件 | 系统行为 | 提示信息 |
|---------|---------|---------|---------|
| 格式不支持 | 选择非JPG/PNG/GIF/WEBP文件 | 跳过该文件，记录失败 | "不支持的格式：xxx.bmp" |
| 文件过大 | 单个文件>50MB | 跳过该文件 | "文件过大：xxx.png（限50MB）" |

| 重复添加 - 内容相同 | 文件哈希值已存在 | 跳过该文件 | "已存在相同内容图片" |
| 文件夹为空 | 选择的文件夹无图片 | 不执行操作 | "未找到可添加的图片" |
| 混合输入后无有效图片 | 输入项展开后均被过滤 | 不执行索引任务 | "未找到可添加的图片" |
| 文件已丢失（添加时） | 添加时发现文件不存在 | 跳过该文件，记录为丢失状态 | "文件不存在，已标记为丢失" |
| 文件已丢失（使用时） | 复制/查看详情时发现文件不存在 | 显示错误提示，提供重新定位选项 | "原文件已丢失，无法复制" / "原文件已丢失，请重新定位" |

| 关闭程序 | 添加过程中用户关闭应用 | 保存当前进度，下次启动时询问是否继续 | "检测到未完成的添加任务，是否继续？" |
| 程序崩溃 | 添加过程中程序异常退出 | 下次启动自动恢复，跳过已完成的图片 | "上次添加未完成，正在继续处理..." |
| 重复启动任务 | 用户在后台任务进行时再次添加 | 根据并发控制规则处理 | 详见并发控制规则表 |
| 索引数据空间不足 | 构建索引时磁盘空间不足 | 暂停任务，提示清理空间 | "索引空间不足，请清理后重试" |
| 输入项含文件和目录 | 来源层未来支持混合输入 | 统一展开后执行同一任务 | 无 |

### 并发控制规则

#### 并发操作互斥规则表

为确保系统稳定性和数据一致性，不同操作之间的并发执行需要遵循以下规则：

| 操作A | 操作B | 是否允许并发 | 说明 |
|------|------|-------------|------|
| 添加图片 | 添加图片 | 否 | 串行执行，避免索引构建冲突 |
| 添加图片 | 搜索 | 是 | 搜索仅访问已索引的图片，不影响正在构建的索引 |
| 添加图片 | 删除 | 否 | 删除操作需要等待添加完成，避免数据不一致 |
| 添加图片 | 编辑标签 | 否 | 标签编辑需要等待索引构建完成 |
| 搜索 | 搜索 | 是 | 多个搜索查询可并发执行 |
| 搜索 | 删除 | 是 | 搜索和删除操作数据隔离，可并发 |
| 搜索 | 编辑标签 | 是 | 搜索和标签编辑互不影响 |
| 删除 | 删除 | 是 | 多个删除操作可并发执行 |
| 删除 | 编辑标签 | 否 | 删除和标签编辑可能产生冲突，需串行 |
| 编辑标签 | 编辑标签 | 是 | 多个标签编辑操作可并发执行 |

#### 并发控制实现机制

**1. 任务队列管理**
- 添加图片任务使用全局队列，确保串行执行
- 其他操作根据互斥规则动态调度

**2. 数据库锁机制**
- 读操作（搜索）使用共享锁，允许多个并发读取
- 写操作（添加、删除、编辑）使用排他锁，确保数据一致性

**3. 状态同步**
- 索引构建期间，新图片对搜索不可见
- 删除操作立即生效，搜索时过滤已删除记录

**4. 用户反馈**
- 并发受限时显示相应提示信息
- 后台任务状态实时更新

#### 边界情况处理

| 场景 | 处理方式 | 用户提示 |
|------|---------|---------|
| 添加任务执行中再次添加 | 加入队列排队 | "已有添加任务进行中，新任务将在完成后开始" |
| 搜索时正在添加图片 | 仅搜索已索引图片 | 无提示，正常返回结果 |
| 删除时正在添加图片 | 等待添加完成后执行 | "正在等待添加任务完成..." |
| 编辑标签时正在添加图片 | 等待添加完成后执行 | "正在等待索引构建完成..." |

#### 开放问题
- 统一后的后端命令是替换现有 `add_images` / `add_folder`，还是先新增一个统一命令再逐步迁移，当前建议优先采用渐进迁移，避免一次性改动过大
- 若未来恢复拖拽导入，拖入目录和拖入文件是否直接映射到同一 `ImportEntry[]` 协议，当前建议是直接映射到同一协议，避免再次分叉导入模型
