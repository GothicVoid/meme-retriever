#### 功能描述
在首页启动态、搜索体验增强和导航重构已完成的基础上，补充一轮面向高频用户的效率增强与关键状态统一反馈。目标不是新增复杂功能中心，而是围绕“导入 -> 搜索 -> 复制 -> 再找回”这条主链路，减少重复操作、降低中断成本，并让用户在加载、失败、恢复等关键场景下始终知道当前发生了什么、下一步能做什么。

本 spec 聚焦会影响产品行为、交互流程和验收口径的 P3 内容，不覆盖纯视觉样式规范。字体、留白、卡片视觉、颜色与层级统一应另行沉淀为设计规范，不在本功能 spec 中展开。

#### 设计目标
- 高频用户可通过键盘和就近入口更快完成“继续搜索、切换结果、复制图片、再次找回”
- 用户重新进入应用后，能够通过少量、直观的入口快速找回刚搜过的词和刚用过的图片
- 鼠标与键盘用户应共享同一套能力模型，即都能稳定完成“预览确认”“直接复制”“进入详情”
- 图片预览、加载、空状态、错误态、文件丢失和入库中断恢复等关键场景必须提供明确反馈，而不是只留下空白区域
- 用户在主链路中不需要理解内部处理细节，但能判断当前是在加载、失败、等待修复还是可以继续下一步

#### 范围说明
**本期做什么：**
- 增加全局快捷操作，包括快速聚焦搜索、搜索结果键盘导航和快速预览
- 为默认快捷键提供可见提示，降低中频操作的记忆成本
- 保留并强化首页已有的继续使用入口，聚焦“最近搜索”和“最近使用”
- 统一结果卡在鼠标与键盘下的主路径，明确“悬浮轻预览 / 正式预览 / 点击复制 / 详情”分层
- 统一关键状态反馈，包括加载态、空状态、错误态、文件丢失提示和入库中断恢复提示

**本期不做什么：**
- 不在本期引入复杂命令面板、可自定义快捷键系统或全局工作台
- 不新增“最近复制”“最近成功搜索”等独立首页模块，避免与现有“最近使用”“最近搜索”形成语义重叠
- 不实现基于用户行为的智能推荐排序算法升级，仅做可解释的最近记录与继续使用入口
- 不在本期输出完整视觉设计规范，不规定字体、颜色、阴影等具体样式 token
- 不做跨设备同步的搜索或使用记录

#### 主链路说明
**主流程 1：高频用户快速找图并复制**
1. 用户通过快捷键快速聚焦搜索框
2. 用户输入查询，系统展示结果列表
3. 用户通过鼠标悬浮轻量确认图片，或使用键盘在结果间移动焦点
4. 用户在需要时可进入正式快速预览放大确认
5. 用户执行复制后，系统给出成功提示
5. 系统通过“最近搜索”和“最近使用”帮助用户在后续会话继续找回

**主流程 2：用户从首页继续上次使用**
1. 用户重新进入应用或清空当前查询
2. 首页展示最近搜索和最近使用等模块
3. 用户点击某条记录后直接继续搜索或重新复制目标图片

**主流程 3：用户遇到异常或中断**
1. 用户在搜索、预览、详情、图库管理或入库过程中遇到加载、失败或文件缺失
2. 页面展示对应状态反馈，包括原因说明、下一步建议和可执行入口
3. 若是入库任务中断，系统应提示恢复方式或继续处理入口
4. 用户完成修复后，可返回主链路继续搜索或复制

#### 交互说明
**1. 全局快捷操作**

页面元素：
- 搜索快捷键提示
- 结果列表焦点态
- 悬浮轻预览
- 快速预览层
- 快捷键帮助信息
- 悬浮快捷入口

交互行为：
- 用户按 `/` 或 `Ctrl+K` 时，系统应将焦点移动到主搜索框，并选中当前输入内容
- 当搜索结果列表存在可见结果时，用户可使用方向键在结果卡之间移动焦点
- 焦点卡应具备清晰的可见态，不要求鼠标悬停
- 键盘路径应与鼠标路径共享同一套能力集合，但允许触发方式不同
- 用户按 `Enter` 时，默认对当前焦点卡执行复制
- 用户按 `Space` 或 `Shift+Enter` 时，打开当前焦点卡的正式快速预览
- 用户按 `Esc` 时，优先关闭快速预览；若当前无预览层，则收起相关浮层并保留当前查询
- 页面应为默认快捷键提供就地可见提示，而不是要求用户自行记忆；搜索框、结果区和正式快速预览中都应展示与当前上下文相关的快捷键
- 默认常驻提示应只展示最关键的 1 至 3 个快捷键，完整快捷键说明可通过帮助入口或设置说明区查看
- 悬浮快捷入口若存在，应只承接高频动作，如 `继续搜索`、`最近搜索`、`最近使用`，不得演化为第二套导航
- 触屏和鼠标场景下，悬浮快捷入口应不遮挡主内容区域和结果卡关键操作

**2. 继续使用入口**

页面元素：
- 最近搜索区
- 最近使用区
- 继续使用入口

交互行为：
- 首页继续使用区域应优先复用用户已经能理解的概念，不新增需要解释的记录类型
- 最近搜索用于帮助用户找回刚输入过的查询词，点击后应立即填入并触发搜索
- 最近使用用于帮助用户找回刚实际用过的图片，点击后应再次执行复制
- 若最近搜索为空，则不展示该区块，不使用空壳占位
- 若最近使用为空，则不展示该区块，不使用空壳占位
- 若后续需要基于复制、点击或结果有效性调整排序，应作为最近使用或最近搜索的内部排序信号，不单独暴露为新模块

**3. 图片预览分层与统一交互**

页面元素：
- 悬浮轻预览
- 预览弹层
- 大图区域
- 基础元信息区
- 快捷操作区

交互行为：
- 结果卡在搜索结果、最近使用、常用表情等图片入口中的核心能力必须一致：轻量预览、正式预览、复制、查看详情
- 鼠标悬浮结果卡时，系统应在短延迟后展示轻量预览，建议延迟 120ms 至 200ms，避免鼠标扫过时频繁闪动
- 轻量预览用于“停一下就看一眼”，不得抢占焦点、锁定背景滚动或阻断继续移动鼠标
- 轻量预览是鼠标路径下为降低移动与确认成本提供的轻量优化，不要求在键盘路径下提供等价层级
- 鼠标点击结果卡时，应直接执行复制，并保留明确的成功反馈
- 鼠标路径必须提供进入正式快速预览的稳定入口，可通过轻量预览中的按钮、卡片显性按钮或右键菜单承接
- 键盘路径下，`Enter` 应直接复制，`Space` 或 `Shift+Enter` 应进入正式快速预览；键盘确认需求统一由正式快速预览承接，不再额外提供轻量预览快捷键
- 正式快速预览应优先用于“放大确认是否是这张”，而不是替代完整详情页
- 正式快速预览至少提供以下操作：复制、查看详情、在文件夹中显示、关闭
- 用户在正式快速预览中切换上下张时，应优先沿用当前来源列表顺序
- 尺寸不一的图片不得改变预览交互规则；系统应使用统一的预览容器承接图片，图片在容器内保持原比例缩放
- 超长图、超高图、小图和 GIF 在轻量预览与正式快速预览中都应遵守统一边界，不得撑破布局、遮挡相邻内容或导致浮层位置剧烈跳动
- GIF 在正式快速预览层中应正常播放；轻量预览可按性能策略选择播放或首帧，但同一页面内策略必须稳定一致
- 正式快速预览打开后，背景主页面不应发生误触滚动或焦点跳失

**4. 关键状态反馈统一**

覆盖页面：
- 首页 / 搜索页
- 图库管理页
- 单图详情页
- 导入与入库相关视图
- 快速预览层

状态要求：
- 加载态：必须说明正在处理什么，避免只有旋转图标
- 空状态：必须说明为什么为空，以及下一步建议
- 错误态：必须说明当前动作失败，并提供重试、返回或替代路径
- 文件丢失态：必须明确该图片原文件不存在，并提供重新定位或删除记录入口
- 入库中断恢复态：必须说明有未完成任务，并提供继续处理或放弃清理入口

统一交互规则：
- 状态文案应优先使用任务语言，例如 `正在整理新导入的图片`、`这张图片的原文件找不到了`
- 空状态不能只显示 `暂无数据`，应补充原因和下一步动作
- 错误态不能只显示技术异常字符串，应对外收敛为可理解提示
- 文件丢失态在搜索结果、最近使用和详情页中的对外语义必须一致
- 入库中断恢复提示应在用户重新打开应用或回到图库管理页时可见，不能只在任务中断当下闪过一次

#### 页面与状态规则
| 场景 | 触发条件 | 页面行为 | 可执行动作 |
|-----|---------|---------|-----------|
| 快速聚焦搜索 | 用户按 `/` 或 `Ctrl+K` | 聚焦主搜索框并高亮现有输入 | 直接输入、继续搜索 |
| 搜索结果键盘导航 | 结果列表存在可见结果 | 显示结果焦点态，支持方向键移动 | `Enter` 复制、`Space` 预览 |
| 快捷键提示 | 用户进入搜索框、结果区或正式快速预览 | 展示当前上下文相关的快捷键提示 | 查看提示、继续操作 |
| 悬浮轻预览 | 鼠标悬浮结果卡并停留短延迟 | 展示轻量预览，不锁滚动、不抢焦点 | 放大查看、继续移动、直接点击复制 |
| 复制成功 | 用户单击图片或在预览中执行复制 | 显示 `已复制` Toast | 继续复制 |
| 最近搜索展示 | 存在最近搜索记录 | 首页展示最近搜索 | 点击重搜 |
| 最近使用展示 | 存在最近使用记录 | 首页展示最近使用图片 | 再次复制、打开预览 |
| 正式快速预览 | 用户通过键盘或鼠标显性入口触发预览 | 打开预览层并锁定背景滚动 | 复制、详情、关闭 |
| 搜索页加载中 | 用户发起搜索且结果未返回 | 展示搜索中状态，不清空结构骨架 | 等待、取消输入 |
| 搜索无结果 | 无可用结果 | 展示失败反馈和下一步建议 | 改搜、去图库管理 |
| 图库为空 | 图库总数为 0 | 展示导入引导空状态 | 导入图片 |
| 原文件丢失 | 图片记录存在但文件不存在 | 展示统一文件丢失态 | 重新定位、删除记录 |
| 入库任务中断 | 上次导入未完成 | 展示恢复提示 | 继续处理、放弃并清理 |

#### 数据要求
| 数据项 | 用途 | 最低要求 |
|-------|------|---------|
| `preview_source_context` | 支撑快速预览翻页 | 标记当前预览来源列表与当前位置 |
| `index_recovery_state` | 支撑入库中断恢复提示 | 保存未完成任务状态、最近中断时间 |
| `missing_file_state` | 支撑跨页面统一文件丢失反馈 | 至少标记文件是否存在及可恢复动作 |

补充要求：
- 若后续引入复制、点击、结果有效性等行为信号，应优先作为 `recent_searches` 或 `recent_used` 的排序增强使用，而不是直接演化为新的用户可见区块
- 轻量预览与正式快速预览应共用同一图片来源上下文，避免用户在不同预览层看到顺序、内容或可执行动作不一致
- 预览层关闭后，应尽量恢复打开前的焦点位置，便于键盘连续操作

#### 边界情况处理
| 场景描述 | 触发条件 | 系统行为 | 提示信息 |
|---------|---------|---------|---------|
| 搜索框已聚焦 | 用户再次按快捷聚焦键 | 保持焦点，不重复触发动画 | 无 |
| 结果列表为空时按方向键 | 当前没有可导航结果 | 不进入焦点态 | 无 |
| 鼠标快速扫过多张卡片 | 悬浮停留时间不足 | 不展示轻量预览，避免频繁闪动 | 无 |
| 快速预览来源被刷新 | 用户打开预览后底层结果重新搜索 | 预览保持当前图片，不强制跳图 | `结果已更新`（可选） |
| 轻量预览内容过大 | 遇到超长图、超高图或异常比例图片 | 预览容器保持固定边界，图片按比例缩放，不撑破布局 | 无 |
| 预览图片文件丢失 | 打开预览时原文件已不存在 | 展示文件丢失态，关闭大图展示 | `这张图片的原文件找不到了` |
| 导入中途退出应用 | 入库任务尚未完成 | 下次启动时展示恢复提示 | `上次有图片还没整理完` |
| 恢复任务失败 | 继续处理时再次报错 | 保留恢复提示并提供放弃清理入口 | `恢复失败，请重试或清理这批记录` |

#### 迭代计划
- 后续可补充轻量命令面板，但前提是不替代现有主导航
- 后续可为常搜主题增加用户手动固定能力，但本期不做
- 后续可评估是否开放快捷键配置，但前提是默认快捷键已稳定、冲突处理规则已明确
- 后续若确认复制或有效结果行为确实能显著提升找回效率，可先在排序层验证，再决定是否需要新的用户可见入口
- 后续可将状态反馈样式抽象为统一组件规范，与设计规范文档联动

#### 验收标准
**功能指标：**
- [ ] 用户可通过快捷键快速聚焦主搜索框
- [ ] 搜索结果支持键盘焦点移动，并可通过键盘直接复制或打开快速预览
- [ ] 搜索框、结果区和正式快速预览中可见默认快捷键提示
- [ ] 鼠标悬浮结果卡时可触发轻量预览，点击结果卡时可直接复制
- [ ] 鼠标与键盘路径都可进入正式快速预览，且两者支持的核心能力一致
- [ ] 轻量预览仅作为鼠标路径优化存在，键盘路径不额外提供独立轻量预览层
- [ ] 首页可展示最近搜索，并支持一键重搜
- [ ] 首页可展示最近使用，并支持再次复制或打开预览
- [ ] 图片支持快速预览，且预览层可执行复制、查看详情和关闭
- [ ] 搜索、图库、详情、预览、导入流程中的加载态、空状态、错误态都有明确文案和动作入口
- [ ] 原文件丢失时，跨页面使用统一语义，并提供修复或清理动作
- [ ] 存在未完成入库任务时，用户重新进入应用后能看到恢复提示

**体验指标：**
- [ ] 高频用户可在更少鼠标操作下完成“聚焦搜索 -> 浏览结果 -> 复制图片”
- [ ] 习惯即时通讯表情交互的用户，无需理解新概念即可形成“悬浮看、点击用、需要时放大确认”的稳定预期
- [ ] 用户重新进入应用后，能通过少量且直观的入口快速找回刚搜过的词和刚用过的图片
- [ ] 用户在异常场景下不会面对无解释空白页，且能理解下一步动作

**可靠性指标：**
- [ ] 最近搜索不会混入空白查询
- [ ] 最近使用顺序与用户实际使用行为保持一致，不生成重复脏数据
- [ ] 不同比例图片进入轻量预览或正式快速预览时，不会导致布局跳变或交互行为改变
- [ ] 预览层关闭后，页面焦点和滚动位置可稳定恢复
- [ ] 入库中断恢复提示不会因页面跳转而意外丢失

#### 假设
- 假设复制行为仍以“写入系统剪贴板成功”为准
- 假设入库中断恢复状态可由本地持久化任务记录提供，不需要单独引入任务中心页面

#### 开放问题
- 快捷聚焦搜索的默认快捷键最终是否同时保留 `/` 与 `Ctrl+K`，还是仅保留一个主快捷键
- 轻量预览是否默认支持 GIF 播放，还是在性能压力较大时只展示首帧
- 常搜主题本期是否先只展示静态聚合词，还是延后到有足够真实数据后再开放
