#### 功能描述
本次改造将产品从“传统独立主窗口应用”收敛为更适合和即时通讯软件并存的聊天伴随型工作台。核心思路不是继续增加首页模块，而是让应用在侧边驻留时尽量少挡聊天窗口，在需要整理图库或做更大范围浏览时，再切换到展开态。

当前实现已经形成一套双态工作模式：
- `sidebar`：默认聊天伴随型侧边栏态，适合边聊天边找图
- `expanded`：展开管理态，适合查看更多结果和整理图库

与这一形态配套，搜索输入被固定在底部，首页主区被收敛为可直接点击使用的候选池，最近搜索和示例词被降级为输入时出现的辅助浮层，从而让主界面更接近聊天软件侧栏而不是传统仪表盘。

#### 设计目标
- 应用默认以不打断聊天主窗口的方式存在，用户能把它理解为“旁边随时可取图的小工作台”
- 用户打开应用后，不需要先理解多个首页模块，而是能直接输入或直接挑一张图
- 搜索入口位置尽量贴近即时通讯产品的输入习惯，降低视线和鼠标往返成本
- 侧边栏态与展开态分工明确，用户能够理解什么时候适合“快速找图”，什么时候适合“展开整理”
- 启动和切换窗口形态时，界面应尽量减少空白、跳变和突兀重排

#### 范围说明
**本期做什么：**
- 提供 `sidebar` / `expanded` 双态窗口模式，但不向用户暴露“默认打开方式”设置
- 将搜索栏固定在主界面底部，作为默认常驻主输入区
- 将首页主区收敛为“直接挑一张”的候选池
- 将最近搜索和搜索示例改为输入聚焦时出现的辅助浮层
- 在启动阶段预先应用默认侧栏布局，并用启动骨架减少白屏观感
- 为侧边栏态补充轻预览、固定窗口约束和更适合窄窗的页面结构
- 静默记住用户上次侧边栏位置与尺寸，减少重复摆放成本
- 首次启动时，侧边栏态默认从屏幕右侧启动

**本期不做什么：**
- 不提供顶部和底部两套搜索入口并存的模式
- 不提供上下面板式主布局作为第三种常驻窗口形态
- 不在首页继续并列扩展“最近搜索 / 最近使用 / 常用表情 / 推荐”等多个大区块
- 不引入复杂的可自由拖拽停靠系统、悬浮吸附系统或多窗口工作台
- 不提供用户可见的“切换左侧 / 切换右侧”或停靠侧设置项
- 不在本期解决所有窗口动画与尺寸恢复的系统级细节问题，文档仅记录当前实现状态

#### 方案结论
从当前用户场景出发，主方案已收敛为“始终侧边栏启动 + 显式进入展开态”，而不是让用户在左右侧边栏、上下面板、双方案自由选择之间自己做产品判断。

结论理由：
- 聊天共存的主要矛盾是横向空间占用和遮挡，侧边栏比上下面板更符合大多数桌面聊天窗口的并排使用方式
- 上下面板会直接压缩聊天记录高度，也会让底部输入区和应用输入区争抢垂直注意力，不适合作为默认方案
- 完全开放多种布局虽然看似灵活，但会显著抬高设置理解成本，并增加适配、测试和状态恢复复杂度
- “双态窗口”已经覆盖绝大多数使用场景：平时用侧边栏，管理时切展开
- 对聊天伴随工具而言，启动入口应保持稳定，不应因上次进入管理态而让下次直接以大窗口启动

#### 用户流程说明
**主流程 1：聊天中快速找图**
1. 用户在聊天软件旁边打开应用
2. 应用始终以侧边栏态显示，并尽量恢复用户上次手动摆放的侧边栏位置；若无历史位置，则默认出现在右侧
3. 用户直接在底部搜索栏输入关键词，或从首页候选池直接点一张
4. 用户点击图片后立即复制，用于发送到聊天软件

**主流程 2：不输入，直接从首页拿图**
1. 用户打开应用但暂时没有明确搜索词
2. 首页主区展示去重后的快速候选池
3. 用户浏览后直接点击图片复制，必要时可通过轻预览做一眼确认

**主流程 3：需要更大空间管理或浏览**
1. 用户点击“进入图库管理”或切换到更适合浏览的上下文
2. 应用切换为展开态
3. 用户在更宽的窗口中查看结果、进入图库管理

**主流程 4：重新启动应用**
1. 用户关闭后再次打开应用
2. 系统读取侧边栏最近一次窗口位置/尺寸快照
3. 后端在窗口展示前应用默认侧边栏布局
4. 前端在内容加载前使用启动骨架兜底，减少白屏感

#### 交互说明
**1. 双态窗口与窗口恢复**

页面元素：
- 侧边栏态主界面
- 展开态主界面

交互行为：
- 应用启动时始终进入侧边栏态，不向用户暴露窗口模式设置项
- 用户仅在点击“进入图库管理”等显式管理入口时进入展开态
- 应用应静默记录用户最近一次手动摆放后的侧边栏位置和尺寸，用于下次启动恢复
- 若不存在历史窗口快照，侧边栏态默认从屏幕右侧启动；展开态在被显式触发时默认居中显示
- 不向用户暴露左右停靠切换，因为该能力在不同平台和窗口管理器下表现不稳定
- 展开态下窗口应居中显示，用于承接更完整的浏览和管理操作
- 侧边栏态下应禁用最大化，避免窄窗模式进入不符合预期的系统窗口状态
- 展开态下应恢复可最大化能力

**2. 侧边栏态页面结构**

页面元素：
- 顶部极简头部
- 中部主内容区
- 底部常驻搜索栏

交互行为：
- 搜索栏固定在底部，始终作为主输入入口存在
- 中部区域承担首页候选池、搜索结果和空状态展示
- 顶部区域只保留必要的页面识别和最小化干扰操作，不再承担主要搜索行为
- 用户在侧边栏态中应始终感知到“底部输入，往上看结果”的稳定结构

**3. 首页候选池收敛**

页面元素：
- 快速候选图片区
- 轻量标题或说明文案

交互行为：
- 空查询首页主区只保留一个“直接挑一张”的候选池，不再并列展示多组语义相近模块
- 候选池数据由 `recentUsedImages + homeImages` 合并去重后得到
- 候选池优先承担“马上点一张就走”的任务，而不是解释系统记录逻辑
- 首页若存在图片，用户应能直接点击复制

补充说明：
- 当前实现下，首页图片列表不是独立推荐源，而是最近使用图片与首页基础图片的去重合并结果

**4. 底部搜索栏与辅助浮层**

页面元素：
- 底部搜索输入框
- 最近搜索辅助区
- 搜索示例词辅助区

交互行为：
- 搜索框固定在底部，符合聊天产品常见输入位置心智
- 用户聚焦搜索框时，可看到最近搜索和示例词等辅助内容
- 最近搜索和示例词不再长期占据首页主区，只在“用户准备输入”的时刻提供帮助
- 用户点击最近搜索后应直接带入并触发搜索
- 用户点击示例词后应快速开始一次搜索，降低冷启动犹豫

**5. 图片卡与轻预览**

页面元素：
- 图片卡片
- 轻预览浮层

交互行为：
- 用户在列表或首页候选池中悬浮图片卡时，可触发轻预览
- 轻预览应采用不受父容器裁切影响的展示方式，保证在窄窗下仍可见
- 轻预览位置应根据视口动态调整，尽量避免被窗口边缘截断
- 用户点击图片卡后，主行为仍为快速复制

**6. 启动观感优化**

页面元素：
- 启动骨架屏

交互行为：
- 应用加载前，页面应先展示与侧边栏结构接近的骨架屏，而不是纯空白画布
- 后端应在 `setup` 阶段读取窗口偏好并应用布局，减少前端挂载后再整体跳变的感知
- 当偏好已存在时，应用应尽量以最终目标布局启动，而不是先以默认大窗出现再二次调整

#### 页面与状态规则
| 场景 | 触发条件 | 页面行为 | 可执行动作 |
|-----|---------|---------|-----------|
| 默认启动 | 应用首次或普通启动 | 始终进入侧边栏态 | 搜索、浏览、打开菜单 |
| 空查询首页 | 搜索框为空 | 展示单一快速候选池 | 直接复制、悬浮预览 |
| 搜索框聚焦 | 用户点击底部搜索栏 | 展示最近搜索和示例词辅助浮层 | 重搜、使用示例词 |
| 侧边栏态 | 默认启动或从管理态返回 | 窄窗展示，底部搜索常驻，禁用最大化 | 快速找图 |
| 展开态 | 用户显式进入管理上下文 | 更大尺寸居中展示，恢复最大化能力 | 浏览、管理 |
| 应用重启 | 本地已有侧边栏窗口快照 | 启动前读取侧边栏快照并预布局 | 继续快速找图 |

#### 数据与状态要求
| 数据项 | 用途 | 最低要求 |
|-------|------|---------|
| `sidebarWindowSnapshot` | 记录侧边栏态最近窗口位置和尺寸 | 至少包含 `x / y / width / height` |
| `window-preferences.json` | 后端持久化窗口偏好 | 存放于应用数据目录，可在启动时读取 |
| `quickPickImages` | 支撑首页候选池 | 由 `recentUsedImages + homeImages` 去重合并 |

补充要求：
- 侧边栏窗口快照保存应与窗口移动和尺寸更新一起发生，避免下次启动丢失
- 首页候选池去重后不应重复展示同一图片
- 最近搜索不应作为常驻首页模块重复出现
- 不应将左右停靠或窗口模式作为用户可见偏好字段持久化

#### 当前实现口径
以下内容为当前代码已落地状态，用于和实现保持一致：
- 首次启动且无历史位置时，侧边栏态默认出现在屏幕右侧
- 侧边栏宽度按工作区宽度比例计算，并限制在合理范围内
- 侧边栏高度按工作区高度约 `88%` 计算，并限制在 `640` 到 `860` 像素之间
- 侧边栏态存在顶部偏移，避免窗口顶到底贴满整屏
- 启动骨架已放在 `index.html` 中，首屏不再是纯空白容器
- 后端已提供 `apply_window_layout` 与 `save_window_preferences` 命令
- 后端已在应用 `setup` 阶段读取侧边栏布局信息并预先应用默认侧栏布局
- 当前不再对用户暴露左右停靠切换；窗口位置恢复按平台能力尽量生效，但不承诺跨平台完全一致

#### 边界情况处理
| 场景描述 | 触发条件 | 系统行为 | 提示信息 |
|---------|---------|---------|---------|
| 首次启动无偏好 | 本地不存在窗口偏好文件 | 使用默认侧边栏布局启动，侧边栏默认出现在右侧 | 无 |
| 窗口偏好文件损坏 | 偏好文件反序列化失败 | 回退到默认窗口偏好 | 无 |
| 屏幕工作区过矮 | 可用高度不足 | 侧边栏高度按最小高度约束显示 | 无 |
| 用户切到展开态 | 需要更大浏览空间 | 恢复可最大化，并居中显示 | 无 |
| 当前平台不稳定支持窗口位置恢复 | 平台或窗口管理器限制主动定位 | 尽量恢复上次窗口位置；若失败则回退到默认布局，不向用户暴露停靠切换入口 | 无 |
| 轻预览靠近窗口边缘 | 窄窗空间不足 | 动态调整预览位置，避免被裁切 | 无 |
| 搜索框未聚焦 | 用户尚未准备输入 | 不显示搜索辅助浮层 | 无 |

#### 验收标准
**功能指标：**
- [ ] 不再向用户暴露停靠侧切换入口或设置项
- [ ] 应用启动时始终进入侧边栏态，不因上次进入管理态而直接以展开态启动
- [ ] 应用重启后可恢复上次侧边栏位置与尺寸；若无历史位置则使用默认布局
- [ ] 侧边栏态下搜索栏固定在底部
- [ ] 空查询首页主区只保留单一候选池，而不是多个并列记录模块
- [ ] 搜索框聚焦时可展示最近搜索和示例词辅助内容
- [ ] 图片卡轻预览在窄窗下不会被列表容器直接裁切

**体验指标：**
- [ ] 用户可以把产品理解为聊天软件旁边的找图侧栏，而不是必须独占桌面的主窗口应用
- [ ] 用户在首页可以直接挑图，也可以自然地在底部开始输入，不需要在页面上来回找入口
- [ ] 首页信息量比旧版更克制，不会同时出现多组难以区分的内容模块
- [ ] 侧边栏态与展开态分工清晰，用户知道何时快速找图，何时展开处理
- [ ] 用户不会因为上次进入图库管理而误以为下次打开应用会先进入管理态
- [ ] 启动时白屏和明显跳变感相较旧版有所收敛

**可靠性指标：**
- [ ] 窗口偏好文件缺失或损坏时，应用仍可正常启动
- [ ] 平台不支持稳定恢复窗口位置时，应用仍可回退到默认布局正常启动
- [ ] 侧边栏态不会因最大化进入异常窗口状态
- [ ] 首页候选池不会因合并数据而出现明显重复项
- [ ] 轻预览在窗口边缘附近仍能保持基本可见

#### 假设
- 假设用户的主要使用场景仍是桌面聊天软件共存，而不是全屏独立使用
- 假设底部输入符合大多数即时通讯用户的路径惯性，能减少寻找搜索入口的成本
- 假设双态窗口已经足以覆盖“快速找图”和“展开管理”两类主任务，不需要再开放第三种主布局

#### 开放问题
- 启动时空白窗口与内容整体放大感知已通过预布局和骨架屏做过一轮缓解，但是否完全收敛仍需继续观察
- 侧边栏初始高度目前已收紧，但不同分辨率下是否还需要更细的分档规则仍待验证
- 左右停靠切换已从需求中收回；后续若重新评估该能力，前提必须是跨平台窗口定位支持显著改善，否则不应重新对用户暴露
