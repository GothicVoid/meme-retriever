### 5.1.2 结果列表展示功能（优先级：P0）

#### 修订记录
| 版本号 | 修订日期 | 修订人 | 修订内容 | 修订原因 |
|-------|---------|--------|---------|---------|
| v1.0 | 2026-04-18 | 开发 | 补充结果卡片中的“得分与排序依据展示”方案，区分普通用户可读展示与开发调试展示 | 避免直接暴露底层三路得分和公式，降低用户理解成本并约束实现范围 |
| v1.1 | 2026-04-18 | 开发 | 补充命中文字与命中标签证据字段，并要求排序说明区位于图片下方，不得遮挡图片主体 | 提高解释可信度，同时修正当前 UI 遮挡图片的问题 |
| v1.2 | 2026-04-18 | 开发 | 将结果展示改为“高相关+较相关默认展示、低相关默认隐藏”的连续滚动方案，并补充低相关改写提示、统一相关度阈值与低相关展开规则 | 根据实际使用反馈修正固定数量展示体验，降低尾部噪音并提升角色搜索的连续浏览体验 |

#### 功能描述
以网格形式展示检索结果，支持图片缩略图预览和基础交互，**带文件格式标识**。对于搜索结果的排序依据，系统应优先以用户可理解的自然语言说明“为什么这张图排在前面”，而不是直接展示底层权重、公式或多路百分比分数。

#### 流程说明
1. 接收检索引擎返回的排序结果
2. 按统一相关度阈值将结果划分为“高相关 / 较相关 / 弱相关”
3. 默认渲染全部高相关和较相关结果，**GIF在缩略图状态下仅显示首帧，不播放动画**
4. 若当前已加载结果仍全部属于高相关或较相关，用户继续向下滚动时系统应自动补齐下一段结果；必要时继续向后加载更多候选
5. 当结果中出现弱相关尾部时，系统应停止自动继续展示，并在列表下方提示“已展示全部高相关和较相关结果”
6. 若用户主动展开低相关结果，则展示当前已加载的全部低相关候选，并提供“收起低相关结果”入口
7. 当整批结果都属于弱相关时，默认不直接展示图片，而是优先提示更精确的搜索写法；用户仍可手动展开查看当前已加载候选
8. 当结果包含排序解释信息时，在卡片上展示简化后的“相关度标签 + 排序理由 + 证据摘要”
9. 当用户进入开发调试模式时，可额外查看排序构成明细

#### 交互说明
**页面元素：**
- 图片网格：3列布局，自适应间距
- 缩略图：等比缩放，最大尺寸150×150px
- **格式标识角标：位于缩略图右上角，半透明圆角矩形背景**
  - JPG/PNG：不显示角标（默认格式）
  - GIF：显示"GIF"标识，橙色背景
  - WEBP：显示"WEBP"标识，蓝色背景
- 连续加载触发区：位于结果列表底部，当当前已加载结果仍属于高相关或较相关时，滚动进入触发区自动补齐结果
- 结果反馈区：位于列表下方，用于提示“已展示全部高相关和较相关结果”或“没找到足够相关的结果”
- 低相关展开按钮：仅当存在弱相关尾部，或整批结果均为弱相关时显示
- **排序说明区：位于卡片下层信息区或悬浮信息区，不与格式角标重叠**
- **排序说明区：位于图片下方的信息区，不得覆盖在图片主体之上**
  - 相关度标签：使用“高相关 / 较相关 / 弱相关”等用户可理解标签
  - 排序理由：使用自然语言说明主导排序原因，如“文字匹配优先 / 语义最接近 / 角色匹配优先”
  - 证据摘要：最多展示 2~3 条辅助证据，如“命中文字：xxx”“标签命中：xxx”“最近常用”
- **开发调试入口：仅在设置中启用开发调试模式后显示**
  - 默认用户不可见
  - 启用后可查看排序构成明细，但不改变普通模式的主展示结构

**交互行为：**
- 单击图片：复制到剪贴板，显示Toast"已复制"
- 双击图片：打开单图详情页（**GIF格式在详情页自动播放动画**）
- 右键菜单：显示操作选项（删除、编辑标签、在文件夹中显示、查看详情）
- 鼠标悬停：显示半透明蒙层和操作图标，**GIF在缩略图状态仍显示首帧**
- 普通模式下，用户应直接看到“为什么排前面”的简化说明，不应看到`OCR 11% / CLIP 11% / 标签 0%`这类底层调试数值
- 普通模式下，排序说明区不得遮挡图片主体；若卡片空间不足，应压缩摘要文案或减少摘要条数，而不是覆盖在图片上
- 若结果卡片空间不足，证据摘要应优先保留首条最强证据，其余省略，不换行为复杂公式说明
- 当搜索结果同时包含高相关和较相关时，系统应默认展示这两类结果，只隐藏弱相关尾部
- 当用户搜索角色且结果持续处于高相关或较相关区间时，页面应支持连续滚动浏览，不应被固定数量截断
- 当整批结果都属于弱相关时，系统应优先给出改写建议，例如补充角色名、动作、台词或场景词，而不是默认灌出整批弱相关图片
- 当用户主动展开弱相关结果后，应提供可见的“收起低相关结果”入口，返回默认视图
- 若用户主动展开“排序依据”区域，可看到更完整的理由说明和辅助项，但仍以自然语言命名展示
- 开发调试模式下，可额外展示最终得分、主路类型和各贡献项数值，供开发排查排序问题

**反馈提示：**
- 复制成功：Toast提示"已复制"，1.5秒后消失
- 删除确认：模态框"确定删除这张图片吗？此操作不可撤销"

#### 算法说明
**缩略图生成规则：**

| 文件格式 | 生成策略 | 缓存策略 |
|---------|---------|---------|
| JPG/PNG | 等比缩放至150×150内，保持宽高比 | 首次生成后缓存 |
| GIF | 提取第一帧，等比缩放至150×150内 | 首次生成后缓存 |
| WEBP | 等比缩放至150×150内，保持宽高比 | 首次生成后缓存 |

**角标显示规则：**
```javascript
if (fileFormat === 'GIF') {
  显示角标: 'GIF', 背景色: '#FF9500'
} else if (fileFormat === 'WEBP') {
  显示角标: 'WEBP', 背景色: '#1890FF'
} else {
  不显示角标
}
```

**排序说明映射规则：**

系统应将底层排序信号转换为用户可读文案，而不是直接展示内部公式。

| 底层信号 | 用户态展示文案 | 说明 |
|---------|---------------|------|
| `mainRoute = ocr` | 文字匹配优先 | 当前结果主要因图片文字命中而靠前 |
| `mainRoute = semantic` | 语义最接近 | 当前结果主要因图片内容语义接近查询而靠前 |
| `mainRoute = private_role` | 角色匹配优先 | 当前结果主要因私有角色匹配命中而靠前 |
| `tagScore > 0` | 标签命中 | 作为证据摘要的一部分出现 |
| `popularityBoost > 0` 且达到展示阈值 | 最近常用 | 仅作为辅助证据，不应成为唯一主理由 |

**相关度标签建议规则：**

| 最终得分区间 | 用户态标签 | 列表展示策略 |
|-------------|-----------|-------------|
| `score >= 0.75` | 高相关 | 默认展示 |
| `0.55 <= score < 0.75` | 较相关 | 默认展示 |
| `score < 0.55` | 弱相关 | 默认隐藏，仅在用户主动展开后展示 |

说明：
- 相关度标签用于帮助用户快速判断结果可信度，不表示概率或正确率承诺
- 列表分层、结果反馈文案和卡片相关度标签必须共用同一套阈值，不得一处判为“较相关”而另一处仍按“高相关”展示
- 若后续调整阈值，应同步更新测试和文案，不得前端硬编码与后端长期脱节

**证据摘要生成规则：**
- 系统应优先展示与主理由一致的证据
- 普通模式下每张卡片最多展示 2~3 条证据摘要
- 证据摘要优先级建议为：主路证据 > 标签证据 > 常用度证据
- 若搜索结果已返回具体证据字段，应优先展示命中词，如“命中文字：撤回”“标签命中：猫猫”
- 当前版本搜索结果应支持至少以下证据字段：
  - `matchedOcrTerms`
  - `matchedTags`
  - `matchedRoleName`
- 仅当具体证据字段缺失时，才允许回退到抽象文案，如“图片文字相关”“标签相关”

**开发调试模式展示规则：**
- 开发调试模式仅用于排查排序问题
- 可展示以下字段：最终得分、主路类型、主路贡献、辅助贡献、标签贡献、热度加成
- 开发调试模式中的字段命名应尽量贴近开发语义，但不应污染普通模式 UI
- 普通模式与调试模式应共用同一结果数据来源，避免两套解释相互矛盾
- 即使开发调试模式展示了热度加成，普通模式下“最近常用”也只能作为辅助证据，不能成为用户理解该结果应排前的唯一理由

#### 边界情况处理
| 场景描述 | 触发条件 | 系统行为 | 提示信息 |
|---------|---------|---------|---------|
| 高相关/较相关结果不足一屏 | 默认结果数量较少 | 直接展示全部相关结果，不显示连续加载触发区 | 无 |
| 高相关/较相关结果很多 | 当前已加载结果仍都属于高相关或较相关 | 滚动到底部自动补齐下一段，必要时继续向后加载更多候选 | 无 |
| 存在弱相关尾部 | 结果中首次出现 `score < 0.55` 的尾部 | 默认停止继续展示，在反馈区提示“已展示全部高相关和较相关结果” | 无 |
| 用户主动查看弱相关结果 | 点击“仍然查看其余 N 张结果” | 展示当前已加载的全部弱相关结果，并显示“收起低相关结果” | 无 |
| 整批结果都为弱相关 | 所有结果 `score < 0.55` | 默认不展示结果卡，优先给出查询改写建议；用户可主动展开查看当前已加载候选 | 建议补充台词、角色、动作或场景词 |
| 图片文件丢失 | 原文件被移动或删除 | 显示占位图"图片不存在" | 鼠标悬停提示"原文件已丢失" |
| 图片加载失败 | 文件损坏或格式错误 | 显示占位图"加载失败" | 无 |
| GIF文件损坏 | 无法提取首帧 | 显示占位图，仍显示GIF角标 | 鼠标悬停提示"GIF文件损坏" |
| 排序依据字段缺失 | 搜索结果未返回解释字段 | 仅展示图片本身，不显示排序说明区 | 无 |
| 仅有抽象信号无具体命中词 | 后端暂未返回命中文字/标签词 | 展示“文字相关 / 标签相关 / 语义接近”等泛化文案 | 无 |
| 排序说明区过长 | 命中证据较多或文案过长 | 最多展示 2~3 条摘要，其余省略 | 无 |
| 热度加成存在但较弱 | 热度值未达到摘要展示阈值 | 不单独展示“最近常用” | 无 |
| 弱相关但高热度 | 结果相关度 `score < 0.55`，但使用次数较高 | 热度不能将其抬入默认展示区，只能在用户主动展开弱相关结果时出现 | 无 |
| 空间不足 | 卡片宽度不足以容纳完整说明 | 折叠为单行摘要或仅保留主理由，但不得遮挡图片 | 无 |
| 开发调试模式关闭 | 普通用户使用默认设置 | 不展示底层分项数值 | 无 |

#### 数据埋点
| 埋点事件 | 触发时机 | 采集参数 | 用途说明 |
|---------|---------|---------|---------|
| image_click_copy | 用户单击图片复制 | image_id, search_query, rank_position, file_format | 分析点击率和格式偏好 |
| image_double_click | 用户双击打开详情 | image_id, file_format | 分析详情页使用率 |
| low_relevance_expand_click | 用户主动展开弱相关结果 | result_count, search_query | 评估默认隐藏策略是否过严 |
| low_relevance_collapse_click | 用户收起弱相关结果 | result_count, search_query | 分析弱相关结果浏览意愿 |
| search_hint_shown | 系统展示低相关改写建议 | search_query, result_count | 评估查询质量与提示触发率 |
| result_reason_expand | 用户展开排序依据说明 | image_id, search_query, rank_position, main_reason | 评估用户是否需要进一步理解排序解释 |

#### 扩展点说明
- 支持用户自定义角标显示规则（如关闭角标）
- 支持自定义缩略图尺寸
- 支持视频格式（后续版本）
- 支持后端返回更具体的命中证据，如 `matchedOcrTerms`、`matchedTags`
- 支持在详情页展示更完整的排序依据说明

#### 验收标准
**功能指标：**
- [x] 默认展示高相关和较相关结果
- [x] 弱相关结果默认隐藏，仅在用户主动展开后展示
- [x] 高相关/较相关结果支持滚动连续补齐与继续加载
- [x] 整批结果均为弱相关时，优先展示查询改写建议
- [x] 低相关展开后可收起返回默认视图
- [x] 单击复制功能正常
- [x] 双击打开详情功能正常
- [x] 右键菜单功能完整
- [x] GIF角标正确显示
- [x] WEBP角标正确显示
- [x] GIF缩略图仅显示首帧
- [ ] 普通模式下不再直接展示 CLIP / OCR / 标签等底层多路得分百分比
- [ ] 普通模式下每张结果卡可展示“相关度标签 + 排序理由 + 证据摘要”
- [ ] 排序理由文案可区分文字匹配、语义匹配、角色匹配等主导原因
- [ ] 后端可返回命中文字、命中标签或命中角色名等具体证据字段
- [ ] 普通模式下排序说明位于图片下方，不遮挡图片主体
- [ ] 当解释字段缺失时，结果卡片可优雅降级，不影响图片浏览和复制操作
- [ ] 开发调试模式开启后，可查看数值化排序构成

**性能指标：**
- [x] 缩略图加载流畅，无明显延迟
- [x] GIF首帧提取 < 200ms
- [x] 网格布局自适应窗口大小
- [ ] 排序说明区的渲染不应明显拖慢结果首屏展示

#### 假设
- 假设当前搜索结果已包含 `score` 与基础 `debugInfo`，可作为第一阶段映射来源
- 假设普通用户无需理解具体加权公式，更关心“为什么排前面”
- 假设具体命中词可能需要后端后续补充，本次 spec 允许先以抽象证据文案落地

#### 开放问题
- 是否需要在普通模式中保留“点击查看排序依据”的显式入口，还是默认直接展示简版理由
- 后端是否在本迭代内补充命中文字、命中标签等证据字段
- “高相关 / 较相关 / 弱相关”的阈值是否需要通过真实样本再校准
