# 搜索与首页主链路

- 状态：approved
- Spec ID：`SPEC-SEARCH`

## 功能描述

本文件统一定义搜索主路、空查询首页、结果展示、搜索体验增强和高频使用反馈。

当前产品的默认入口就是搜索工作台。用户不需要先理解图库管理、角色库或设置，只需要围绕一条主链路完成使用：

`导入 -> 搜索 -> 复制 -> 再找回`

## 当前结构

- 图库为空且查询为空：展示冷启动首页
- 图库非空且查询为空：展示普通首页
- 查询非空：展示搜索结果页

三种状态共享同一套产品语言，不应做成三张风格割裂的页面。

## 搜索原则

- 用户按记忆线索搜索，不要求维护文件名或结构化标签
- 搜索主路由当前查询决定，常见线索包括台词、角色、动作、表情、场景
- OCR 和语义是主路，用户自己添加的标签和热度只做增强
- 如果用户自己添加的标签命中查询，该图片必须作为可见相关结果出现，不能只藏在“查看更多/其余结果”里
- 默认结果只展示高相关和较相关，弱相关结果不应直接灌给用户
- 普通用户看到的是“为什么这张排前面”的自然语言，不是底层打分公式

## 首页与冷启动

### 冷启动首页

- 首屏主动作是 `导入图片`
- 示例词和示例图区只用于表达产品能力，不触发真实结果
- 用户首次进入后应在 5 到 10 秒内理解主链路是 `导入 -> 搜索 -> 复制`

### 普通首页

- 查询为空时展示最近常用或最近使用图片
- 搜索历史不作为首页常驻区块，而是在搜索框聚焦后通过辅助面板展示
- 首页应优先服务“继续使用”，而不是堆叠说明和导航

## 结果展示与交互

- 结果卡以图片主体优先
- 单击图片默认复制
- 双击图片进入详情
- GIF 缩略图默认显示首帧，详情页中再播放动画
- 排序说明展示在图片下方的信息区，不得遮挡图片主体
- 结果卡提示只保留 `相关性判断 + 一个主原因` 的单行结构，不展示第二行补充证据
- 主原因使用用户语言，例如 `命中文字`、`角色匹配`、`标签匹配`、`画面接近`，不暴露 `主路`、`语义`、`辅路` 等内部术语
- 无结果或仅弱相关时，必须给出明确下一步建议

## 搜索体验增强

- placeholder 使用任务语言，例如 `搜台词、角色、动作、场景`
- 搜索框为空且聚焦时展示最近搜索
- 普通首页和搜索框聚焦下拉不再展示静态建议词，避免系统推荐与当前图库脱节
- 用户可以一键重搜、删单条历史
- 最近搜索只记录稳定保留至少 2 秒的查询；输入后若很快继续改字、删字、清空，则视为撤回，不记入历史
- 最近搜索最多保留最近 20 条；搜索框聚焦下拉只展示其中最新的 5 条
- 最近搜索弹层底部提供 `清除` 入口，一次性清空全部最近搜索
- 失败反馈区不重复搜索前提示，而是优先帮助用户判断 `图库里有没有`、`这次词是否写偏`、`当前是否该切去补救路径`

## 搜索失败与无结果反馈

### 设计目标

- 无结果反馈不再复读 `试试原文 / 试试角色 / 试试动作 / 试试场景`
- 搜索前提示负责告诉用户“可以拿什么搜”；无结果反馈负责告诉用户“现在改走哪条补救路径”
- 失败反馈区优先给一个默认推荐动作，再补充少量次级入口，不做成同权按钮墙
- 系统只能对明确可观察事实使用确定语气；对猜测类原因只能使用条件式建议

### 触发范围

- 查询非空且搜索返回 0 条可见结果
- 查询非空且仅存在弱相关候选，默认主结果区为空
- 查询非空且本次搜索请求失败

### 原因分层

#### 1. 明确可判断

- 图库为空：当前库中没有图片
- 存在未完成导入任务：当前搜索范围不完整
- 搜索请求失败：本次搜索动作未完成
- 已发现失效图片：当前库中存在原文件已丢失的图片

这类情况可以直接使用确定语气，并给出明确补救动作。

#### 2. 只能高概率推断

- 关键词可能写偏、打错字、记错叫法
- 图可能在库里，但当前线索无法命中
- 当前问题可能和角色识别有关，但系统无法确认用户搜的是不是角色

这类情况只能提供条件式建议，不得伪装成系统诊断结果。

#### 3. 当前无法判断

- 目标图是否根本不在库里
- 图是否存在，但用户只记得模糊线索
- 当前词是否属于临时梗、谐音词、错别字或个人叫法

这类情况不编造失败原因，只提供探索路径。

### 信息结构

无结果反馈区固定使用以下结构：

1. 状态标题
2. 一句说明
3. 一个默认推荐动作
4. 2 到 3 个次级入口

补充规则：

- 主动作只能有一个
- 次级入口只保留不同方向的补救路径，不能只是同义改写建议
- `查看低相关候选` 使用次级或文字样式，不与主补救动作同权
- `补标签` 属于优化下次搜索的长期动作，权重必须低于“解决这次”的动作

### 默认动作优先级

系统应按以下优先级选择默认动作：

1. 图库为空：`导入图片`
2. 存在未完成导入任务：`去图库继续导入`
3. 搜索失败且当前库中存在失效图片：`查看失效图片`
4. 其他普通无结果：`去图库确认有没有这张图`

说明：

- 普通无结果时，默认动作不再是“换个说法”，而是先帮助用户确认目标图是否真的在库里
- 当多个条件同时成立时，按上面的优先级只选择一个默认动作

### 次级入口规则

普通无结果时，次级入口最多保留以下几类：

- `检查是不是词写偏了`
  - 用于提示用户可能记错叫法、写错字或用了不稳定梗词
- `查看低相关候选`
  - 用于承接“系统不确定，但用户仍愿意自己翻”的场景
- `角色搜索增强`
  - 仅作为条件式建议出现，用于提示角色相关搜索可切去增强路径
- `给难搜图片补标签`
  - 仅作为低权重长期优化入口出现，不作为本次搜索失败的主修复路径

### 文案口径

应使用：

- `没找到这类图片`
- `这次没有找到足够接近的结果`
- `先去图库确认有没有这张图`
- `也可能是关键词写偏了，换个更接近的叫法试试`
- `如果你找的是某个角色，可以试试角色搜索增强`
- `这类图以后总是不好搜的话，可以补个标签`

不应使用：

- `试试原文`
- `试试角色名`
- `试试动作`
- `试试场景`
- `系统没认出这个角色`
- `这张图没有文字`
- `这是 OCR 失败`
- `这是同义词问题`

### 角色搜索增强口径

- 系统不应假设自己能准确判断用户搜的是不是角色
- 角色增强只能作为可选补救路径，不作为普通无结果的默认主动作
- 角色增强文案必须使用条件句，例如：`如果你找的是某个角色，可以试试角色搜索增强`
- 不得使用 `你搜的是角色名`、`系统没认出这个角色` 这类判断式文案

### 状态文案草稿

#### 普通无结果

- 标题：`没找到这类图片`
- 说明：`这次没有找到足够接近的结果。`
- 主动作：`去图库确认有没有这张图`
- 次级入口：
  - `检查是不是词写偏了`
  - `看看低相关候选`
  - `如果你找的是某个角色，可以试试角色搜索增强`
- 低权重补充：`这类图以后总是不好搜的话，可以补个标签`

#### 有未完成导入任务

- 标题：`还没找到这类图片`
- 说明：`你的图库还有图片没导完，当前结果可能不完整。`
- 主动作：`去图库继续导入`
- 次级入口：
  - `去图库确认有没有这张图`
  - `检查是不是词写偏了`

#### 已发现失效图片

- 标题：`这次没找到目标图片`
- 说明：`图库里有失效图片，目标图可能已经找不到原文件。`
- 主动作：`查看失效图片`
- 次级入口：
  - `去图库确认有没有这张图`
  - `检查是不是词写偏了`

#### 搜索请求失败

- 标题：`这次搜索没成功`
- 说明：`刚才这次搜索没有完成。`
- 主动作：`重试`
- 次级入口：
  - `去图库继续导入` 或 `查看失效图片`
  - 按当前状态二选一，不并列平铺多个同权入口

## 高频使用体验

- `/` 或 `Ctrl+K` 聚焦搜索框
- 结果列表支持键盘焦点移动
- `Enter` 默认复制当前焦点卡
- `Space` 或 `Shift+Enter` 打开快速预览
- 关键状态必须有明确反馈，包括加载、失败、文件丢失、导入中断恢复
- 搜索框底部操作区固定保留 `图库` 快捷入口，便于快速切去治理路径

## 开发模式补充

- 正式用户路径不展示开发调试概念
- 当前实现仅在开发模式下，于搜索框底部操作区显示开发工具按钮
- 开发工具以轻浮层承接 `开发调试模式`、`重新生成图像索引`、`清空图库`
- `开发调试模式` 的可见状态会持久化到本地 `settings.devDebugMode`，用于继续显示结果调试叠层

## 不负责

- 不承担批量治理和图库问题治理
- 不暴露私有角色库维护和开发调试概念
- 不把设置或内部维护能力混入主搜索路径

## 验收要点

- 用户能清楚区分冷启动首页、普通首页和搜索结果页三种状态
- 用户首次进入时能理解首个动作是导入，而不是进入图库管理
- 默认搜索结果只展示高相关和较相关结果
- 用户能看懂排序理由和失败反馈，而不需要理解算法术语
- 高频用户可以稳定完成“聚焦搜索、切换结果、复制、预览、再次找回”
- 无结果反馈区不会重复 placeholder 已经表达过的搜索前提示
- 失败反馈区始终先给一个默认推荐动作，不把多个补救路径做成同权按钮墙
- 普通无结果时，默认动作是先确认目标图是否在库里，而不是继续复读改写建议
- 角色搜索增强只作为条件式次级入口出现，不作为系统判断结果
