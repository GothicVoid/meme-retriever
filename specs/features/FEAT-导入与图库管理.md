# 导入与图库管理

- 状态：approved
- Spec ID：`SPEC-LIBRARY`

## 功能描述

本文件统一定义导入协议、图库管理页、导入结果承接、已发现失效图片治理、任务恢复、删除与详情边界。

图库管理不是首次使用默认必经页，而是导入后的整理、失败复核、已发现问题治理和批量补充导入工作页，同时承接部分搜索补修入口。

涉及“导入中 / 中断恢复 / 首页角标 / 图库顶部承接 / 恢复结果摘要”的跨界面状态联动，统一以 [`import-state-machine.md`](../architecture/import-state-machine.md) 为准；[`modules/library.md`](../modules/library.md) 仅保留模块级摘要入口。

## 图库管理职责

- 导入图片和导入文件夹
- 查看全部图片
- 查看最近一次导入结果、失败项与本次新增定位反馈
- 处理已发现失效图片
- 删除图片、批量删除、进入详情
- 恢复或放弃未完成入库任务
- 承接从搜索失败反馈跳转来的修复动作
- 提供低干扰的角色维护入口

## 导入入口

### 搜索页和冷启动页

- 使用单一主按钮 `导入图片`
- 点击后通过轻弹层分流到 `选择图片` 或 `选择文件夹`
- 导入动作尽量在当前页内完成，不默认跳去图库管理

### 图库管理页

- 保留 `导入图片` 和 `导入文件夹` 两个独立按钮
- 适合重度治理和批量补充导入

## 导入协议

- 图片导入和文件夹导入只在输入源采集阶段不同
- 进入执行阶段后，共享同一套导入状态机和索引任务管线
- 存储策略仍然是路径引用，不复制原文件
- 后端统一负责展开目录、过滤格式和去重

## 图库管理页结构

### 页面目标

用户进入图库管理页后，应在很短时间内回答三个问题：

- 现在图库有没有需要我处理的问题
- 当前最值得优先处理的是什么
- 处理完后我能否继续浏览或回到搜索

### 页面级区块顺序

1. 页面头部
2. 当前主任务区
3. 轻量工具条
4. 图片列表区

### 视觉一致性

- 图库管理页应与搜索页保持同一套桌面端视觉语言，包含信息密度、圆角、边框强度、留白节奏和标题层级
- 不应通过连续的大卡片、大留白和重边框把图库管理做成明显不同于搜索页的另一套产品观感
- 当页面没有高优任务需要强调时，工具条和列表区应更接近搜索页“内容直接落在工作台上”的轻量结构
- 在展开态窗口中，图库管理页的主内容区应优先利用可用宽度，不应因为通用居中容器留下大块无效侧边空白

### 页面头部

- 承载页面身份、总量信息和返回动作
- 页面标题固定为 `图库管理`
- 展示图库总量 `共 X 张`
- 如需提供离开治理路径的返回动作，只允许放一个低干扰 `返回搜索`
- 不重复放说明段落或高权重切页按钮
- 图库非空且无冷启动空状态时，`导入图片 / 导入文件夹` 可与页面标题压缩在同一行展示，减少顶部垂直占用

### 当前主任务区

同一时刻只允许一个主承接块占据首屏主位，其余问题降级为次级入口或暂不出现。

建议优先级顺序：

`未完成入库任务` > `最近一次导入失败项` > `已发现失效图片待处理` > `本次新增待确认` > `普通浏览`

主任务区状态：

#### 状态 A：存在未完成导入任务

- 标题：`上次导入中断，还有 X 张未处理`
- 说明：`继续处理这批图片`
- 主动作：`继续导入`
- 次动作：`放弃剩余图片`

#### 状态 B：最近一次导入存在失败项

- 标题：`最近一次导入有 X 项失败`
- 说明：`先看失败原因`
- 主动作：`查看失败项`
- 次动作：`查看本次新增`
- 辅助信息：显示 `新增 / 已存在 / 失败`

#### 状态 C：存在已发现失效图片

- 标题：`发现 X 张失效图片`
- 说明：`这些图片的原文件可能已被移动或删除`
- 主动作：`查看失效图片`
- 次动作：`清理失效图片`

#### 状态 D：最近刚完成导入且无异常

- 标题：`最近一批图片已入库`
- 说明：`先看本次新增`
- 主动作：`查看本次新增`
- 辅助信息：显示 `新增 / 已存在 / 失败`

#### 状态 E：无待处理任务

- 不显示主任务卡
- 页面直接进入工具条和列表区

#### 状态 F：图库为空

- 图库为空时不复制搜索页冷启动首页，也不承担首次使用主引导
- 页面头部仍只展示 `图库管理`、`共 0 张` 和低干扰 `返回搜索`
- 隐藏常规工具条，避免在无可管理对象时呈现“空表格工具栏”
- 空状态内部在示例图片区下方提供 `导入图片`、`导入文件夹` 两个独立入口
- 空状态标题：`图库还没有图片`
- 空状态说明：`这里用于查看全部图片、补充导入、处理导入失败和失效文件。`
- 空列表区应展示轻量占位，说明 `导入后，全部图片会按入库时间显示在这里`
- 不展示 `角色维护` 入口，避免用户误以为必须先配置角色库才能开始使用
- 不展示与搜索页冷启动同权的示例词、搜索框或完整 `导入 -> 搜索 -> 复制` 教育内容

### 轻量工具条

- 只保留一行
- 默认常驻动作：`导入图片`、`导入文件夹`
- 条件动作：`删除选中`
- 图库非空时，右侧允许保留低干扰 `角色搜索增强` 按钮
- `角色搜索增强` 旁边提供 `?` 或 `i` 说明标记，完整说明通过悬浮或聚焦提示承接，不常驻展开说明文案
- 图库为空时不显示常规工具条，导入动作由空状态承接
- 视觉上不要求独占一整行；在非空图库中可并入页头左侧，与 `图库管理` 标题同排
- 不堆叠说明性提示和低频工具
- 页面在常见桌面窗口高度下应优先保持首屏紧凑，避免因为容器留白或高度计算不当产生页面级无效滚动

### 图片列表区

- 常驻主视图只有全部图片列表，但不再展示不可切换的 `全部图片` 伪标签
- `全部图片` 按入库时间倒序浏览
- 列表与搜索结果尽量共享卡片语言，但图库管理强调“治理感”而不是“搜索感”
- 顶部路径说明使用短句，例如 `原文件移动、重命名或删除后会失效`
- 空状态遵循“状态 F：图库为空”，强调图库管理职责，不替代搜索页冷启动
- 列表滚动应优先发生在图片区内部，而不是让整个页面出现额外滚动条

## 导入结果承接

一次导入完成后，前端应能承接结构化结果，至少区分 `新增成功`、`已存在`、`处理失败`。

### 导入结果摘要

- 图库管理页中的导入结果区不只展示数量，还要给出一个默认主推荐动作
- 仅有新增成功、无失败时，默认主推荐动作为 `查看本次新增`
- 存在失败项时，默认主推荐动作为 `查看失败项`
- 仅有重复、无新增、无失败时，不强行引导额外治理

### 本次新增

`本次新增` 不作为常驻视图存在，而是 `全部图片` 内的一次性定位反馈。

定义：

- 只包含本轮普通导入或恢复导入中 `新增成功` 的图片
- 不包含 `已存在`、失败项、历史图片或最近编辑过的老图

交互：

- 点击 `查看本次新增` 后，页面切到 `全部图片` 并滚动到顶部
- 本轮新增图片在当前列表中显示 `新` 角标
- 页面顶部可显示轻提示：`已定位到本次新增，共 X 张`
- 不提供 `结束查看`
- 单纯滚动不清除 `新` 角标
- 切换视图或产生新一轮导入结果后退场

## 失败项明细

`查看失败项` 是“失败结果 + 下一步动作”的说明入口，不是错误日志页，也不默认承担跨边界补救流程。

### 基本要求

- 在当前页展开失败列表，不单独跳页
- 至少展示文件名和用户语言的失败原因
- 用户看完后可以继续当前任务，不强制处理全部失败项

### 动作原则

- 只有应用内能直接闭环的情况，才提供补救动作
- 需要用户离开当前应用去修文件、改权限、重新准备素材的情况，只做原因说明
- 默认允许用户知情后继续整理图库

### 当前产品口径

- `继续导入` 只用于恢复任务，不通过普通失败项猜测
- `重试导入` 只有在后端明确标注可重试时才出现
- 其他失败项默认主动作使用 `查看本次新增`，次动作使用 `继续整理图库`

### 结构化失败分类方案

为了稳定支撑失败项动作选择，失败项返回结构应新增：

- `failureKind`
- `retryable`
- `userMessage`

建议结构：

```ts
interface ImportBatchFailure {
  taskId: string;
  filePath: string;
  errorMessage?: string;
  failureKind: string;
  retryable: boolean;
  userMessage: string;
}
```

首版建议支持的 `failureKind`：

| failureKind | 用户文案 | retryable | 默认动作 |
| --- | --- | --- | --- |
| `interrupted_recoverable` | `导入中断，这批图片还没处理完` | `true` | `继续导入` |
| `transient_retryable` | `这张图片这次没导入成功，可以再试一次` | `true` | `重试导入` |
| `file_missing` | `原文件找不到，没能导入` | `false` | `查看本次新增` |
| `file_damaged` | `文件已损坏，没能导入` | `false` | `查看本次新增` |
| `access_denied` | `文件当前无法访问，没能导入` | `false` | `查看本次新增` |
| `unsupported_format` | `文件格式暂不支持，没能导入` | `false` | `查看本次新增` |
| `decode_failed` | `图片内容无法识别，没能导入` | `false` | `查看本次新增` |
| `unknown` | `这张图片没能导入` | `false` | `查看本次新增` |

补充规则：

- `duplicated` 不进入失败项，继续归入 `已存在`
- `继续导入` 继续由恢复状态机单独承接
- 做不到稳定判断时，一律落入 `unknown`

## 已发现失效图片

### 定义

“失效图片”在产品口径上应理解为“系统已发现原文件不可用的图片记录”，而不是“全库实时、完整的失效集合”。

### 展示策略

- `失效图片` 不再作为常驻视图存在
- 只有在系统已发现至少一条 `file_status = missing` 记录时，页面才出现失效治理入口
- 没有已发现失效记录时，入口完全消失

### 查看失效图片

- 点击 `查看失效图片` 后，不跳新页
- 页面进入同页过滤态，只展示当前已发现 `file_status = missing` 的图片记录
- 顶部轻状态条：`正在查看已发现的失效图片，共 X 张`
- 显式保留 `查看全部图片`，由用户手动切回完整列表
- 若失效图片已被全部清理，入口自动退场，但不替代用户的返回动作

### 卡片表现

- 失效图片直接复用普通图片卡片组件
- 不引入独立卡片类型
- 缩略图区域增加轻蒙层
- 蒙层文案使用 `原文件已丢失` 或 `已失效`
- 保持卡片主体结构、尺寸、选择逻辑、点击逻辑与普通卡片一致

### 治理动作

- 批量动作：`清理失效图片`、`删除选中`
- 单卡动作优先通过详情承接 `重新定位` 和 `删除记录`
- 不把失效治理做成长期并列主路径

## 未完成任务恢复

- 当存在未完成入库任务时，图库管理页应展示恢复横幅，并提供 `继续导入` 和 `放弃剩余图片`
- `放弃剩余图片` 只影响未完成的任务，不影响已成功入库的图片
- `放弃剩余图片` 必须真实生效，不让用户在刷新后再次看到同一批待处理任务
- 恢复提示应明确这是“上次导入中断后仍有未处理图片”
- `继续导入` 开始后，当前页应用进行中状态承接，而不是继续显示待决策提示
- `继续导入` 完成后，用户应能立即看到本次恢复结果摘要
- 若本次恢复仍有失败项，失败项明细应按本次恢复结果查看

## 导入过程中的操作边界

- 导入中或恢复处理中，不应把整页粗暴锁死
- 浏览、搜索、预览、复制、查看详情等非破坏性操作应尽量允许
- 会改变图库基数或打乱任务边界的操作应限制，例如：
  - 在当前导入尚未结束前再发起新一轮导入
  - 删除图片、批量删除、清理失效图片
  - 清空图库等重置类能力
- 被限制的操作必须给出可理解原因和下一步，而不是静默禁用

## 导入统计与异常提示的生命周期

所有提示都必须有明确出现条件和退场条件；没有退场条件的提示，最终都会变成页面噪音。

### 提示优先级

1. 未完成恢复任务待决策
2. 恢复进行中
3. 最近一批导入存在失败项
4. 已发现失效图片待处理
5. 最近一批导入成功摘要
6. 无提示，进入普通图库页

### 生命周期规则汇总

| 提示类型 | 出现条件 | 退场条件 |
| --- | --- | --- |
| 导入进行中 | 普通导入开始且未完成 | 导入完成或被恢复进行中接管 |
| 恢复任务提示 | 存在 pending task 且未决策 | 用户继续导入、放弃剩余图片，或状态收口 |
| 最近一批导入结果摘要 | 一轮导入刚完成，且仍在短时承接窗口内 | 用户查看主动作、主动忽略、超出承接窗口，或新结果覆盖 |
| 失败项展开 | 用户点击 `查看失败项` | 用户收起、执行主动作、或新结果覆盖 |
| 已发现失效图片提示 | 已发现失效记录且数量 > 0 | 失效记录归零或被更高优先级问题压下 |
| 本次新增定位反馈 | 用户点击 `查看本次新增` | 切换视图或新结果覆盖 |

补充规则：

- `最近一批导入结果摘要` 只用于承接“刚完成”的结果，不允许跨天常驻在图库页首屏
- 首版承接窗口按 `30 分钟` 控制；超过后即使用户未主动处理，也应自动退场
- 用户执行了默认主动作，例如 `查看本次新增`，或主动选择忽略后，同一批结果不应在后续刷新或重进页面时再次顶上来
- 恢复结果摘要仍按恢复状态机单独承接，不受普通导入历史摘要的短时窗口限制

## 删除与详情

- 单张删除和批量删除都要二次确认
- 删除只清数据库记录和索引，不影响原始图片
- 详情页用于查看大图、元数据、异常状态和快捷操作
- 失效图片详情需提供失效状态说明、识别线索展示、`重新定位` 和 `删除记录`
- GIF 在详情页中自动播放
- 详情页是图库管理和搜索结果共同复用的单图工作面板

## 内部维护能力

- `清空图库` 属于内部重置能力，不作为普通产品功能暴露
- 它只删除数据库记录、索引和相关缓存，不删除原始图片
- 若需要对普通用户说明该能力，应明确标成维护工具或内部入口

## 不负责

- 不替代搜索页承接按记忆线索找图
- 不承载底层检索调参
- 不把角色示例图补强做成普通用户一级主导航
- 不把导入失败、已发现失效图片和搜索召回问题重新混成单一“异常图片”概念

## 验收要点

- 用户能在冷启动页和图库管理页看到不同形态但同一套协议的导入能力
- 图片和文件夹导入共享同一套导入状态与进度逻辑
- 图库管理页常驻主视图只保留 `全部图片`
- 导入完成后，系统能向用户反馈新增、已存在、失败三类结果
- `本次新增` 通过 `全部图片` 内的定位反馈承接，不新增常驻视图
- 失败项在当前页展开，并给出克制的下一步动作
- 只有在系统已发现失效记录时，页面才出现失效治理入口
- `查看失效图片` 进入同页过滤态，并保留 `查看全部图片`
- 失效图片复用普通图片卡片，只通过失效态样式区分
- 当存在未完成入库任务时，图库管理页会出现恢复承接，用户可以继续导入或真实有效地放弃剩余图片
- 用户完成一次中断恢复后，图库管理页会优先反馈本次恢复结果
- 导入中或恢复处理中，用户仍可执行非破坏性操作；高风险操作会被明确限制并给出原因
- `清空图库` 不会被误当成普通用户设置项
