### 设置模块优化与高级能力收口功能（优先级：P2）

- 状态：approved
- Spec ID：`SPEC-SETTINGS`

#### 功能描述
本次迭代不再把“设置”视为一个继续扩展的稳定模块，而是回到产品定位本身，重新判断设置模块中各项能力是否真的属于用户产品功能。

经复盘后，本次结论如下：
- `私有角色库` 解决的是“模型不认识冷门角色，导致无字图难搜”的真实用户问题，但当前实现仍先以开发模式维护入口承接
- `开发调试模式`、`重新生成图像索引`、`清空图库` 本质上都是开发期或维护期能力，不应继续以普通用户功能的方式暴露
- `默认打开方式` 与“恢复上次窗口状态”不属于真实用户需要的稳定设置，应从设置模块中删除
- 产品默认入口应始终是侧栏主工作台，展开态只通过显式进入图库管理触发
- 旧设置模块直接下线，不再保留为独立页面、轻量入口或过渡容器

#### 设计目标
- 普通用户默认只看到和“找图”“整理图库”“提升可检索性”相关的能力
- 私有角色库以“高级检索增强能力”身份存在，而不是以“必须理解的设置概念”出现
- 开发期和维护期工具不再污染普通用户信息架构
- 产品主入口保持稳定，不因伪需求设置破坏“聊天伴随”体验
- 设置相关内容整体收缩，不再承担内部工具面板职责

#### 范围说明
**本期做什么：**
- 明确四类能力的产品定位与归属层级
- 删除设置模块中的“默认打开方式”和“恢复上次窗口状态”口径
- 明确私有角色库从“设置高级能力”转为“图库检索增强能力”
- 明确开发调试模式、重新生成图像索引、清空图库的内部化方案
- 明确旧设置模块直接退出当前产品信息架构
- 收口侧栏更多菜单、图库管理与内部维护工具之间的入口关系

**本期不做什么：**
- 不改变私有角色主路本身的算法逻辑
- 不改变图库索引或数据库底层实现机制
- 不在本期设计完整 CLI 语法或隐藏维护入口的具体实现
- 不在本期确定所有开发构建与正式构建的打包差异细节

#### 问题复盘
原设置模块中混入了四种完全不同性质的能力：

| 能力 | 原始加入原因 | 当前真实属性 | 是否属于普通用户产品能力 |
|------|-------------|-------------|------------------------|
| 私有角色库 | 解决模型不认识角色导致的搜索失败 | 高级检索增强能力 | 否，当前仅开发模式可见维护入口 |
| 开发调试模式 | 调试排序和召回构成 | 内部调试工具 | 否 |
| 重新生成图像索引 | 入库流程调整后的手动补救按钮 | 维护工具 / 升级兜底能力 | 否 |
| 清空图库 | 调试冷启动与重置本地状态 | 高风险内部重置工具 | 否 |

与此同时，`默认打开方式` 这类设置在之前讨论中被确认不成立：
- 用户不会自然地产生“我希望以后默认打开大窗口”这种稳定需求
- “恢复上次窗口状态”会把一次临时管理动作错误升级为下一次启动入口
- 对聊天伴随工具而言，启动入口应始终稳定为侧栏主工作台，而不是让用户或系统决定“下次是否先进管理态”

因此，本次调整的重点不是“怎么把这些能力重新排布到设置页里”，而是“哪些能力根本不该再属于设置模块，以及设置模块本身为何应直接下线”。

#### 调整结论

**1. 私有角色库**

私有角色库应重新定义为：
- 面向高级用户的检索增强能力
- 用示例图片将“模型不认识的角色”翻译为可召回的向量锚点
- 本质上属于图库可检索性治理，而不是应用偏好设置

定位结论：
- 不再作为“设置模块”的核心内容
- 不再以“普通设置项”方式呈现
- 当前实现归入开发模式下的 `图库管理 > 高级能力`，并通过开发路由补充进入
- 在侧栏主工作台中最多只保留一个弱引导入口，不应与找图主路径同权

对外语义要求：
- 强调它解决的是“冷门角色 / 私有对象搜不到”的问题
- 不让普通用户误以为必须先配置角色库才能搜索
- 不把“角色库”作为搜索主路径中的默认概念

**2. 开发调试模式**

开发调试模式应重新定义为：
- 面向开发者和维护人的内部调试能力
- 用于排查排序构成、召回结果、主路判断和得分解释
- 不面向普通用户，也不应继续成为稳定产品入口

定位结论：
- 从普通设置中移除
- 当前实现只在开发构建的搜索页开发工具浮层中可见
- 不进入正式产品信息架构
- 不出现在侧栏更多菜单和图库管理普通路径中

**3. 重新生成图像索引**

重新生成图像索引应重新定义为：
- 面向升级、迁移或异常修复的维护工具
- 用于模型更新、入库流程变化、索引版本不一致时的兜底修复
- 不属于普通用户日常会主动理解和使用的功能

定位结论：
- 不在普通设置中保留按钮
- 当前实现短期保留在开发构建的搜索页开发工具浮层中
- 长期目标应是由系统检测版本变化并自动触发或提示，而不是让用户自己理解何时需要重建
- 不作为侧栏更多菜单中的常规项目显示

**4. 清空图库**

清空图库应重新定义为：
- 开发期和测试期的内部重置工具
- 高风险、低频、非普通用户目标
- 不属于正式产品功能

定位结论：
- 不作为普通用户正式 UI 能力存在
- 当前实现仅在开发构建的搜索页开发工具浮层中保留入口
- 未来可继续迁移到 CLI、开发命令或测试工具

**5. 默认打开方式**

默认打开方式应明确判定为伪需求：
- 普通用户不会以“我希望默认进入展开窗口”来理解自己的需求
- 展开态是任务态，不是启动偏好
- 聊天伴随型工具的启动入口应稳定，而不是基于上次状态波动

定位结论：
- 从设置模块移除
- 不再提供“默认打开方式”或“恢复上次窗口状态”给用户配置
- 应用始终从侧栏主工作台启动
- 图库管理始终通过显式操作进入展开态

**6. 设置模块去留**

设置模块在本轮优化后的结论是：
- 旧设置模块直接下线
- 不再作为独立一级页面存在
- 不再保留为侧栏轻量入口
- 不再承担“高级能力总入口”职责
- 其原有内容按能力类型拆分并迁入图库管理或内部工具层

#### 调整后信息架构图
```text
表情包检索器
├── 侧栏主工作台
│   ├── 首页 / 搜索
│   ├── 侧栏更多菜单
│   │   ├── 进入图库管理
│   │   └── 少量帮助或说明入口
│   └── 开发模式下可打开开发工具浮层
├── 展开态
│   └── 图库管理
│       ├── 全部图片
│       ├── 导入结果承接：查看失败项 / 查看本次新增
│       ├── 已发现失效图片入口（按需出现）
│       ├── 高级能力（仅开发模式）
│       │   └── 私有角色库
│       └── 不向普通用户暴露重建索引与清空图库
└── 内部工具
    ├── 搜索页开发工具浮层中的开发调试模式
    ├── 搜索页开发工具浮层中的重建图像索引
    └── 搜索页开发工具浮层中的清空图库
```

#### 页面职责调整
| 区域 | 应承接内容 | 不应承接内容 |
|------|-----------|-------------|
| 侧栏主工作台 | 找图、复制、快速进入图库管理 | 开发调试、索引维护、危险重置、角色维护主入口 |
| 图库管理 | 导入、整理、异常排查、私有角色检索增强 | 普通设置页、排序调试面板、清空图库 |
| 内部工具层 | 调试搜索、重建索引、重置数据 | 普通用户高频路径 |

#### 用户流程说明

**主流程 1：普通用户继续找图**
1. 用户打开应用，始终进入侧栏主工作台
2. 用户直接搜索或点击候选图
3. 用户不会接触角色库、调试模式、索引重建或清空图库等概念

**主流程 2：高级用户提升冷门角色检索能力**
1. 用户从侧栏进入 `图库管理`
2. 用户在开发模式下进入图库管理的 `高级能力`
3. 用户看到私有角色库入口及说明
4. 用户补充角色名、别名、示例图片
5. 系统将该角色转化为可用于检索的向量锚点
6. 用户返回搜索路径，使用“角色名 + 动作”进行检索

**主流程 3：用户查看侧栏附加入口**
1. 用户在侧栏中打开更多菜单
2. 用户看到 `进入图库管理` 和少量帮助或说明入口
3. 用户不会在这里看到私有角色库、调试模式、重建索引、清空图库或旧设置模块入口

**主流程 4：维护人排查搜索问题**
1. 维护人在开发构建的搜索页开发工具浮层中开启调试模式
2. 用户查看召回构成、主路判断和排序得分
3. 维护人根据问题继续调整模型、策略或数据
4. 普通用户界面不暴露这些信息

**主流程 5：升级后重建索引**
1. 系统检测到模型版本、索引版本或入库结构变化
2. 系统自动触发重建流程或向维护人提示需要处理
3. 当前实现如需人工介入，则通过开发构建的搜索页开发工具浮层执行 `重新生成图像索引`

**主流程 6：开发者重置本地环境**
1. 开发者通过开发构建的搜索页开发工具浮层、CLI 或测试命令执行 `清空图库`
2. 系统清空记录和索引数据
3. 普通用户产品界面不提供该入口

#### 交互说明

**1. 侧栏更多菜单**

页面元素：
- `进入图库管理`
- 帮助或说明入口

交互要求：
- 菜单应继续保持极简
- 不在侧栏更多菜单中堆放高级能力和维护工具
- 不再保留旧设置模块入口

**2. 私有角色库入口**

页面元素：
- `图库管理` 中的轻量角色增强图标入口
- 悬浮或聚焦提示中的入口说明
- 入口动作：`角色搜索增强`

交互要求：
- 入口应出现在图库管理语境中，而不是普通设置语境中
- 入口说明应强调“提升冷门角色检索能力”
- 不应让用户误以为这是日常必须配置项
- 入口说明不常驻展开；应在悬浮、聚焦或等价轻提示中直接说明适用场景：搜索词里没有字、模型又认不出角色时，可用示例图补强

文案建议：
- 标题：`角色搜索增强`
- 说明：`当搜索词里没有字、模型又认不出角色时，可以补几张示例图帮助搜索`

**3. 开发调试模式**

交互要求：
- 不在正式产品设置中显示
- 当前实现仅在开发构建的搜索页开发工具浮层中出现
- 结果页内部字段、得分项和主路判断只对内部人员可见

**4. 重新生成图像索引**

交互要求：
- 不在普通设置中显示
- 当前实现若仍需人工触发，则放在开发构建的搜索页开发工具浮层中
- 若可自动判断版本变化，应优先由系统自动处理或引导

**5. 清空图库**

交互要求：
- 不在正式用户 UI 中显示
- 不在图库管理普通路径中显示
- 当前实现若保留内部入口，则位于开发构建的搜索页开发工具浮层，且必须带强确认，并明确仅用于开发和测试

#### 页面与状态规则
| 场景 | 触发条件 | 页面行为 | 可执行动作 |
|------|---------|---------|-----------|
| 应用启动 | 首次或后续打开应用 | 始终进入侧栏主工作台 | 搜索、复制、进入图库管理 |
| 侧栏菜单打开 | 用户打开更多菜单 | 只展示进入图库管理和少量帮助入口 | 进入图库管理、查看说明 |
| 普通搜索 | 普通用户使用搜索 | 不展示调试信息和维护按钮 | 搜索、复制 |
| 图库增强 | 开发模式用户进入图库管理的高级能力区 | 展示私有角色库入口 | 打开角色搜索增强 |
| 开发调试 | 开发构建开启 | 展示调试信息 | 查看排序构成 |
| 索引维护 | 版本变化或内部维护需要 | 开发工具浮层或自动触发 | 重建索引 |
| 数据重置 | 开发或测试场景 | 仅开发工具或内部工具可执行 | 清空图库 |

#### 数据与状态要求
| 数据项 | 用途 | 最低要求 |
|-------|------|---------|
| `developer_mode` | 控制内部调试信息可见性 | 默认关闭，普通用户不可见 |
| `index_version` | 判断索引是否需要重建 | 可与模型版本或 schema 版本关联 |
| `model_version` | 判断嵌入模型是否发生变化 | 支持版本比对 |
| `maintenance_entry_source` | 记录内部维护入口来源 | 至少区分 `auto_prompt` / `hidden_ui` / `cli` |
| `private_role_library_enabled` | 控制私有角色库入口是否启用 | 当前实现由开发模式路由和入口共同约束 |

补充要求：
- 不再记录或暴露“默认打开方式”“恢复上次窗口状态”这类窗口偏好字段
- 当前实现里私有角色库入口依赖开发模式显示；若后续对高级用户开放，需要另行回写 spec
- 开发调试、索引重建和清空图库不应进入普通用户埋点体系的公开能力统计

#### 边界情况处理
| 场景描述 | 触发条件 | 系统行为 | 提示信息 |
|---------|---------|---------|---------|
| 模型不认识角色 | 普通搜索无法稳定识别冷门角色 | 当前实现通过开发模式入口提供私有角色库增强维护 | `可以用示例图片帮助系统认识这个角色` |
| 私有角色库未启用 | 当前构建未开放该能力 | 隐藏入口或展示禁用说明 | `当前版本未启用角色搜索增强` |
| 普通用户进入图库管理 | 未开启开发模式 | 看不到角色增强入口、调试和重置工具 | 无 |
| 索引版本变化 | 版本检测到不一致 | 自动触发重建或向维护人提示 | `检测到图库索引需要更新` |
| 用户误以为需要重建索引 | 普通使用场景 | 不展示该按钮 | 无 |
| 用户误以为可以清空图库 | 正式产品环境 | 不提供入口 | 无 |

#### 权限说明
- 可访问角色：普通用户、高级用户、维护人、开发者
- 权限规则：
  - 普通用户可访问搜索和图库管理
  - 高级用户当前不在正式 UI 中单独开放私有角色库入口
  - 维护人和开发者可访问调试模式与索引维护能力
  - 清空图库仅限维护人和开发者通过开发工具或内部工具执行

#### 迭代计划
- 后续可评估是否将“私有角色库”对外文案进一步收敛为“角色搜索增强”
- 后续可将索引重建逐步从人工维护按钮迁移为自动版本检测与自动修复
- 后续可把清空图库完全迁移出产品仓库 UI，只保留测试命令和脚本
- 后续应同步删除旧设置模块文档与总目录中的相关入口，确保当前口径唯一

#### 验收标准
**功能指标：**
- [ ] 私有角色库被明确定位为高级检索增强能力，而不是普通设置项
- [ ] 私有角色库入口位于开发模式下的图库管理相关语境中，而不是普通设置页中
- [ ] 开发调试模式不再作为正式用户设置项展示
- [ ] 重新生成图像索引不再作为普通设置按钮展示
- [ ] 清空图库不再作为普通用户正式 UI 按钮展示
- [ ] 不再展示“默认打开方式”或“恢复上次窗口状态”相关设置
- [ ] 旧设置模块入口不再出现在产品信息架构中

**体验指标：**
- [ ] 普通用户不会在主路径中接触开发调试、索引重建或清空图库
- [ ] 维护人和开发者能够理解私有角色库是为“搜不到的角色”服务的增强能力
- [ ] 产品整体感知更像“找图工具 + 图库治理”，而不是“带后台的桌面应用”
- [ ] 启动行为稳定，用户始终预期应用先进入侧栏主工作台

**可靠性指标：**
- [ ] 模型或索引版本变化时，系统能通过自动检测或开发模式受控入口完成修复
- [ ] 内部调试能力不会污染普通用户界面

#### 假设
- 假设私有角色库仍将长期存在，因为冷门角色检索问题无法完全依赖通用模型解决
- 假设后续仍会有模型、schema 或入库流程迭代，因此索引维护能力短期内不能完全删除
- 假设清空图库只服务于开发和测试，不存在稳定的普通用户场景
- 假设旧设置模块退出后，用户不需要额外的“设置页”也能完成主路径与高级路径

- 私有角色库的产品层文案是否正式改名为“角色搜索增强”
- 重建索引是保留一个隐藏 UI 入口，还是彻底改成自动检测 + 内部命令
- 清空图库是否需要保留一个仅开发构建可见的入口，还是完全迁移到 CLI
