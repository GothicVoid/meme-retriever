# 私有角色库维护

- 状态：approved
- Spec ID：`SPEC-ROLELIB`

## 功能描述

本文件统一定义角色主体识别增强能力的定位、维护内容、入口层级和与搜索主路的关系。

角色主体识别增强解决的是“图片里没字、模型不认识角色、但用户希望按角色名搜到图片”的问题。它是正式的搜索补强能力，不是开发模式专属工具，也不是通用设置页的一部分。

## 适用对象

- 项目维护人
- 高级用户
- 需要为少量冷门角色建立稳定召回锚点的人

它适合少量高价值角色，不适合作为每次搜索失败后的默认重操作。

## 维护内容

- 角色名与别名
- 示例图片
- 当前规则下的结构校验与保存前风险提示

维护工具应优先暴露少量核心字段，不继续外露旧知识库时代的复杂兼容字段和调参语义。

明确不再作为正式维护内容的能力：

- `category`、`match_mode`、`notes`、`priority` 等旧知识库兼容字段
- `match_terms`
- 补充搜索词或查询扩展词库
- 基于 OCR / 文件名的自动角色打标

## 入口层级

- 不作为普通用户一级导航
- 可以作为图库管理页中的正式入口存在，也可以在特定搜索失败场景下被引导进入
- 入口应低干扰呈现，不压过导入、失效治理和任务恢复
- 对外文案要强调“按角色名搜不到时的补强”或“维护角色示例图”，而不是抽象的知识库概念
- 图库管理页中的正式入口应采用“轻量说明 + 次级按钮”形态，先解释适用场景，再提供进入动作
- 入口说明应直接回答“什么时候需要它”：当搜索词里没有字、模型又认不出角色时，可以补几张示例图帮助系统学会识别

## 与搜索主路的关系

- 只有当查询明显包含角色主体线索时，才适合触发私有角色增强
- 私有角色增强只负责识别角色主体，不负责补充动作、表情、场景等非主体线索
- 识别出角色主体后，剩余查询仍继续走 OCR / 语义主路完成动作、表情、场景等细排
- 角色库命中失败时，系统应能降级回 OCR / 语义主路，而不是把搜索彻底带偏
- 当搜索失败属于“角色名搜不到，但图片可能已在库里”的情况时，系统可以引导用户进入角色示例图补强
- 页面和文案应帮助用户理解：这是偶尔使用的补强能力，不是每张图都要维护

## 维护页交互原则

- 维护页优先服务“少量补强，尽快试一下”，不应做成高学习成本的后台
- 页面首屏应直接进入操作区，只保留一行适用场景说明，不再堆叠大标题区和说明卡片
- 维护流程应尽量收敛为最短闭环：选择角色或新建角色、补充名称/别名/示例图、保存、回到搜索页真实试用
- 页面不再要求用户在维护页内理解“测试召回”或其他内部验证概念；保存后应直接引导用户回搜索页按真实叫法试用
- 页面交互只围绕 `角色主名称`、`别名`、`示例图` 和结果导向的风险提示展开，不再暴露“匹配线索”或其他旧知识库字段
- 示例图是必填项；缺少示例图时应直接阻止保存，而不是提示“降级为普通搜索”
- 表单字段应保持紧凑，单行输入框不做大块卡片化展示；短文本场景如 `角色主名称`、`别名` 应使用单行窄宽输入，不占满整块编辑区
- `角色主名称` 与 `别名` 默认应并排放置，优先服务少量维护与快速录入，不为低频长内容预留大块输入区域
- 左侧角色列表应低干扰且节省宽度，不应为了少量角色占用过多编辑空间
- 当角色库为空时，页面应切换到明确的“创建第一个角色”冷启动状态，而不是继续展示完整编辑器骨架
- 空角色库状态下不展示左侧角色栏、筛选框、空列表占位或高权重保存动作，避免让用户误以为自己漏看了已有内容
- 空角色库状态应只回答三件事：这里是做什么的、第一步该点哪里、保存后能得到什么结果；主按钮应直接触发新建第一个角色
- 结构校验应由系统在用户编辑后自动刷新，不再要求用户额外点击一次 `校验`
- 风险提示区不应再额外写“系统说明”或替用户下结论；有问题时直接列提示，无问题时只写可直接保存
- 页面超过一屏时必须可以稳定纵向滚动，不能让示例图或校验区域被壳层裁掉
- 维护页必须提供明确返回动作，默认返回 `图库管理`，不让用户困在独立维护页里

## 不负责

- 不承担普通图片标签体系
- 不面向普通用户做大规模内容运营
- 不替代搜索主路的基础 OCR 和语义能力
- 不承担动作、表情、场景等非主体词的查询扩展
- 不承担 OCR / 文件名自动角色打标
- 不作为普通用户一级主导航
- 不再以“开发模式专属能力”作为产品口径

## 验收要点

- 维护人可以创建、编辑并保存少量私有角色卡片；有问题时直接提示，无问题时明确可保存
- 角色卡片字段足够少，维护成本可控
- 私有角色库不会误入普通用户主导航
- 运行时命中和维护工具校验使用同一套语义
- 角色主体识别增强作为正式功能存在，只用于角色名搜索补强
- spec 中不把 `match_terms`、补充搜索词、自动角色打标定义为正式能力
- 保存后会明确引导用户回搜索页用真实搜索词试用，而不是要求用户在维护页内理解测试机制
- 对普通用户的入口文案能表达“按角色名搜不到时，可以补几张示例图”
- 图库管理页中的入口说明能表达“搜索词里没有字、模型又认不出角色时，可以补几张示例图”
- 冷启动空角色库时，页面首屏应将“新建第一个角色”作为唯一主行动，且不出现左侧角色栏、筛选框或“当前筛选下没有角色”等伪列表提示
