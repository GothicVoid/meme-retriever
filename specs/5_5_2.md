#### 功能描述
提供一套面向项目维护人或高级用户的私有角色库维护工具，用于创建、编辑、校验和测试少量私有角色卡片。角色库主要用于解决“无字图 + 冷门角色 + 模型不认识角色”的搜索问题，不再承担通用知识库和全局查询标准化的中心职责。

#### 目标说明
- 降低维护人或高级用户为冷门角色建立视觉锚点的成本
- 让角色名、别名与示例图维护流程清晰可控
- 支持在当前项目规则下快速验证某个私有角色能否稳定召回目标图片
- 保持角色库格式与主项目实现同步演进，避免维护工具与项目规则脱节

#### 工具形态
第一阶段建议实现为“同仓库内的内部维护工具”或开发模式下可见的角色管理页面。

推荐形态优先级：
1. CLI 工具
2. 仅开发模式可见的维护页面
3. 面向正式用户的高级管理入口

第一阶段结论：
- 工具与当前项目放在同一仓库
- 默认采用 CLI 形式
- 可同步提供开发模式专用页面，作为当前主要可视化维护入口
- 不作为普通用户默认可见功能发布

#### 当前实现约束
- 当前阶段允许内部类型、命令前缀和文件名继续沿用 `kb` / `knowledge_base` 命名
- 但所有面向维护人的页面标题、导航入口、状态文案、测试描述应统一使用“私有角色库”语义
- 本阶段目标是先完成“外部语义收敛”，不要求立即完成所有内部类型重命名
- 真正的角色卡片 schema 收缩与搜索主路接入，在后续步骤单独推进
- 维护页面在状态未加载完成前，不应使用 `app_data/knowledge_base.json` 作为默认展示文案

#### 使用角色
| 角色 | 描述 | 权限 |
|-----|------|------|
| 项目维护人 | 负责私有角色库录入、修订和规则治理的内部人员 | 可创建、编辑、校验、测试和导出角色库 |
| 高级用户 | 对特定冷门角色有维护需求的用户 | 可在受控入口中维护少量私有角色 |
| 普通用户 | 应用使用者 | 默认不接触角色库维护能力 |

#### 流程说明
1. 维护人启动私有角色库维护工具
2. 选择创建角色、编辑角色、校验角色库或测试召回
3. 工具根据当前角色库 schema 执行输入校验
4. 维护人为角色填写名称、别名并关联示例图
5. 工具输出修改结果或校验报告
6. 维护人可进一步执行召回测试，验证角色名或别名能否正确召回目标图片

#### 核心功能
**基础能力：**
- 创建私有角色
- 编辑已有角色
- 删除角色
- 查询和列出角色
- 按角色名或别名搜索角色

**校验能力：**
- 校验角色库结构是否合法
- 校验角色名是否唯一
- 校验别名是否重复
- 校验示例图是否存在且可用
- 检测角色名或别名冲突

**测试能力：**
- 输入角色名、别名或记忆线索，查看命中的角色卡片
- 预览角色示例图 embedding 召回结果
- 在修改角色后快速回归测试召回效果

**输出能力：**
- 写回私有角色库文件
- 导出格式化后的角色库文件
- 输出校验报告或冲突报告

#### 推荐命令能力
若采用 CLI 形式，建议支持以下命令：
- `private-role list`
- `private-role add`
- `private-role edit`
- `private-role delete`
- `private-role validate`
- `private-role test-search --query "<角色名或别名>"`
- `private-role format`

#### 数据规则
工具应遵循主项目当前私有角色 schema，不允许定义独立于主项目之外的格式。

第一阶段建议至少支持以下字段：
- `name`
- `aliases`
- `example_images`
- `notes`（可选）
- `match_terms`（可选，用于动作 / 表情 / 场景线索）

工具不应承担通用知识库、通用标签分类和全局 OCR 匹配测试职责。

说明：
- 私有角色库对外推荐 schema 应以 `name / aliases / example_images / notes` 为准
- 旧字段 `canonical / description` 仅保留为兼容读取，不应继续作为新增条目的首选字段
- `match_terms` 继续保留，作为角色动作、表情、场景线索字段
- 维护工具 UI 第一阶段只应直接暴露 `name / aliases / match_terms / notes / example_images`
- `category / match_mode / priority` 可在内部保留兼容，但不应继续作为面向维护人的可编辑字段

#### 当前角色卡片 schema
维护工具输出应满足以下规则，才能被搜索运行时视为“可用私有角色卡片”：

| 字段 | 角色语义 | 约束 |
|---------|---------|------|
| `name` | 角色主名称 | 必填，运行时视为角色名 |
| `aliases` | 角色别名 | 可选，建议填写常见昵称或圈内叫法 |
| `example_images` | 角色示例图 | 至少 1 张，私有角色主路依赖此字段 |
| `notes` | 角色备注 | 可选，仅供维护人理解 |
| `match_terms` | 角色动作 / 表情 / 场景线索 | 可选，用于测试和后续扩展 |
| `category` | 条目分类 | 当前私有角色应固定使用 `person` |

补充说明：
- 若 `category != person`，该条目不应被视为私有角色卡片
- 若缺少 `example_images`，该条目可保留在维护工具中，但运行时只作为弱文本线索，不触发私有角色主路
- 维护工具应引导用户优先创建“少量但有示例图”的高质量角色卡片，而不是大量空壳条目
- 若读到旧字段 `canonical / description`，系统应兼容映射到 `name / notes`
- 维护工具新建角色时，应默认按私有角色卡片处理，不要求维护人手动选择 `person`
- 若读取到历史条目的 `match_mode / priority`，可继续兼容写回，但不要求在 UI 中暴露编辑入口
- 校验提示应优先使用“是否可启用私有角色主路”的表述，而不是泛化的分类建议文案

#### 边界情况处理
| 场景描述 | 触发条件 | 系统行为 | 提示信息 |
|---------|---------|---------|---------|
| 角色名重复 | 新增或编辑后与已有角色冲突 | 阻止保存 | "角色名已存在" |
| 别名重复 | 同一角色或跨角色存在重复别名 | 阻止保存或输出冲突提示 | "检测到重复别名" |
| 示例图缺失 | 角色未绑定示例图 | 允许保存，但提示无法启用角色召回 | "缺少示例图，将降级为普通搜索" |
| 分类不是 person | 条目被误配为 meme/source/custom | 允许保存，但提示不会启用私有角色主路 | "当前条目不是私有角色卡片" |
| 查询无命中 | 测试输入未命中任何角色 | 返回空结果 | "未命中任何私有角色" |
| 文件格式损坏 | 角色库文件不合法 | 阻止写回 | "角色库文件格式错误" |

#### 产品边界
**本期做什么：**
- 面向维护人的私有角色录入、校验与召回测试工具
- 与当前项目规则强绑定
- 本地使用，不依赖联网
- 开发模式页面与 CLI 的对外文案统一为私有角色库语义

**本期不做什么：**
- 通用知识库后台
- 面向所有普通用户开放的大型角色管理系统
- 在线协作编辑
- 云端同步角色库

#### 验收标准
**功能指标：**
- [ ] 维护人可通过工具完成私有角色增删改查
- [ ] 可对角色库执行结构校验和冲突检测
- [ ] 可通过输入角色名或别名测试召回结果
- [ ] 可安全写回格式化后的角色库文件

**体验指标：**
- [ ] 常见维护操作无需直接手写 JSON
- [ ] 校验错误信息清晰可定位
- [ ] 工具输出与主项目私有角色规则保持一致
