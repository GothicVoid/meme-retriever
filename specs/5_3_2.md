#### 功能描述
在现有手动标签功能基础上，增加入库时自动生成标签、标签分类、分类权重检索和详情页结构化编辑能力。该能力面向本地离线应用设计，第一阶段以 OCR 结果和本地知识库文本匹配完成自动打标，文件名仅作为可选弱辅助信号，CLIP 自动打标能力留作后续增强，不要求联网，也不依赖在线模型服务。

#### 目标说明
- 提升表情包搜索命中率，尤其是梗、人物、出处等语义型检索场景
- 减少用户手动录入标签成本
- 保持用户对自动标签的最终控制权，允许修改、删除和补充
- 在现有详情页与搜索模块上平滑演进，避免一次性引入过多后台管理能力

#### 范围定义
**本期做什么：**
- 入库时自动生成结构化标签
- 支持四类标签：梗、人物、出处、自定义
- 为自动标签记录来源、自动/手动状态和基础置信度
- 搜索阶段按标签分类执行差异化加权
- 详情页展示分类标签，并支持用户编辑分类和文本

**本期不做什么：**
- 在线知识库同步
- 热门梗自动抓取
- 多角色权限系统
- 用户纠错反馈自动训练
- 复杂的知识库后台管理界面

#### 子需求拆分
1. `5.3.2.1` 自动标签录入流程
2. `5.3.2.2` 标签分类与存储模型
3. `5.3.2.3` 标签搜索加权规则
4. `5.3.2.4` 详情页标签展示与编辑增强

#### 设计原则
- 分类是标签的业务属性，搜索权重是检索策略，二者需要关联但不应写死耦合
- 自动标签是建议值，不是最终真值，用户编辑结果优先级最高
- 入库阶段负责“生成标签”，搜索阶段负责“解释标签的重要性”
- 自动打标需要支持降级，不因为单一路径失败而中断整张图片入库
- 通用无效标签应尽量少写入，避免污染标签库和自动补全结果
- OCR 是第一阶段自动打标的主要输入信号，CLIP 保持为搜索能力和后续增强能力
- 文件名默认不作为可靠主信号，仅在满足可读文本条件时参与弱匹配
- CLIP 文本相似度是“图片检索标签文本”，示例图匹配是“图片检索标签样本图”，两者职责不同
- 后续若启用示例图，优先将其用于 `person`、`source` 等视觉识别更强的分类

#### 总体流程
```
图片入库
  → OCR 提取 + 可选文件名解析
  → 知识库生成候选标签
  → 文本匹配、去重与阈值过滤
  → 写入结构化标签
  → 更新搜索索引
  → 用户在详情页查看和修正
```

#### 与现有需求关系
- 依赖 `5.2.1 添加图片功能` 的入库与索引流程
- 扩展 `5.3.1 标签编辑功能` 的标签数据结构与交互能力
- 扩展 `5.4.1 详情页展示功能` 的标签展示和编辑区域

#### 自动打标信号分层策略
第一阶段：
- 主信号：OCR + 知识库文本匹配
- 弱信号：可读文件名
- 不启用：CLIP 文本相似度、示例图匹配

第二阶段候选增强：
- 优先补充：示例图匹配
- 适用分类：`person`、`source`
- 使用方式：候选补充或候选重排，不作为唯一高置信判定依据

第三阶段语义增强：
- 可选补充：CLIP 文本相似度
- 适用场景：无字图、抽象视觉反应图、知识库未配置示例图的标签
- 使用方式：低置信候选召回或 rerank 信号，不建议直接替代文本规则

#### 总体验收标准
**功能指标：**
- [ ] 入库完成后可自动生成结构化标签
- [ ] 用户可在详情页编辑自动标签文本与分类
- [ ] 搜索可识别并利用分类标签进行排序
- [ ] 自动标签失败时不影响图片入库

**约束指标：**
- [ ] 保持本地离线运行
- [ ] 新能力不破坏现有手动标签检索
- [ ] 无知识库命中时不写入无意义高频垃圾标签
