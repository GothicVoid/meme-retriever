# 1. 产品概述

## 1.1 背景与问题
在日常聊天场景中，用户往往知道自己想发送哪张表情包，却难以在本地图库中快速定位。传统的文件管理方式只能通过文件名或时间查找，当图库规模达到数百上千张时，"翻图"成本极高，严重影响沟通体验。

## 1.2 产品定义
一款基于语义检索的本地表情包管理工具，让用户通过自然语言描述即可秒找目标图片。

## 1.3 目标用户
| 用户类型 | 用户特征 | 核心诉求 | 使用场景 |
|---------|---------|---------|---------|
| 重度表情包用户 | 本地存有500+张表情包，日均使用10次以上 | 快速定位、快速复制、批量管理 | 日常聊天、社群互动 |
| 内容创作者 | 需要引用特定梗图制作内容，对图片出处有要求 | 按出处/角色/台词精准检索 | 短视频制作、文章配图 |
| 轻度用户 | 图库100张以内，偶尔使用 | 界面简洁、操作简单 | 偶尔发送表情包 |

## 1.4 产品目标
**业务目标：**
- 获得广泛的用户群体和积极的市场反馈
- 用户日均使用频次达到较高水平
- 通过社区口碑和用户推荐实现自然增长

**用户目标：**
- 降低用户查找表情包的时间成本
- 提升表情包管理效率

**成功指标：**
- 检索响应时间 < 500ms（图库≤5000张）
- 用户找到目标图片的平均操作次数 ≤ 2次
- 用户留存率（7日）达到较高水平

## 1.5 范围说明
### 本期做什么
- 本地语义检索功能
- 图库基础管理（添加、删除、查看）
- 标签系统
- 快捷复制功能
- 基础设置项

### 本期不做什么
- 本地图片文件管理
- 云端同步功能
- 表情包社区/分享功能
- 在线表情包下载
- 移动端适配
- 图片编辑功能
- 表情包制作工具

# 2. 系统概要

## 2.1 系统组成
系统由以下核心模块组成：

1. **检索引擎模块**：负责语义理解与图片匹配
2. **图库管理模块**：负责图片的增删改查
3. **索引构建模块**：负责图片内容分析与索引生成
4. **标签系统模块**：负责用户自定义标签管理
5. **设置模块**：负责应用配置管理
6. **UI交互模块**：负责用户界面展示与交互

## 2.2 数据流图
```
用户输入查询
    ↓
检索引擎模块 ← 读取 → 索引数据库
    ↓
计算语义得分
    ↓
返回结果列表 → UI展示
    ↓
用户操作（复制）
    ↓
更新使用频次 → 索引数据库

添加图片流程：
用户选择图片 → 图库管理模块 → 索引构建模块
                                    ↓
                            分析图片内容（OCR）
                                    ↓
                            生成索引 → 索引数据库
```

### 5.2.3 搜索输入功能（优先级：P0）

#### 功能描述
提供智能搜索输入框，支持用户通过多种方式描述目标图片，自动触发语义检索。

#### 流程说明
1. 用户在搜索框中输入查询文本（中英文混合）
2. 系统检测输入停顿300ms后自动触发检索
3. 检索引擎分析查询意图，匹配图库中的图片
4. 返回按相关性排序的结果列表
5. 用户可点击清空按钮清除输入

#### 交互说明
**页面元素：**
- 搜索框：占据页面顶部，圆角矩形设计
- 搜索图标：位于输入框左侧
- 清空按钮：输入内容后显示在右侧，点击清空输入

**交互行为：**
- 页面加载时自动聚焦搜索框
- 支持中英文、标点符号输入
- 实时Debounce，停顿300ms后触发检索
- 按Esc键清空搜索内容
- 按Ctrl/Cmd + F重新聚焦搜索框

**反馈提示：**
- 输入时无加载状态（Debounce期间）
- 检索开始后显示加载动画（如果耗时>100ms）
- 无结果时显示：「没找到相关图片，试试其他描述？」

#### 算法说明

检索排序采用**相关性+热度**双因子融合机制：

**最终得分公式：**
```
Final_Score = α·Relevance + β·Popularity
```

**权重配比：**
- α（相关性权重）= 0.75
- β（热度权重）= 0.25

---

**一、相关性得分 Relevance**

采用三路融合策略，加权求和计算相关性得分：

```python
Relevance = w1 · S_keyword + w2 · S_ocr + w3 · S_clip
```

**三路得分计算规则：**

| 路径 | 计算方式 | 得分范围 |
|-----|---------|---------|
| S_keyword（实体匹配） | 完全匹配=1.0<br>部分匹配=0.8<br>关联标签=0.5 | 0~1.0 |
| S_ocr（文字匹配） | 字符覆盖率 <br>覆盖率 = len(query ∩ ocr_text) / len(query) | 0~1.0 |
| S_clip（语义相似度） | (cosine_similarity + 1) / 2 | 0~1.0 |

**三路权重配置：**
```
默认值：w1 = 0.3, w2 = 0.4, w3 = 0.3
约束条件：w1 + w2 + w3 = 1
```
（支持在设置页面手动调整，下次查询生效。调整时自动归一化确保总和为1）

---

**二、热度得分 Popularity**

基于使用频次的对数归一化：

```python
freq_score = log(1 + click_count) / log(1 + max_click)
```

对数化处理避免高频图片过度集中，给新图片更多曝光机会。

---

**三、边界情况处理**

| 场景 | 判断条件 | 处理规则 |
|-----|---------|---------|
| 冷启动（新图） | use_count = 0 | popularity = 0.1（给予较低初始值） |
| 相关性过低 | relevance < 0.2 | 直接过滤不展示 |

---

**计算示例：**

```
查询："你礼貌吗"

图片A：
- S_keyword = 0（无标签匹配）
- S_ocr = 1.0（完整包含"你礼貌吗"）
- S_clip = 0.85
- Relevance = 0.3×0 + 0.4×1.0 + 0.3×0.85 = 0.655
- click_count = 50, max_click = 200
- Popularity = log(51)/log(201) = 0.73
- Final_Score = 0.75×0.655 + 0.25×0.73 = 0.674

图片B：
- S_keyword = 0.5（关联标签"质疑"）
- S_ocr = 0.3（包含"你"）
- S_clip = 0.92
- Relevance = 0.3×0.5 + 0.4×0.3 + 0.3×0.92 = 0.576
- click_count = 0（新图）
- Popularity = 0.1（冷启动默认值）
- Final_Score = 0.75×0.576 + 0.25×0.1 = 0.457

结果：图片A排在前面（综合考虑相关性和热度）
```

### 5.2.4 结果列表展示功能（优先级：P0）

#### 功能描述
以网格形式展示检索结果，支持图片缩略图预览和基础交互，**带文件格式标识**。

#### 流程说明
1. 接收检索引擎返回的排序结果
2. 渲染前9张图片的缩略图（**GIF在缩略图状态下仅显示首帧，不播放动画**）
3. 在缩略图右上角显示文件格式标识
4. 用户滚动可查看"展示更多"
5. 点击"展示更多"后加载至最多21张
6. 超出21张不再展示

#### 交互说明
**页面元素：**
- 图片网格：3列布局，自适应间距
- 缩略图：等比缩放，最大尺寸150×150px
- **格式标识角标：位于缩略图右上角，半透明圆角矩形背景**
  - JPG/PNG：不显示角标（默认格式）
  - GIF：显示"GIF"标识，橙色背景
  - WEBP：显示"WEBP"标识，蓝色背景
- "展示更多"按钮：在第9张图片后显示（若结果>9张），点击后隐藏，下次查询时再按情况显示

**交互行为：**
- 单击图片：复制到剪贴板，显示Toast"已复制"
- 双击图片：打开单图详情页（**GIF格式在详情页自动播放动画**）
- 右键菜单：显示操作选项（删除、编辑标签、在文件夹中显示、查看详情）
- 鼠标悬停：显示半透明蒙层和操作图标，**GIF在缩略图状态仍显示首帧**

**反馈提示：**
- 复制成功：Toast提示"已复制"，1.5秒后消失
- 删除确认：模态框"确定删除这张图片吗？此操作不可撤销"

#### 算法说明
**缩略图生成规则：**

| 文件格式 | 生成策略 | 缓存策略 |
|---------|---------|---------|
| JPG/PNG | 等比缩放至150×150内，保持宽高比 | 首次生成后缓存 |
| GIF | 提取第一帧，等比缩放至150×150内 | 首次生成后缓存 |
| WEBP | 等比缩放至150×150内，保持宽高比 | 首次生成后缓存 |

**角标显示规则：**
```javascript
if (fileFormat === 'GIF') {
  显示角标: 'GIF', 背景色: '#FF9500'
} else if (fileFormat === 'WEBP') {
  显示角标: 'WEBP', 背景色: '#1890FF'
} else {
  不显示角标
}
```

#### 流程说明
1. 用户点击按钮选择添加方式

5. 触发索引构建（OCR、特征提取）
6. 更新图库列表，显示进度
7. 后台任务持久化：索引构建任务记录到本地队列
8. 进度保存：每完成一张图片的索引，更新进度记录

#### 数据持久化设计

```
task_queue
```

4. 如选择继续，从断点处恢复执行

#### 去重机制设计

**去重策略：**
- **轻量级去重**：通过文件路径、文件大小、修改时间等元数据快速筛选
- **文件哈希去重**：计算SHA-256哈希值，确保内容完全相同的图片不被重复添加

**性能优化策略：**
- **批量添加优化**：先通过轻量级方式初步筛选，再对潜在重复文件计算哈希值
- **单张添加**：直接进行完整去重检查
- **索引优化**：在file_path和file_size字段上建立复合索引，提升查询效率

**去重流程：**

**批量添加场景：**
1. **第一步**：轻量级筛选
   - 检查文件路径是否已存在于数据库
   - 检查文件大小和修改时间组合是否与现有记录冲突
   - 快速过滤明显不重复的文件
2. **第二步**：哈希值计算（仅对潜在重复文件）
   - 对轻量级筛选后的可疑文件计算SHA-256哈希值
   - 查询数据库中是否存在相同哈希值
   - 如存在相同哈希值，跳过该文件并记录重复

**单张添加场景：**
1. **第一步**：检查文件路径是否已存在于数据库
2. **第二步**：计算文件SHA-256哈希值
3. **第三步**：查询数据库中是否存在相同哈希值
4. **第四步**：如存在相同哈希值，跳过该文件并记录重复

**数据库设计：**
```
images
├── image_id (主键)
├── file_path (文件路径，索引)
├── file_size (文件大小，索引)
├── file_modified_time (文件修改时间，索引)
├── file_hash (SHA-256哈希值，唯一索引)
├── file_status (文件状态：正常/丢失/移动)
├── last_check_time (最后检查时间)
└── add_time (添加时间)
```

**索引设计：**
- 复合索引：(file_path, file_size, file_modified_time) - 支持轻量级去重查询
- 唯一索引：file_hash - 支持哈希值快速查找

### 并发控制规则

#### 并发操作互斥规则表

为确保系统稳定性和数据一致性，不同操作之间的并发执行需要遵循以下规则：

| 操作A | 操作B | 是否允许并发 | 说明 |
|------|------|-------------|------|
| 添加图片 | 添加图片 | 否 | 串行执行，避免索引构建冲突 |
| 添加图片 | 搜索 | 是 | 搜索仅访问已索引的图片，不影响正在构建的索引 |
| 添加图片 | 删除 | 否 | 删除操作需要等待添加完成，避免数据不一致 |
| 添加图片 | 编辑标签 | 否 | 标签编辑需要等待索引构建完成 |
| 搜索 | 搜索 | 是 | 多个搜索查询可并发执行 |
| 搜索 | 删除 | 是 | 搜索和删除操作数据隔离，可并发 |
| 搜索 | 编辑标签 | 是 | 搜索和标签编辑互不影响 |
| 删除 | 删除 | 是 | 多个删除操作可并发执行 |
| 删除 | 编辑标签 | 否 | 删除和标签编辑可能产生冲突，需串行 |
| 编辑标签 | 编辑标签 | 是 | 多个标签编辑操作可并发执行 |

#### 并发控制实现机制

**1. 任务队列管理**
- 添加图片任务使用全局队列，确保串行执行
- 其他操作根据互斥规则动态调度

**2. 数据库锁机制**
- 读操作（搜索）使用共享锁，允许多个并发读取
- 写操作（添加、删除、编辑）使用排他锁，确保数据一致性

**3. 状态同步**
- 索引构建期间，新图片对搜索不可见
- 删除操作立即生效，搜索时过滤已删除记录

**4. 用户反馈**
- 并发受限时显示相应提示信息
- 后台任务状态实时更新
